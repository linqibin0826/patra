# 功能规格说明: MeSH 数据首次导入

**特性分支**: `001-mesh-data-import`
**创建日期**: 2025-11-20
**状态**: 草稿
**输入**: 用户描述: "MeSH 数据首次导入需求"

## 澄清

### 会话 2025-11-20

- 问：6 张表的导入应该如何编排？ → 答：按依赖顺序串行导入：Descriptor → Qualifier → TreeNumber/EntryTerm/Concept → publication_mesh
- 问：MeshImportAggregate 的聚合边界应该如何设计？ → 答：MeshImportAggregate 作为聚合根，包含任务状态、总体进度和失败批次摘要；每张表的详细进度作为值对象（TableProgress）；批次详情存储在 Infrastructure 层
- 问：是否为聚合根和实体使用强类型 ID？ → 答：是，Domain 层使用强类型 ID（MeshImportId、DescriptorId、QualifierId），Infrastructure 层使用 Long 雪花 ID（MyBatis-Plus ASSIGN_ID）
- 问：导入任务是否需要发布领域事件？ → 答：发布关键状态变更事件：MeshImportCompleted、MeshImportFailed（仅在任务完成或失败时发布）
- 问：导入任务的可观察性策略如何设计？ → 答：详细追踪：每个批次处理记录 INFO 日志，SkyWalking 追踪所有批次操作，使用 Micrometer + Spring Boot Actuator 暴露监控指标（任务级别、表级别、批次级别统计）
- 问：聚合根命名规范是什么？ → 答：遵循 Patra 项目约定，使用 MeshImportAggregate（领域概念 + Aggregate 后缀），参考 patra-ingest 模块的 TaskAggregate、PlanAggregate 命名模式

## 用户场景与测试 *(必填)*

### 用户故事 1 - 管理员发起 MeSH 数据首次导入 (优先级: P1)

系统管理员通过管理接口启动 MeSH 数据的首次导入任务。系统从 NLM 官方网站下载最新的 MeSH XML 数据文件，解析并分批导入到 patra-catalog 数据库的 6 张表中。整个过程支持断点续传，即使导入过程中断，也能从上次停止的位置继续。

**为什么是这个优先级**: 这是核心功能，没有数据导入就无法使用 MeSH 数据。首次导入是后续所有功能的基础，必须优先实现。

**独立测试**: 可以通过调用"开始导入"接口完全测试并交付"MeSH 数据从零到全部导入完成"的完整价值。测试时创建空数据库，调用导入接口，验证 6 张表的数据完整性和数量（主题词 ~3.5 万、限定词 ~100、树形编号 ~8 万、入口术语 ~25 万、概念 ~18 万）。

**验收场景**:

1. **假设** patra-catalog 数据库 6 张 MeSH 表为空且无历史导入记录, **当** 管理员调用"开始导入"接口, **那么** 系统创建导入任务记录，状态为"处理中"，并开始下载和解析 NLM MeSH XML 文件

2. **假设** 导入任务正在进行中，已成功导入 5000 条主题词, **当** 系统因网络波动或其他原因中断, **那么** 任务状态标记为"失败"，进度记录保存为"主题词: 5000/35000"

3. **假设** 上次导入任务在主题词 5000 条处中断, **当** 管理员调用"重试失败任务"接口, **那么** 系统从第 5001 条主题词继续解析和导入，不重复处理前 5000 条

4. **假设** 所有数据（35000 主题词、100 限定词、80000 树形编号、250000 入口术语、180000 概念）导入完成, **当** 管理员调用"查询导入进度"接口, **那么** 返回任务状态为"成功"，各表导入数量统计准确，总耗时和完成时间可见

5. **假设** 导入过程中某批次（如第 10 批主题词）因数据格式异常失败, **当** 该批次重试 3 次仍失败, **那么** 系统标记该批次为"失败"并记录错误详情，继续处理下一批次，不阻塞整体导入流程

---

### 用户故事 2 - 管理员实时监控导入进度 (优先级: P2)

管理员在导入过程中可以随时查询当前进度，了解各表的导入完成情况、当前处理速度（每秒处理记录数）、预计剩余时间，以及是否有失败的批次需要人工介入。

**为什么是这个优先级**: 进度监控是运维必需功能，但不影响导入本身的执行。即使没有实时进度查询，导入任务仍可在后台正常完成，因此优先级略低于核心导入功能。

**独立测试**: 可以通过在导入过程中多次调用"查询导入进度"接口测试。验证返回的进度信息（已处理/总数、处理速度、预计剩余时间）是否准确且实时更新。

**验收场景**:

1. **假设** 导入任务正在处理主题词（已完成 15000/35000）, **当** 管理员调用"查询导入进度"接口, **那么** 返回详细进度：主题词 15000/35000（43%）、限定词 100/100（100%）、树形编号 30000/80000（37.5%）、入口术语 50000/250000（20%）、概念 0/180000（0%）、当前处理速度约 500 条/秒、预计剩余时间 15 分钟

2. **假设** 某批次数据导入失败并记录错误, **当** 管理员查询进度, **那么** 在进度详情中看到失败批次列表（批次编号、失败原因、失败时间、重试次数）

3. **假设** 导入任务已完成, **当** 管理员查询进度, **那么** 返回状态为"成功"，各表最终导入数量、总耗时、开始时间、结束时间一目了然

---

### 用户故事 3 - 管理员处理失败任务和重新导入 (优先级: P3)

管理员可以在导入失败后重试失败的批次，或者在需要时清除所有已导入数据和进度记录，从头重新开始全新的导入任务。所有操作通过直接调用 REST API 完成，无需前端页面。

**为什么是这个优先级**: 这是异常场景处理能力。正常情况下导入任务会自动完成，重试和清除功能主要用于数据源更新、质量问题修复等特殊场景，不影响正常导入流程，因此优先级最低。

**独立测试**: 可以通过模拟导入失败场景，调用"重试失败任务"API 验证重试机制；调用"清除进度重新开始"API 验证数据清空并重新导入。

**验收场景**:

1. **假设** 导入任务状态为"失败"且有多个批次重试失败, **当** 管理员通过 API 调用"重试失败任务", **那么** 系统仅重新处理之前失败的批次，已成功批次不重复处理

2. **假设** 导入任务已完成 50% 但管理员发现数据源有更新, **当** 管理员通过 API 调用"清除进度重新开始", **那么** 系统清空 6 张表的所有数据、删除进度记录、删除任务历史，允许重新发起全新的导入任务

---

### 边界情况

- **NLM 数据源不可用**: 当下载 MeSH XML 文件时 NLM 服务器不响应或返回错误（404、500），系统应记录错误并将任务状态标记为"失败"，允许管理员稍后重试，不应无限重试导致资源耗尽。

- **XML 文件格式异常**: 当 XML 文件结构不符合预期（如缺少必填字段、嵌套层级错误），系统应记录解析错误的具体位置（如第 X 个 Descriptor），跳过该记录继续处理，不应因个别记录导致整个导入失败。

- **数据库连接中断**: 当批次提交时数据库连接断开，系统应标记该批次为"失败"并记录错误，重试时仅重新处理失败批次，已成功提交的批次不重复插入（需确保幂等性）。

- **磁盘空间不足**: 当下载 XML 文件或导入数据时磁盘空间耗尽，系统应捕获异常并将任务状态标记为"失败"，错误信息中包含磁盘空间不足提示。

- **并发导入请求**: 当已有导入任务正在运行时，管理员再次调用"开始导入"接口，系统应拒绝新请求并返回错误提示"导入任务正在进行中，请勿重复提交"。

- **数据量与预期不符**: 当实际导入完成后，某张表的记录数与官方声明的数量差异超过 5%（如主题词预期 35000 但实际只导入 30000），系统应在任务完成后标记警告状态，提示管理员人工核查。

- **批次大小动态调整**: 当某张表数据量特别大（如入口术语 25 万条），固定批次大小 1000 可能导致过多批次提交，系统应根据表大小动态调整批次大小（如入口术语使用 2000 条/批，限定词使用 100 条/批）。

## 需求 *(必填)*

### 功能需求

- **FR-001**: 系统必须能够从 NLM 官方网站（https://nlm.nih.gov/mesh/filelist.html）下载最新的 MeSH XML 数据文件

- **FR-002**: 系统必须能够解析 MeSH XML 文件并提取 6 类数据：主题词（Descriptor）、限定词（Qualifier）、树形编号（TreeNumber）、入口术语（EntryTerm）、概念（Concept）及其关联关系

- **FR-003**: 系统必须将解析后的数据分批导入到 patra-catalog 数据库的 6 张表中，按以下依赖顺序串行执行：(1) cat_mesh_descriptor（主题词）、(2) cat_mesh_qualifier（限定词）、(3) cat_mesh_tree_number（树形编号）/ cat_mesh_entry_term（入口术语）/ cat_mesh_concept（概念）、(4) cat_publication_mesh（文献关联）。每批次大小为 1000-2000 条记录

- **FR-004**: 系统必须在每批次数据导入后独立提交事务，避免大事务导致数据库锁定或回滚成本过高

- **FR-005**: 系统必须记录导入任务的执行状态（待处理、处理中、成功、失败），状态变更时更新数据库记录

- **FR-006**: 系统必须记录每张表的导入进度（已处理记录数/总记录数），进度信息持久化到数据库以支持断点续传

- **FR-007**: 系统必须支持任务中断后从上次成功批次的下一批次继续处理，不重复导入已成功的数据

- **FR-008**: 系统必须提供管理接口"开始导入"，允许管理员发起新的导入任务（仅当无正在运行的任务时）

- **FR-009**: 系统必须提供管理接口"重试失败任务"，允许管理员手动触发重新处理失败批次或中断的任务

- **FR-010**: 系统必须提供管理接口"清除进度重新开始"，允许管理员清空已导入数据、删除进度记录，从零开始重新导入

- **FR-011**: 系统必须提供管理接口"查询导入进度"，返回当前任务状态、各表导入进度（已处理/总数、百分比）、处理速度、预计剩余时间、失败批次列表

- **FR-012**: 系统必须在批次导入失败时记录详细错误信息（失败批次编号、失败原因、失败时间、重试次数）

- **FR-013**: 系统必须确保数据一致性由业务层保证（数据库无外键约束），导入时验证关联关系的有效性（如 TreeNumber 引用的 Descriptor ID 必须存在）

- **FR-014**: 系统必须在导入完成后验证各表记录数，如与预期数量（主题词 ~35000、限定词 ~100、树形编号 ~80000、入口术语 ~250000、概念 ~180000）差异超过 5%，标记警告状态

- **FR-015**: 所有管理接口均为 REST API，无需前端页面，管理员通过 API 工具（如 Postman、curl）直接调用

### 非功能需求

**架构约束**:
- **六边形架构层次**: 此功能主要涉及 Application 层（导入任务编排）、Infrastructure 层（XML 解析、数据库批量插入、HTTP 下载）、Adapter 层（管理接口 REST API）。Domain 层包含 MeSH 领域实体（Descriptor、Qualifier 等）及其验证逻辑。
- **依赖方向**: 遵守 Adapter → Application → Domain ← Infrastructure。Domain 层不依赖任何框架（如 MyBatis、Spring），仅包含纯 Java 实体和业务规则。Infrastructure 层实现 Domain 层定义的 Repository 接口。
- **领域模型设计**（DDD）:
  - **聚合根**: MeshImportAggregate 作为 MeSH 导入聚合根，封装导入任务的完整生命周期和状态机流转，保护任务状态一致性（待处理 → 处理中 → 成功/失败）
  - **值对象**: TableProgress 表示每张表的导入进度（表名、已处理数/总数、当前状态），作为 MeshImportAggregate 的一部分
  - **实体边界**: 批次详细信息（BatchDetail）存储在 Infrastructure 层，不属于聚合根，通过 Repository 按需查询
  - **不变量**: MeshImportAggregate 确保任务状态转换的合法性（例如，不允许从"成功"转到"处理中"），保证导入顺序的一致性（Descriptor 必须在 TreeNumber 之前完成）
  - **身份标识**: 使用强类型 ID（MeshImportId、DescriptorId、QualifierId、TreeNumberId、EntryTermId、ConceptId），提供编译期类型安全，防止 ID 混淆
  - **领域事件**: 发布关键状态变更事件 MeshImportCompleted（包含 MeshImportId、导入统计、耗时）和 MeshImportFailed（包含 MeshImportId、失败原因），用于触发后续流程（如索引构建、管理员通知）
- **配置管理**: MeSH 数据导入任务的所有配置（如批次大小、超时时间、NLM 数据源 URL）通过 Nacos 配置中心管理，不应硬编码在代码中。

**性能要求**:
- **NFR-001**: 系统必须在 60 分钟内完成全量 MeSH 数据导入（约 35 万条记录），平均处理速度 ≥ 100 条/秒
- **NFR-002**: 单批次数据库提交时间 P95 ≤ 5 秒（批次大小 1000-2000 条）
- **NFR-003**: 导入任务不应占用超过 2GB 内存（避免一次性加载整个 XML 文件到内存）

**安全要求**:
- **NFR-004**: 所有管理接口必须验证管理员权限，禁止未授权访问
- **NFR-005**: 导入任务的错误日志不应泄露敏感信息（如数据库连接字符串）

**可观察性要求**:
- **NFR-006**: 每个批次处理必须记录 INFO 级别日志，包含批次编号、处理记录数、耗时、成功/失败状态
- **NFR-007**: 关键操作（任务启动、表切换、任务完成/失败）必须记录详细日志，包含上下文信息（任务 ID、表名、时间戳）
- **NFR-008**: 必须集成 SkyWalking 追踪所有批次操作，生成分布式追踪链路，包含 Span 信息（操作名称、开始/结束时间、标签）
- **NFR-009**: 必须使用 Micrometer 暴露监控指标，通过 Spring Boot Actuator 端点（`/actuator/metrics`）提供，包含：任务级别统计（总耗时、总记录数、成功率）、表级别统计（每张表的处理进度、耗时）、批次级别统计（批次处理速度、失败批次数）
- **NFR-010**: 失败批次必须记录完整的错误堆栈和上下文信息（XML 行号、记录内容摘要），便于故障排查

**配置管理说明** ⚠️:
- 所有导入任务配置（NLM 数据源 URL、批次大小、超时时间、重试策略等）通过 Nacos 配置中心管理
- 不引入 patra-registry 的 Provenance 配置（简化架构依赖）
- MeSH 数据本身就是数据字典，不从 patra-registry 获取

## 成功标准 *(必填)*

### 可衡量的结果

- **SC-001**: 管理员可以在 5 分钟内通过管理接口启动 MeSH 数据首次导入任务，并在 60 分钟内完成约 35 万条记录的导入（6 张表总计）

- **SC-002**: 导入任务在中断后（如网络波动、服务器重启）能够在 1 分钟内通过"重试失败任务"接口恢复，从断点继续处理，不重复导入已成功的数据

- **SC-003**: 导入过程中管理员可以随时查询进度，响应时间 ≤ 1 秒，进度信息准确反映各表的实时处理情况（已处理/总数、处理速度、预计剩余时间）

- **SC-004**: 导入完成后，6 张表的数据完整性 ≥ 99.5%（与 NLM 官方数据对比，允许个别格式异常记录跳过）

- **SC-005**: 导入任务失败率 ≤ 5%（因网络、数据格式等原因导致的失败批次占比），失败批次可通过"重试"接口 100% 恢复

- **SC-006**: 系统在处理 35 万条记录时内存占用峰值 ≤ 2GB，CPU 使用率 P95 ≤ 70%

- **SC-007**: 管理员在发现数据质量问题后，可以在 2 分钟内通过"清除进度重新开始"接口清空数据并重新导入，整个清除过程不影响其他模块的正常运行

- **SC-008**: 导入任务的状态变更（待处理 → 处理中 → 成功/失败）100% 准确记录，管理员可以通过日志或 API 追溯完整的任务历史
