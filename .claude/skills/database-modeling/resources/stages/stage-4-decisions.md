## 🎯 常见设计决策示例

### ADR-001: 主键类型选择

**日期：** 2025-01-15
**状态：** 已接受
**决策者：** 数据库架构师

#### 背景
需要为所有表选择统一的主键策略，确保系统能支持大规模数据增长。

#### 决策驱动因素
- 数据量：预计 5 年内单表可能超过 10 亿条记录
- 性能：主键查询和插入性能要求高
- 兼容性：需要与现有系统集成

#### 考虑的方案

**方案 A：INT AUTO_INCREMENT**
- 优点：
  - ✅ 占用空间小（4 字节）
  - ✅ 插入性能好
- 缺点：
  - ❌ 最大值约 21 亿，可能不够
  - ❌ 分库分表时容易冲突

**方案 B：BIGINT AUTO_INCREMENT**
- 优点：
  - ✅ 范围足够大（8 字节）
  - ✅ 插入性能好
  - ✅ 索引效率高
- 缺点：
  - ❌ 占用空间比 INT 大一倍
  - ❌ 分布式环境需要额外处理

**方案 C：UUID**
- 优点：
  - ✅ 全局唯一
  - ✅ 分布式友好
- 缺点：
  - ❌ 占用空间大（36 字节）
  - ❌ 索引性能差
  - ❌ 不利于范围查询

#### 决策
选择**方案 B（BIGINT AUTO_INCREMENT）**，因为：
1. 空间开销可接受（8字节 vs 4字节）
2. 性能最优（顺序插入，索引效率高）
3. 足够支持未来增长（2^63-1 ≈ 9×10^18）

#### 影响
- **正面：** 查询性能最优，支持超大规模数据
- **负面：** 比 INT 多占用 4 字节/行
- **风险：** 分布式部署需要 ID 生成器

#### 实施说明
```sql
-- 所有表主键定义
`id` BIGINT UNSIGNED NOT NULL AUTO_INCREMENT PRIMARY KEY
```

### ADR-002: 软删除 vs 硬删除

**日期：** 2025-01-15
**状态：** 已接受

#### 背景
需要决定数据删除策略，平衡数据安全和查询性能。

#### 考虑的方案

**方案 A：硬删除**
- 优点：
  - ✅ 节省存储空间
  - ✅ 查询简单
- 缺点：
  - ❌ 数据不可恢复
  - ❌ 无法审计

**方案 B：软删除**
- 优点：
  - ✅ 数据可恢复
  - ✅ 保留审计信息
  - ✅ 支持回收站功能
- 缺点：
  - ❌ 需要所有查询加 WHERE deleted = 0
  - ❌ 占用存储空间

#### 决策
选择**方案 B（软删除）**，并通过以下措施缓解负面影响：
1. 使用 MyBatis-Plus 自动过滤
2. 定期归档超过 1 年的删除数据
3. 为 deleted 字段建立索引

#### 实施说明
```sql
-- 所有表添加软删除字段
`deleted` TINYINT(1) NOT NULL DEFAULT 0 COMMENT '软删除标志',
INDEX `idx_deleted` (`deleted`)
```

### ADR-003: JSON vs 关系表存储配置

**日期：** 2025-01-16
**状态：** 已接受

#### 背景
系统需要存储大量灵活的配置信息，需要在灵活性和查询性能间权衡。

#### 决策
采用**混合策略**：
- 核心配置：使用关系表（便于查询和约束）
- 扩展配置：使用 JSON 字段（灵活性高）

#### 实施示例
```sql
-- 核心字段使用列
`core_config_a` VARCHAR(100),
`core_config_b` INT,

-- 扩展配置使用 JSON
`extra_config` JSON COMMENT '扩展配置 {"key1": "value1", ...}'
```

## 🔍 决策评估框架

### 技术债务评分

| 维度 | 权重 | 评分标准 |
|------|------|----------|
| 可维护性 | 30% | 1-5 分 |
| 性能影响 | 25% | 1-5 分 |
| 扩展性 | 20% | 1-5 分 |
| 复杂度 | 15% | 1-5 分 |
| 成本 | 10% | 1-5 分 |

### 风险评估矩阵

```
概率 ↑
高 │ 中风险 │ 高风险 │ 极高风险
中 │ 低风险 │ 中风险 │ 高风险
低 │ 极低风险│ 低风险 │ 中风险
   └─────────┴────────┴──────────→
     低       中       高    影响
```

