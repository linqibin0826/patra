# 架构评审检查清单

快速评审指南，确保架构方案符合六边形架构 + DDD 最佳实践。

## 🎯 评审决策树

```
收到方案 → 识别类型 → 应用检查清单
    ├─ 新功能 → 功能设计检查清单
    ├─ 重构 → 重构评审清单
    ├─ 技术升级 → 技术选型清单
    ├─ 性能优化 → 性能方案清单
    └─ 系统集成 → 集成方案清单
```

## ✅ 功能设计检查清单

### 层级设计
- [ ] **Adapter 层**: 只负责协议转换，无业务逻辑
- [ ] **Application 层**: 只编排，不决策
- [ ] **Domain 层**: 包含所有业务规则
- [ ] **Infrastructure 层**: 只实现技术细节

### 依赖检查
- [ ] 依赖方向: Adapter → App → Domain ← Infra
- [ ] Domain 层无框架依赖 (仅 Lombok, Hutool)
- [ ] 接口定义在 Domain，实现在 Infrastructure
- [ ] 无循环依赖

### DDD 设计
- [ ] 聚合边界清晰，一个事务一个聚合
- [ ] 实体有唯一标识 (使用强类型 ID)
- [ ] 值对象不可变 (使用 record)
- [ ] 领域事件使用过去式命名
- [ ] 业务不变性在聚合内维护

### 事务管理
- [ ] @Transactional 只在 Orchestrator
- [ ] 读操作考虑是否需要事务
- [ ] 长事务拆分为多个短事务
- [ ] 跨聚合使用最终一致性

## 🚫 常见违规模式

| 违规类型 | 错误示例 | 正确做法 |
|---------|---------|----------|
| **业务逻辑泄露** | Controller 验证业务规则 | 业务规则在 Domain |
| **依赖倒置** | Domain 依赖 Spring | 定义接口，Infra 实现 |
| **事务滥用** | Domain 或 Controller 使用事务 | 仅 Orchestrator 使用 |
| **贫血模型** | 实体只有 getter/setter | 实体包含业务行为 |
| **过度设计** | 简单 CRUD 使用复杂模式 | 按需选择合适模式 |

## 📊 评审评分标准

### 评分维度
1. **架构合规性** (40%)
   - 层级划分 (10%)
   - 依赖规则 (15%)
   - 模式应用 (15%)

2. **领域设计** (30%)
   - 聚合边界 (10%)
   - 业务规则封装 (10%)
   - 领域事件设计 (10%)

3. **技术实现** (20%)
   - 技术选型 (10%)
   - 性能考虑 (10%)

4. **可维护性** (10%)
   - 代码清晰度 (5%)
   - 测试策略 (5%)

### 评审结论
- **🟢 通过** (≥80分): 可以实施
- **🟡 条件通过** (60-79分): 修正后实施
- **🔴 不通过** (<60分): 需要重新设计

## 💡 快速决策指引

### 何时需要 Coordinator？
- 关注点复杂 (>3个)
- 需要并行处理
- 职责明显可分离

### 何时需要领域服务？
- 跨多个聚合的业务逻辑
- 不属于任何单一实体
- 无状态的业务计算

### 何时使用 Outbox？
- 需要保证事件发送
- 事务与消息的原子性
- 跨服务数据一致性

### 何时拆分聚合？
- 事务边界不同
- 并发修改冲突
- 生命周期独立
