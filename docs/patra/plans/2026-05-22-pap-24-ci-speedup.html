  3. 步骤 3：删除 build-and-test job 内所有 --no-daemon 标志

    找到 build-and-test job 中的 4 处 ./gradlew ... --no-daemon，删除 --no-daemon：

    # Before
- name: ✅ Check (test + spotbugsMain + spotlessCheck)
  run: ./gradlew check --no-daemon

- name: 🔬 Integration tests (Testcontainers)
  run: ./gradlew integrationTest --no-daemon

- name: 🎯 E2E tests
  run: ./gradlew e2eTest --no-daemon

- name: 📊 JaCoCo aggregate reports
  run: ./gradlew testCodeCoverageReport integrationTestCodeCoverageReport --no-daemon

# After
- name: ✅ Check (test + spotbugsMain + spotlessCheck)
  run: ./gradlew check

- name: 🔬 Integration tests (Testcontainers)
  run: ./gradlew integrationTest

- name: 🎯 E2E tests
  run: ./gradlew e2eTest

- name: 📊 JaCoCo aggregate reports
  run: ./gradlew testCodeCoverageReport integrationTestCodeCoverageReport

    理由：Gradle 9 + setup-gradle@v4 下 --no-daemon 是反模式（daemon 复用 JVM warmup 反而更快）。spec §1 已说明。

  4. 步骤 4：给 setup-gradle@v4 加 cache-encryption-key

    修改 setup-gradle step：

    # Before
- name: 🐘 Setup Gradle (build cache + dependency cache + wrapper verify)
  uses: gradle/actions/setup-gradle@v4

# After
- name: 🐘 Setup Gradle (build cache + dependency cache + configuration cache + wrapper verify)
  uses: gradle/actions/setup-gradle@v4
  with:
    cache-encryption-key: ${{ secrets.GRADLE_ENCRYPTION_KEY }}

    理由：setup-gradle@v4 自带 configuration-cache 缓存能力，但只有在 cache-encryption-key 存在时才会保存到 GHA cache。secret 缺失时此参数为空字符串，setup-gradle 容忍降级（CI 不会失败，仅 cc cache 不持久化）。

  5. 步骤 5：新增 Pre-pull container images step

    在 setup-gradle step 之后、check step 之前插入：

    - name: 🐳 Pre-pull container images (并发预热,藏入编译关键路径)
  run: |
    docker pull "$POSTGRES_IMAGE" &
    docker pull "$ROCKETMQ_IMAGE" &
    wait
  env:
    POSTGRES_IMAGE: ${{ env.POSTGRES_IMAGE }}
    ROCKETMQ_IMAGE: ${{ env.ROCKETMQ_IMAGE }}

    说明：GHA step 之间串行执行，但 step 内部并发 pull。两镜像并行 pull 总时长 ~40-50s，在 check step 的 Gradle 编译期间已完成。

  6. 步骤 6：新增镜像版本一致性校验 step

    在 Pre-pull step 之后插入：

    - name: 🔍 Verify container image versions consistency
  run: |
    set -e
    FAIL=0

    PG_FILE="linqibin-commons/linqibin-spring-boot-starter-test/src/main/java/dev/linqibin/starter/test/container/initializer/PostgreSQLContainerInitializer.java"
    if ! grep -q "private static final String POSTGRESQL_IMAGE = \"$POSTGRES_IMAGE\";" "$PG_FILE"; then
      echo "::error file=$PG_FILE::POSTGRESQL_IMAGE 常量与 workflow env POSTGRES_IMAGE=$POSTGRES_IMAGE 不一致"
      FAIL=1
    fi

    for COMPOSE in \
      "linqibin-commons/linqibin-spring-boot-starter-test/src/main/resources/docker-compose-rocketmq.yml" \
      "patra-api/patra-ingest/patra-ingest-boot/src/test/resources/docker-compose-rocketmq.yml"; do
      COUNT=$(grep -c "image: $ROCKETMQ_IMAGE\b" "$COMPOSE" || true)
      if [ "$COUNT" -ne 2 ]; then
        echo "::error file=$COMPOSE::期望 2 处 'image: $ROCKETMQ_IMAGE',实际 $COUNT 处"
        FAIL=1
      fi
    done

    if [ "$FAIL" -ne 0 ]; then
      echo "镜像版本治理 SSOT 失守。修改镜像版本时需同步 ci.yml env + Java 常量 + 2 份 compose 文件。"
      exit 1
    fi
    echo "✓ 3 个文件的镜像版本与 workflow env 一致"

    说明：失败时给出明确指引；通过时打印 ✓ 行便于 CI 日志检索。grep 用 \b 防止 5.3.10 误中 5.3.1。

  7. 步骤 7：手工添加 GRADLE_ENCRYPTION_KEY repo secret

    用户在本地终端执行（注意：此步骤需要用户操作，AI 不能代为修改 GitHub secret）：

    openssl rand -base64 16

    把输出的字符串通过 GitHub repo → Settings → Secrets and variables → Actions → New repository secret 添加，名为 GRADLE_ENCRYPTION_KEY。

    说明：PR1 不依赖此 secret 也能 merge（setup-gradle 容忍降级），但缺失时 cc cache 不会持久化，PR1 收益打折。建议在 PR1 实测前完成。

  2. 步骤 2：新建 linqibin.jacoco-aggregation convention plugin

    创建 build-logic/src/main/kotlin/linqibin.jacoco-aggregation.gradle.kts：

    /**
 * Patra JaCoCo Aggregation Convention Plugin
 *
 * 在根项目应用,聚合所有子项目的 jacoco *.exec 文件,
 * 输出统一的 XML + HTML 报告供 CI coverage job 上传 Codecov。
 *
 * 用法：根 build.gradle.kts 加 plugins { id("linqibin.jacoco-aggregation") }
 *
 * 与 linqibin.java-base 中各模块各自的 jacocoTestReport 共存：
 *   - jacocoTestReport (per-module)  → 供本地 ./gradlew check 顺带跑
 *   - jacocoAggregatedReport (root)  → 供 CI coverage job 跑,跨模块聚合
 */

plugins {
    base
    id("jacoco-report-aggregation")
}

reporting {
    reports {
        val jacocoAggregatedReport by creating(JacocoCoverageReport::class) {
            testSuiteName = "test"
        }
    }
}

// 聚合 test + integrationTest 两类 .exec 文件,与现有 testCodeCoverageReport /
// integrationTestCodeCoverageReport 等价但走单一聚合任务,便于 CI 下载 artifact 后跑
tasks.named("jacocoAggregatedReport", JacocoReport::class) {
    // 显式扫描所有子项目的 .exec (含 test / integrationTest)
    executionData.setFrom(
        fileTree(rootDir) {
            include("**/build/jacoco/test.exec")
            include("**/build/jacoco/integrationTest.exec")
        }
    )

    // 类与源 path 沿用 linqibin.java-base 中的 jsqlparser exclude
    classDirectories.setFrom(
        files(
            subprojects.flatMap { sub ->
                sub.layout.buildDirectory.dir("classes/java/main").get().asFile.let {
                    if (it.exists()) listOf(fileTree(it) { exclude("net/sf/jsqlparser/**") }) else emptyList()
                }
            }
        )
    )
    sourceDirectories.setFrom(
        files(subprojects.mapNotNull { it.layout.projectDirectory.dir("src/main/java").asFile.takeIf(java.io.File::exists) })
    )

    reports {
        xml.required = true
        html.required = true
        xml.outputLocation = layout.buildDirectory.file("reports/jacoco/aggregated/jacocoAggregatedReport.xml")
        html.outputLocation = layout.buildDirectory.dir("reports/jacoco/aggregated/html")
    }
}

    说明：jacoco-report-aggregation plugin 是 Gradle 内置（无需 buildscript 依赖）。这里手工配置 executionData / classDirectories / sourceDirectories 是因为聚合多个 source set（test + integrationTest），plugin 默认配置只聚合 test。

  5. 步骤 5：检查 XML 格式符合 Codecov 期望
