## 1. 版本目标 / 价值

作为 Patra 唯一开发者，我希望提交一个 PR 后能自动获得：

1. **Gradle CI 全绿验证**（编译 / 单元测试 / 集成测试 / E2E，跑 Gradle 而非过时的 Maven）
2. **CodeRabbit AI Review** 自动评论
3. **静态分析零告警**（SpotBugs 抓逻辑 bug + Spotless 抓格式）
4. **分支保护**强制 CI 通过才能合并到 main（CI 真正成为门禁）
5. **Dependabot 依赖自动升级 PR**（节省手动盯版本时间）

同时本地 IDE 打开项目时，**模块树按 `linqibin-commons` / `patra-starters` / `patra-api` 三段清晰分组**（修复 `settings.gradle.kts` 扁平 path），根项目名大小写一致。

完成后业务功能开发就真正"开箱即敲"——不再被基建分心。

## 4. 关键决策

### A. 构建工具统一：所有 CI / pre-commit / 质量门禁一律 Gradle 命令

- **影响 Issue**：#1 #3 #4 #6 #7
- **判断规则**：跨 ≥ 2 Issue + 不可逆
- **决策**：`.github/workflows/*.yml` / `scripts/git/*.sh` / `.pre-commit-config.yaml` 中 `mvn` 命令全部删除或改写为 `./gradlew`

### B. 静态分析工具链最小化：SpotBugs + Spotless

- **影响 Issue**：#3 #4 + 后续添加新模块时的质量规则
- **判断规则**：跨 ≥ 2 Issue + 决定项目 "Java quality stack"
- **决策**：SpotBugs（抓逻辑 bug）+ Spotless（抓格式）双工具；不引入 Checkstyle / PMD / Sonar / Semgrep（与 Spotless 部分重叠 + 规则海洋容易陷入配置地狱）

### C. 质量门禁阻断策略：传统静态分析阻断 PR，AI Review 不阻断

- **影响 Issue**：#1 #3 #4 #5 #6
- **判断规则**：跨 ≥ 2 Issue + 不可逆（一旦设 required check 所有 PR 受约束）
- **决策**：
  - ✅ 阻断 PR：CI 测试失败、`spotlessCheck` 失败、`spotbugsMain` 失败
  - ❌ 不阻断：CodeRabbit Review 评论（AI 建议性质）、`spotbugsTest`（测试代码不做静态分析）

### D. 覆盖率工具与阈值策略：JaCoCo + Codecov 上传，v0.3 不设阈值门禁

- **影响 Issue**：#1 + 未来业务模块的测试要求
- **判断规则**：跨 ≥ 2 Issue + 跨层级（CI + build.gradle.kts + GitHub repo）
- **决策**：
  - ✅ 接：JaCoCo 报告产出 + Codecov 上传可视化
  - ❌ 不接：覆盖率阈值阻断（绿地项目还在快速演化，过早设阈值会自缚；留待 v0.4+）

### E. Gradle 模块路径与物理目录强一致（不再扁平 path）

- **影响 Issue**：#2 + 未来每次新增模块
- **判断规则**：不可逆（一旦定下命名规则就是项目宪法）
- **决策**：Gradle path `:parent:child` 必须与物理目录 `parent/child/` 一一对应
  - 例：`:linqibin-commons:linqibin-spring-boot-starter-core` → `linqibin-commons/linqibin-spring-boot-starter-core/`
  - 禁止：扁平 path + projectDir 重映射的拼装（当前 `includeAt` 的副作用）

### F. 依赖更新策略：Dependabot 分组 PR + 不自动合并

> ⚠️ **已反转（2026-05-22, PAP-20）**：综合评估后移除 Dependabot。单人项目按业务节点手动控制依赖升级节奏比"每周 5 个 PR 被动 review"更高效；安全升级走 GitHub Security Advisories 单独跟进。配置 `.github/dependabot.yml` 已删除，4 个遗留 PR (#63–66) 已关闭。

- **影响 Issue**：#6 + 未来每周的依赖 PR review 节奏
- **判断规则**：跨层级（CI + GitHub + 人为 review 工作量）
- **原决策**（已反转）：
  - ✅ 分组：minor / patch 同 ecosystem 合并到 1 个 PR（Gradle / npm / github-actions 三组）
  - ❌ 不自动合并：单人项目无 review 兜底
