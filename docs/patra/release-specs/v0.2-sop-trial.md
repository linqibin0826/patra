## 1. 版本目标 / 价值

用户打开 Patra portal 首页，能实时看到 patra-api 网关及 registry / ingest / catalog 三个微服务的可用性状态与时延。

验证目标：
- portal ↔ api ↔ services 端到端连通
- SOP 全流程（Tech Design → BE → Design → FE → Review → Done）状态切换

## 2. 功能列表（2 个 Issue）

| # | 主题 | Area | 关键能力 |
|---|------|------|----------|
| 1 | 后端：聚合 health 接口 | `area:api` | `GET /api/v1/health` 返回 3 个微服务可用性 + 时延 |
| 2 | 前端：health 状态卡片 | `area:portal` | Portal 首页显示 3 个服务卡片（绿/红 + 服务名 + 时延 + 30s 自动刷新） |

两个 Issue 默认 label：`path:full`（走完整 SOP 链路）。

## 3. 范围边界（不做）

- 不做认证（health 接口 public）
- 不做健康历史持久化（仅实时）
- 不做告警 / 邮件 / Slack 通知
- 不做服务自动重启 / 自愈
- 不做 Prometheus metrics 端点

## 4. 关键决策

- **聚合点在 patra-gateway-boot**：网关层聚合各微服务 actuator/health，portal 只面向网关一个端点
  - *判断规则*：跨 ≥2 Issue（影响 BE 接口设计 + 未来其他聚合类 Issue）
- **FE 用 TanStack Query 30s 轮询**，不引入 WebSocket
  - *判断规则*：跨层级（影响所有"实时数据"类 FE Issue 的统一选择）
- **不引入 Sentry / 监控基础设施**
  - *判断规则*：不可逆决策（本版及后续暂不接入，保持栈最小）

## Issue 池（建议清单）

### Issue 1: BE - 聚合 patra-gateway-boot health 接口

- **Title**: BE: 聚合 patra-gateway-boot /api/v1/health 接口
- **Labels**: `path:full`, `area:api`
- **AC**:
  - `GET /api/v1/health` 返回 200 + JSON
  - JSON 含 3 个 service entries（registry / ingest / catalog）
  - 每个 entry 包含 `status` (UP / DOWN) + `latencyMs`
  - 任一服务 DOWN 时，整体 status = DEGRADED 但不返回 5xx
  - 单元测试 + integration test 全绿
- **备注**：网关层聚合实现，单服务 actuator/health 仍保留；时延通过 actuator 调用时长度量

### Issue 2: FE - portal 首页 health 状态卡片

- **Title**: FE: portal 首页 health 状态卡片（3 卡片 + 30s 轮询）
- **Labels**: `path:full`, `area:portal`
- **AC**:
  - 首页路由 `/` 显示 3 个服务状态卡片（registry / ingest / catalog）
  - 每张卡片含：服务名 + 状态点（绿 UP / 红 DOWN / 黄 DEGRADED）+ 时延 ms
  - 使用 TanStack Query 30s 间隔轮询 `/api/health` BFF
  - BFF 转发到 BE `/api/v1/health`
  - Vitest 组件测试 + BFF 路由测试全绿
- **备注**：不引入 WebSocket，纯轮询；错误降级时显示"无法连接"灰色卡片
