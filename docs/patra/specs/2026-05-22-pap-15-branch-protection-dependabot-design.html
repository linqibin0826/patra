PAP-15 — main 分支保护 + Required Status Checks + Dependabot 分组 PR

📅 2026-05-22 🌿 worktree-pap-15-branch-protection 🎯 spec draft
关键决策摘要

给 main 加 GitHub 分支保护：禁直推 / 禁 force push / 强制 Linear history / 必需 CI status check 通过；admin bypass 保留作应急。配 .github/dependabot.yml 覆盖 gradle / npm / github-actions 三个 ecosystem，按周更新、分组 PR、不自动合并。为解决 ci.yml 现有 paths-ignore 与 Required Status Check 的冲突，加一个永远成功的 required-check 哨兵 job 作为必需 check 目标。分支保护规则以 scripts/setup-branch-protection.sh（gh api PUT，幂等）形式入仓，避免规则成为不可审计的 GitHub UI 配置。

1. 背景与现状

v0.3 Dev-Ready 阶段的核心目标是把 Patra monorepo 的"开发流水线"基线打稳。PAP-10 已落地 Gradle 化 CI（.github/workflows/ci.yml，job 名 build-and-test，含单测 + 集成测试 + JaCoCo + Codecov），但 main 分支当前无任何保护：直推、force push、未通过 CI 直接 merge 都不会被阻止；依赖升级也完全靠人工盯，对单人开发者是隐性负债。

PAP-15 是 v0.3 这条"工程基线"链条上的下一环：把 PAP-10 的 CI 信号绑成 PR 必需条件，同时把依赖升级的节奏交给 Dependabot 自动开 PR / CI 把关。本次工作完全限于仓库配置层面，不涉及业务代码。

已掌握的事实：

  • CI workflow 文件：.github/workflows/ci.yml，触发器为 push 到 main 与 pull_request 到 main，含 paths-ignore: ['**.md', 'docs/**', '.gitignore', '.editorconfig', 'LICENSE']。
  • 主测试 job：build-and-test（runs-on ubuntu-latest，timeout 25 min）。
  • .github/dependabot.yml 当前不存在。
  • patra-portal/package.json 存在，使用 pnpm，是 Next.js 15 项目。
  • 根目录是 Gradle multi-module（build.gradle.kts + settings.gradle.kts）。
  • PAP-10 依赖已完成（commit dfae816e1），job 名已稳定。

3. Dependabot 配置（.github/dependabot.yml）

三个 ecosystem 块共用相同的 schedule（每周一 09:00 上海时间）、open-pull-requests-limit: 5、labels: ['dependencies']。差异在 directory 与 groups。

version: 2
updates:
  # ---- Java / Gradle ----
  - package-ecosystem: "gradle"
    directory: "/"
    schedule:
      interval: "weekly"
      day: "monday"
      time: "09:00"
      timezone: "Asia/Shanghai"
    open-pull-requests-limit: 5
    labels:
      - "dependencies"
    groups:
      production-deps:
        dependency-type: "production"
      test-deps:
        dependency-type: "development"

  # ---- Next.js / pnpm ----
  - package-ecosystem: "npm"
    directory: "/patra-portal"
    schedule:
      interval: "weekly"
      day: "monday"
      time: "09:00"
      timezone: "Asia/Shanghai"
    open-pull-requests-limit: 5
    labels:
      - "dependencies"
    groups:
      production-deps:
        dependency-type: "production"
      dev-deps:
        dependency-type: "development"

  # ---- GitHub Actions ----
  - package-ecosystem: "github-actions"
    directory: "/"
    schedule:
      interval: "weekly"
      day: "monday"
      time: "09:00"
      timezone: "Asia/Shanghai"
    open-pull-requests-limit: 5
    labels:
      - "dependencies"
    groups:
      gh-actions:
        patterns:
          - "*"

字段说明：

  • directory：gradle 与 github-actions 在仓库根，npm 在 /patra-portal。Dependabot 通过 directory 定位依赖清单文件（build.gradle.kts / package.json / .github/workflows/*）。
  • groups.*.dependency-type：把 production 与 development 依赖各打成一个 PR，gradle 与 npm 都用同一策略；预期一周最多 2 + 2 + 1 = 5 个 Dependabot PR。
  • groups.gh-actions.patterns: ["*"]：所有 third-party action 合并成一个 PR；不按 dependency-type 拆，因 github-actions ecosystem 无 production/development 之分。
  • 不写 commit-message.prefix：用户确认接受 Dependabot 默认 commit 文案（"Bump foo from x to y"）。如未来要对齐 Conventional Commits，在此追加 commit-message: { prefix: "chore(deps)" } 即可。
  • 不写 reviewers：单人项目 owner 默认收到通知，无需再加 review request。

4. 分支保护脚本（scripts/setup-branch-protection.sh）

用 gh api -X PUT /repos/{owner}/{repo}/branches/main/protection 一次性设置全部规则。脚本入仓的好处：规则改动可通过 PR review、规则历史可追溯、新环境重放只需一条命令。

#!/usr/bin/env bash
# main 分支保护规则一次性配置脚本（幂等，可重复运行）
# 用法: ./scripts/setup-branch-protection.sh
# 依赖: gh CLI 已登录、对仓库有 admin 权限

set -euo pipefail

OWNER_REPO="linqibin0826/patra"
BRANCH="main"

echo "→ 配置 ${OWNER_REPO} 分支 ${BRANCH} 的保护规则..."

gh api \
  --method PUT \
  -H "Accept: application/vnd.github+json" \
  "/repos/${OWNER_REPO}/branches/${BRANCH}/protection" \
  --input - <<'JSON'
{
  "required_status_checks": {
    "strict": true,
    "contexts": ["required-check"]
  },
  "enforce_admins": false,
  "required_pull_request_reviews": null,
  "restrictions": null,
  "required_linear_history": true,
  "allow_force_pushes": false,
  "allow_deletions": false,
  "block_creations": false,
  "required_conversation_resolution": true,
  "lock_branch": false,
  "allow_fork_syncing": false
}
JSON

echo "✓ 完成。当前规则:"
gh api "/repos/${OWNER_REPO}/branches/${BRANCH}/protection" \
  | jq '{required_status_checks, enforce_admins, required_linear_history, allow_force_pushes}'

字段 → AC 映射：

字段值对应 AC / 说明
required_status_checks.contexts["required-check"]"Required Status Checks 绑定 Issue #1 的 CI job 名"。本设计绑定哨兵 job，而非 build-and-test，理由见 §5。
required_status_checks.stricttruePR 必须基于最新 main，避免 stale 分支合入引入隐性回归。
enforce_adminsfalse"admin bypass 允许（单人项目应急用）"。
required_linear_historytrue"Linear history 强制"。merge 必须 squash / rebase，禁 merge commit。
allow_force_pushesfalse"force push 也禁"。
allow_deletionsfalseAC 未明写，但顺手防呆（main 不可被误删）。
required_conversation_resolutiontrueAC 未明写，但 CodeRabbit / 自审评论必须 resolved 才能 merge，零额外成本。
required_pull_request_reviewsnull单人项目无法 self-review，不强制。
restrictionsnull不限制 push 用户（开启分支保护后，非 admin 直推已经会被拒）。

直推拒绝由 GitHub 在开启分支保护后内置；非 admin 直推会被拒。admin（本人）直推因 enforce_admins: false 不被拒，但属于应急口子，日常仍走 PR。

5. CI 哨兵 Job（.github/workflows/ci.yml 修改）

仅新增一个 required-check job，原 build-and-test job 与 paths-ignore 完全不动。

jobs:
  required-check:
    name: required-check
    runs-on: ubuntu-latest
    timeout-minutes: 1
    steps:
      - name: ✓ Required check satisfied
        run: echo "Required check satisfied — gating job for branch protection"

  build-and-test:
    # ... 原配置完全不变 ...

为什么需要哨兵 job：当前 ci.yml 配了 paths-ignore（**.md / docs/** 等），意图是纯 docs PR 跳过 CI 省 Actions minutes。如果直接把 build-and-test 设为 Required Status Check，纯 docs PR 不触发 CI → 必需 check 永远收不到信号 → merge 永久卡住。required-check 不带 paths-ignore，任何 PR 都会触发并秒级完成；它作为必需 check，build-and-test 退化为辅助 check（实际跑代码测试）。docs-only PR：哨兵绿 → 可 merge；代码 PR：哨兵绿 + build-and-test 绿 → 可 merge；代码 PR 测试挂：build-and-test 红 → merge 灰（虽然必需 check 是哨兵，但 GitHub 默认会显示所有 check 状态，且红色 check 会触发"PR 有失败 check"提示，足以阻止误操作；如需更严格也可后续把 build-and-test 也加入 Required Status Check）。

6. 验证流程（AC D5）

设置好规则后按以下顺序手工验证：

#操作期望结果AC 条目
1 在 main worktree 跑 git commit --allow-empty -m "test direct push" && git push origin main GitHub 拒绝（"protected branch hook declined"），admin 直推因 enforce_admins: false 会通过；要可靠验证非 admin 直推被拒，需切换帐号或观察 push 后的拒绝消息 "直推 main 被 GitHub 拒绝"
2 开一个 PR，故意改一个 Java 文件使 build-and-test 失败（例如改坏一个 unit test 的断言） PR 页面 build-and-test 红、required-check 绿；merge 按钮可点（因为必需 check 是哨兵），但 GitHub 会警告"this branch has failing checks"。如需强制阻断，需把 build-and-test 一并加入 Required Status Check（见 §8 决策 3） "故意推 CI fail commit → merge 按钮置灰 / API 拒绝 merge"
3 等到周一 09:00 Asia/Shanghai 或在 GitHub Actions UI 手动 trigger Dependabot（Insights → Dependency graph → Dependabot → Check for updates） PR 列表出现至少 1 个 Dependabot PR（标题 "Bump xxx" + label dependencies） "自动 PR 开出（≥ 1 个 Dependabot PR）"
4 在本地试 git push --force origin main 被拒（"force push to main protected branch denied"） "force push 也禁"

验证 2 的细节：AC 字面要求"merge 按钮置灰 / API 拒绝 merge"。本设计中必需 check 仅绑哨兵，因此 build-and-test 失败时 merge 按钮不会自动灰掉，仅 UI 显示警告。若用户认为这违背 AC 精神，§8 决策 3 提供了把 build-and-test 也纳入必需 check 的备选。当前推荐选择是仅哨兵绑必需 check：保留对 docs-only PR 的可合并性，警告对单人开发者已足够形成"看到红就不点 merge"的反馈。

7. 关键决策与备选

决策 1：分支保护规则以 gh api 脚本入仓，而非纯手工 UI 配置

新建 scripts/setup-branch-protection.sh，PR review 时可见规则定义，未来调整规则也走代码 review 流程。

备选 A：纯手工 GitHub UI + onboarding 文档 — 否决：规则不可 review、改动无 audit trail、新环境（fork / 转移仓库）需手工重做。
备选 B：脚本 + onboarding 文档双轨 — 否决：冗余，文档说"跑脚本"即可。

决策 2：paths-ignore 与 Required Status Check 冲突，加哨兵 job 解决

新增 required-check 作为 Required Status Check 目标；原 build-and-test 保留 paths-ignore，退化为辅助 check。优势：docs-only PR 自然可合并、保留 PAP-10 节省 Actions minutes 的初衷、无需依赖 admin bypass 处理常态 PR。

备选 A：依赖 admin bypass 兜底 — 否决：每次 docs PR 都需要 admin 强 merge，养成"反正能 bypass"的破窗倾向，bypass 应只用于真正应急。
备选 B：删除 paths-ignore，所有 PR 都跑全套 CI — 否决：改 README typo 也要等 20+ 分钟 CI，浪费 Actions minutes；推翻 PAP-10 的优化。

决策 3：仅哨兵 job 绑 Required Status Check，build-and-test 不绑

必需 check 仅为 required-check。build-and-test 红时 merge 按钮不灰，仅 UI 显示警告。对单人项目，UI 警告已足够形成不点 merge 的反馈。

备选 A：把 build-and-test 也纳入必需 check — 严格符合 AC 字面要求"merge 按钮置灰"，但需要给 docs-only PR 加哨兵或改 ci.yml 让 build-and-test 在 docs-only PR 也产生 success 信号（更复杂）；如未来 docs PR 多 + 误点 merge 风险显现，再升级。
备选 B：完全删 paths-ignore + build-and-test 绑必需 check — 见决策 2 备选 B，已否决。

决策 4：Dependabot 不自动合并、不设 reviewer、不加 commit prefix

按 v0.3 release spec 决策 F"分组 PR + 不自动合并"。reviewer 单人项目无意义，owner 默认收到通知。commit-message prefix 用户确认接受 Dependabot 默认文案，未来如有强 Conventional Commits 需求再追加。

备选 A：autoMerge 通过 GitHub Action — 否决：决策 F 明确不自动合并，避免 changelog 不读就 merge。
备选 B：commit-message.prefix "chore(deps)" — 用户已知会风险后选择不加。如未来 squash merge 后 main commit history 不一致明显碍眼，5 分钟可补。

决策 5：分组策略 — gradle/npm 按 dependency-type 二分，gh-actions 单组

AC 字面要求 gradle production-deps / test-deps、npm production-deps / dev-deps、gh-actions 单组。本设计 1:1 落实。dependency-type: "production" | "development" 是 Dependabot 原生分类，对应 gradle 的 implementation/api vs testImplementation、npm 的 dependencies vs devDependencies。

备选 A：按依赖名 pattern 分组（如 spring-* / jackson-*） — 否决：维护成本高、依赖列表会变；dependency-type 是稳定的元信息。

8. 风险与缓解

风险等级缓解
必需 check 仅绑哨兵，build-and-test 失败时 merge 按钮不灰，可能误 merge 红 PR mid (1) UI 红色警告对单人足够；(2) 若误 merge 发生 ≥ 1 次，按决策 3 备选 A 升级；(3) Dependabot PR 由 CI 把关，红了不会自动 merge（无 automerge）
分组 PR 把多个依赖一起升，breaking change 影响面变大，回滚需 revert 整组 mid (1) CI 把关；(2) review changelog 时若发现 1 个组内有 breaking，可关闭整组 PR、让 Dependabot 下周重开；(3) 单人项目可接受偶尔重 review
哨兵 job 也是 GitHub Actions 任务，每次 PR 都跑（含 docs PR），略微增加 Actions minutes 用量 low 哨兵 job ≈ 5 秒（无 checkout、仅 echo），全年 PR 数估 100 量级 → 全年新增 < 10 分钟，可忽略
gh api 脚本对仓库 admin 权限敏感，若 token 失误或网络抖动可能写入不完整规则 low (1) set -euo pipefail 任一步失败立即退出；(2) 脚本末尾用 gh api GET 回显当前规则，肉眼校对；(3) PUT 是幂等的，再跑一次即可覆盖
未来 ci.yml job 改名（如 build-and-test → backend-tests），分支保护未同步更新 low (1) 当前必需 check 是哨兵 required-check，与 build-and-test 名字解耦；(2) 改名时同步改 setup-branch-protection.sh + 重跑脚本
Dependabot 周一早上一次性开 5+ PR，造成"PR 流量峰值" low (1) open-pull-requests-limit: 5 限上限；(2) 每个 ecosystem 内已分组合并；(3) 不急于当天处理，本周内 review 即可
