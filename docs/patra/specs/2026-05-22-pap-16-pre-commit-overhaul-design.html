PAP-16 — pre-commit hooks 重整 + commitlint + scripts/git Maven 清理

📅 2026-05-22 🌿 worktree-pap-16-pre-commit-overhaul 🎯 spec draft
关键决策摘要

把当前".pre-commit-config.yaml 全注释 + 手写 .git/hooks/pre-commit bash 脚本 + 一批 Maven 时代的 scripts/git/mvn_*.sh"三件套替换成 pre-commit 框架统一管理的新体系。pre-commit 阶段挂 spotless（auto-apply）+ 文件 hygiene + gitleaks + commit-msg 阶段挂 commitlint（conventional commits，9 + 1 种 type）。不在本地挂 spotbugs（业界调研结论：spotbugs 单模块 10–60s、与 --no-verify 形成博弈，Spring Boot 等大型 Java OSS 均只在 CI 跑静态分析），spotbugs 完全交给 CI。删 scripts/git/ 全部 5 个 .sh + lib/maven_utils.sh + lib/jacoco_utils.sh（成孤儿）。.gitignore 加宽匹配 **/__pycache__/。最终结果：.pre-commit-config.yaml 一文件描述全部本地门控，无 sh 包装。

1. 背景与现状

v0.3 Dev-Ready 的工程基线链条上，PAP-10 已落地 Gradle 化 CI，PAP-15 已落地 main 分支保护 + Required Status Checks + Dependabot。本 issue（PAP-16）把"本地开发机的 commit 时门控"补齐——这是单人开发者每天面对最高频的反馈回路，必须既快又稳。

Linear AC 原文叙述与仓库实际状态存在差异，必须先校准事实再设计。

1.2 与 Linear AC 表述的差异

Linear PAP-16 AC 写 "审计现有 .pre-commit-config.yaml，移除引用 Maven 的 hook" —— 但 .pre-commit-config.yaml 全文件被注释，根本没有"在生效的 Maven hook"可以移除。实际工作是整文件改写（立新）而非"重整"：从一个被注释禁用的旧配置，变成一个全新设计的活配置。本 spec 把 Linear AC 的精神（"删 Maven 残留 + 加 commitlint + 增量格式化检查"）翻译为基于真实仓库状态的执行计划。

3. 业界调研采纳：为什么 spotbugs 不进 hook

Linear AC 原文写 "新增 spotless / spotbugs 增量检查 hook（仅变更模块，调用 Gradle）"。设计阶段对此条做了一次有针对性的业界调研，结论与 AC 原文相左，本 spec 采纳调研结论并显式修订 AC 措辞。

3.1 调研要点

  • Spring Boot 官方 git/hooks/ 目录只有 forward-merge 一个脚本（自动建 GitHub issue），无任何静态分析 hook。
  • Elasticsearch / Apache 系大型 Java OSS 同理：本地 hooks 只做轻量门控（格式、commit-msg），SpotBugs / SonarQube 等静态分析全部交给 CI。
  • 2024–2026 共识：本地 hook 执行 > 几秒 = 坏体验，会被 git commit --no-verify 绕过，反而养成绕过习惯。
  • SpotBugs 单模块典型耗时 10–60s，需完整编译产物，且误报率不低。
  • 对单人 greenfield 项目：CI 已是不可绕过的强制门控（PAP-15 落地），本地重复跑同一检查零增量价值。

3.2 采纳结论

采纳
  • spotless（格式化，auto-apply，秒级）→ pre-commit 阶段。
  • commitlint（commit message 校验，毫秒级）→ commit-msg 阶段。
  • spotbugs（静态分析）→ CI only，本地任何阶段都不挂。
  • pre-push 阶段 → 不挂任何 hook（pre-commit 框架仍 install pre-push 入口文件，但配置中无 pre-push hook，触发时立即放行）。

这条调整不削弱质量保证：CI 的 spotbugsMain 仍在 PR 合并前强制执行（PAP-15 Required Status Checks 已绑定）。

5. .pre-commit-config.yaml 设计

整文件改写（立新）。结构：顶部全局配置 → 4 个 repos: 段。

default_install_hook_types: [pre-commit, commit-msg]
default_stages: [pre-commit]
fail_fast: false
minimum_pre_commit_version: "3.5.0"

repos:
  # 1) 通用 hygiene
  - repo: https://github.com/pre-commit/pre-commit-hooks
    rev: v5.0.0
    hooks:
      - id: check-merge-conflict
      - id: check-yaml
        exclude: ^docs/patra/specs/
      - id: check-json
      - id: check-toml
      - id: check-added-large-files
        args: ['--maxkb=2000']
      - id: detect-private-key
      - id: end-of-file-fixer
      - id: trailing-whitespace
      - id: mixed-line-ending
        args: ['--fix=lf']

  # 2) Secret scanning
  - repo: https://github.com/gitleaks/gitleaks
    rev: v8.21.2
    hooks:
      - id: gitleaks

  # 3) Java 格式化（auto-apply）
  - repo: local
    hooks:
      - id: spotless-apply
        name: spotless apply (Java + gradle.kts)
        entry: ./gradlew spotlessApply -q --no-daemon
        language: system
        pass_filenames: false
        files: '\.java$|\.gradle\.kts$'
        stages: [pre-commit]

  # 4) commit-msg：commitlint
  - repo: https://github.com/alessandrojcm/commitlint-pre-commit-hook
    rev: v9.18.0
    hooks:
      - id: commitlint
        stages: [commit-msg]
        additional_dependencies: ['@commitlint/config-conventional@19.5.0']

5.1 设计要点

  • spotless 走 language: system + local hook，调 gradlew。pass_filenames: false 是因为 Gradle 任务不接受文件参数；files: '\.java$|\.gradle\.kts$' 对齐 build-logic/src/main/kotlin/linqibin.java-base.gradle.kts 中 spotless 实际 target（java { target("src/**/*.java") } + kotlinGradle { target("*.gradle.kts") }）——避免改 Markdown / 资源文件时触发空跑。
  • --no-daemon 避免开发机 daemon 与 IDE Gradle daemon 抢端口；冷启动慢 1–2s，可接受。
  • hook 执行后修改文件 → 框架自动报错 "files were modified by this hook"（标准行为）→ 开发者需 git add 重 commit。这与现有手写脚本的"悄悄 git add"行为相比，更显式更安全。
  • commitlint 走 alessandrojcm/commitlint-pre-commit-hook，自带 Node 容器，根仓库不引入 package.json / node_modules。
  • gitleaks v8.21.2：使用其官方仓库的 pre-commit hook（默认对 staged 区扫描）。
  • 不要 codespell：中文 commit/docs 占主，对英文 typo 价值不高。

6. commitlint 规则（commitlint.config.cjs）

// commitlint.config.cjs
module.exports = {
  extends: ['@commitlint/config-conventional'],
  rules: {
    'type-enum': [2, 'always', [
      'feat', 'fix', 'docs', 'refactor',
      'test', 'chore', 'perf', 'style', 'ci',
      'build',
    ]],
    'type-case': [2, 'always', 'lower-case'],
    'type-empty': [2, 'never'],
    'subject-empty': [2, 'never'],
    'subject-full-stop': [2, 'never', '.'],
    'header-max-length': [2, 'always', 100],
    'scope-case': [2, 'always', 'lower-case'],
  },
};

6.1 规则取舍说明

  • type-enum：Linear AC 给的 9 种 + build（Gradle 相关 commit 已经在用，例如最近的 build(gradle): settings.gradle.kts...）。
  • header-max-length: 100：默认 72 对中文 commit 过紧（中文 commit 经常需要 60+ 字符），100 是社区常见折衷。
  • scope 不强制白名单：绿地阶段模块还在演进（v0.3 仍在 Dev-Ready），强制 scope 白名单会频繁打断。仅要求 scope-case: lower-case。
  • 不强制 PAP-XX 前缀：Linear 的 issue 关联走 branch name + PR title 自动匹配（PAP-15 PR #62 已验证生效）。强制 commit 前缀会大量打断重构 / 文档 / 配置类 commit。未来若确实需要，单独开 issue 加 references-empty: [2, 'never'] 与自定义 parser。

7.2 .gitignore 改动

# 在 "Build outputs" 之后 / "Gradle" 之前合适位置
+
+##############################
+# Python script caches
+##############################
+**/__pycache__/

用宽匹配 **/__pycache__/（而非 Linear AC 的窄路径 scripts/__pycache__/）：pre-commit 框架自身、portal/migrate-javadoc 等多个位置都会产生 __pycache__，一次到位避免后续二改。AC D6 的 git check-ignore scripts/__pycache__/ 仍因 **/__pycache__/ 匹配返回 0。

7.3 .git/hooks/pre-commit 与 .git/hooks/pre-push

这两个文件不在 git 跟踪范围（.git/ 全部 ignored），不能"删 commit"。处理方式：本地执行 pre-commit install --install-hooks --overwrite，该命令会自动覆盖现有手写 pre-commit 与老 pre-push stub，替换为 pre-commit 框架的入口。

8. 落地步骤与 AC D6 验证

8.1 落地步骤（实施期顺序）

  1. 整文件改写 .pre-commit-config.yaml（第 5 节内容）。
  2. 新建 commitlint.config.cjs（第 6 节内容）。
  3. 删除 scripts/git/ 整个目录（git rm -r scripts/git/）。
  4. 追加 .gitignore：**/__pycache__/。
  5. 本地执行 pre-commit install --install-hooks --overwrite。
  6. 跑 pre-commit run --all-files 做首次清扫（hygiene hooks 会修一批 EOL / trailing-ws），首次清扫单独 commit（不与功能改动混 PR）。
  7. 跑 8 步 AC D6 验证（下表）。
  8. 开 PR，等 CI 全绿合并。

8.2 AC D6 验证矩阵

#命令期望
1 pre-commit install --install-hooks --overwrite 输出 pre-commit installed at .git/hooks/pre-commit + .git/hooks/commit-msg
2 pre-commit run --all-files 全部 hooks 跑一遍，首次会修一批 EOL/trailing-ws

8 开 PR，等 CI 跑完 spotlessCheck 通过 + spotbugsMain 通过（回归保护）

9. 关键决策与备选

决策 1：spotbugs 完全移出本地 hook，只在 CI 跑

本地任何 stage 都不挂 spotbugs。CI 的 spotbugsMain + PAP-15 Required Status Checks 是唯一强制门控。

备选 A：pre-push 跑 spotbugsChanged（仅变更模块）— 否决：业界调研显示 Spring Boot 等无此实践；本地 push 慢，会被 --no-verify 绕过；CI 已是强制门控，本地重复零增量价值。
备选 B：pre-commit 跑 spotbugs — 否决：commit 阶段几十秒延迟会被频繁绕过，反而劣化质量；唯一可能的好处（更早反馈）被 CI 反馈周期 < 5 分钟覆盖。

决策 2：commitlint 走 alessandrojcm/commitlint-pre-commit-hook

用 pre-commit 框架原生的 commitlint hook（自带 Node 容器）。根仓库不引入 package.json / node_modules。

备选 A：根仓库引 @commitlint/cli + husky — 否决：根仓库出现 Node 生态产物（package.json / node_modules），与 Java-first monorepo 结构混杂；且与 portal 的 husky 形成双套路。
备选 B：commitlint-rs 单二进制 — 否决：每台开发机要自行装 Rust 二进制，跨平台麻烦，价值不抵成本。

决策 3：spotless 走 ./gradlew spotlessApply 全量，不引 ratchetFrom

local hook 调用 ./gradlew spotlessApply -q --no-daemon，信任 Gradle 任务级 UP-TO-DATE 缓存。

备选 A：build-logic 配置 spotless { ratchetFrom 'origin/main' }（文件级增量）— 否决：在 detached HEAD / shallow clone（如 CI runner 默认 fetch-depth:1）下可能失败；与 spotbugs 不再挂 hook 后，对称性需求消失，仅留复杂度。
备选 B：build-logic 写聚合任务 spotlessChanged（仅变更模块）— 否决：spotbugs 不再挂 hook，对称性需求消失；spotless 任务本身已极轻量（秒级），全量已足够。

决策 4：.gitignore 用宽匹配 **/__pycache__/

一次到位匹配所有路径下的 Python 缓存。Linear AC 的窄匹配 scripts/__pycache__/ 仍命中（被宽匹配覆盖）。

备选：严格按 AC 只加 scripts/__pycache__/ — 否决：pre-commit 框架、portal 内的 Python 工具、migrate-javadoc 脚本都会产生 __pycache__，窄匹配后续需补充，浪费一次改动机会。

决策 5：portal /.husky/ 不动

当前不生效（被根 .git/hooks/pre-commit 手写脚本覆盖），保留待未来独立 issue 处理。本 issue 严格保持 surgical changes，不扩散到 portal。

备选：顺手删 portal husky + 把 portal lint/typecheck 迁到根 pre-commit 框架 — 否决：超出 PAP-16 范围；portal 工具链（biome / typecheck）与 Java hook 的混编需独立设计（例如 lint-staged 子目录过滤策略）。

决策 6：首次 pre-commit run --all-files 清扫单独 commit

hygiene hooks（end-of-file-fixer / trailing-whitespace / mixed-line-ending）首次跑会改一批文件，单独成 commit，不与本 issue 主要功能改动混 PR diff，便于审查。

备选：合并到一个 commit — 否决：PR diff 中 hygiene 噪音会淹没核心配置改动，审查体验差。

10. 风险与缓解

风险等级缓解
首次 pre-commit run --all-files 改一大批文件（EOL/trailing-ws），PR diff 难审查 mid 实施步骤明确将首次清扫单独成 commit；主功能改动 commit 不含 hygiene 噪音。
./gradlew spotlessApply --no-daemon 冷启动 1–2s，commit 体验略慢 low 接受。仍符合 < 3s 体验目标。后续若 daemon 与 IDE 抢端口问题不严重可考虑去 --no-daemon。
