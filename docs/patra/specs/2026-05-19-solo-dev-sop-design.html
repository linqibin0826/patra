Patra Solo Dev SOP

📅 2026-05-19 🌿 chore/solo-dev-sop 🎯 spec draft
关键决策摘要

为单人完成"产品 → UI/UX → BE → FE → QA"全链路制定清晰可执行的 SOP。核心决策：Linear 作为单一信息源（Project / Issue / Sub-Issue 三层）；Issue 状态机使用 Linear 默认 Backlog → Todo → In Progress → In Review → Done 5 态（对齐 Linear Method "Write issues, not user stories"）；跨界功能拆 area:be / area:fe / area:design 多个独立 Issue，阶段顺序由 Project Milestone 表达；三层质量门（Husky / CodeRabbit CLI / CodeRabbit App）自动守门；Claude Code 跨会话上下文恢复只依赖 Linear Issue + Sub-Issue。Issue 字段（AC / Design URL / PR 链接 / Review Outcome）均通过 Issue Description 模板、Attachments 和 Label Group 实现，无需付费 Custom Fields。

1. 背景与现状

Patra 是单人开发的医学出版物数据平台，开发者兼任产品、UI/UX、前端、后端、QA 五个角色，存在以下四类痛点：

  1. 角色切换累：5 个角色都是自己，频繁切换上下文消耗精力
  2. 优先级乱：能做的事太多，每天不知道先动哪个
  3. 质量没人把关：自测靠不住，缺少独立 review 视角
  4. 节奏失控 + 上下文易丢：无外部督促易拖延；几天后回头看自己的代码 / 决策要重新理解

本 SOP 针对以上四类痛点，通过五条设计原则建立一套可持续执行的全链路开发流程：

  • AI 主导执行，人专注决策：你是决策者 + 把关者，Claude 是执行层；介入点在"起点（需求）"和"终点（review）"
  • Linear 是单一信息源：所有 feature 生命周期状态挂在 Linear，Claude 跨会话恢复上下文只看 Linear
  • 事件驱动而非时间驱动：节奏由 Issue 状态变化触发，不预设"周几做什么"
  • 故意粗糙以聚焦：PRD 阶段视觉故意粗糙（防完成感幻觉），UI 阶段直接 hi-fi（AI 时代低保真过时）
  • 回路成本最小化：默认串行顺序选择回路代价最低的路径（消费外部 API 项目 BE 先）

2. 范围边界

In Scope
  • Project 层：产品维度版本规划 + Issue 池派生
  • Issue 层：Linear 默认 5 态状态机（Backlog → Todo → In Progress → In Review → Done）+ area:be / area:fe / area:design 分流 + Project Milestone 表达阶段顺序
  • 横切机制：上下文管理、中断恢复、回路、质量门、节奏
  • 补丁路径：fast-track、design-first
  • 工具栈选型与落地清单
Out of Scope
  • 产品定位 / 市场策略 / 竞品分析方法论（自用，不需要）
  • 团队协作流程（团队只有你一个人）
  • E2E 测试体系（成本不对称，先不上）
  • 用户运营 / 增长流程（无用户阶段不需要）
  • 持续部署管道（已有 GitHub Actions，本 SOP 不涉及）

3. 双层架构

SOP 由两个层次组成，Project 层管产品维度，Issue 层管工程维度，两层通过 Issue 池连通：

┌─────────────────────────────────────────────────────────┐
│  Project 层（产品维度）                                  │
│  Draft → Active → Done                                  │
│  · Entry Gate: 写完版本规划文档 + 派生 Issue 池          │
│  · 产物: docs/patra/release-specs/<version>.md          │
└──────────────────────┬──────────────────────────────────┘
                       │ Issue 池
                       ↓
┌─────────────────────────────────────────────────────────┐
│  Issue 层（工程维度）                                    │
│  每个 Issue 走 Linear 默认 5 态状态机：                  │
│  Backlog → Todo → In Progress → In Review → Done        │
│  跨界功能（BE+FE+Design）拆多 Issue，                    │
│  阶段顺序用 Project Milestone 表达                       │
│  · 产物: docs/patra/specs/<topic>-design.html +         │
│         GitHub PR(s) + Linear Sub-Issues（按需）        │
└─────────────────────────────────────────────────────────┘

Linear 三层结构对应关系（对齐 Linear Method）：

层级含义状态数量
Project 大版本 / Release（如 v0.2 PubMed 搜索）；Description = 5 字段 release spec；Milestone 表达阶段顺序 Draft / Active / Done 1（每版本 1 个）
Issue 具体工程任务（如"实现 GET /api/v1/health"、"portal Health 卡片 + 30s 轮询"）；按 area:* 标签分流到 BE / FE / Design Backlog / Todo / In Progress / In Review / Done 多个/版本，每个 ≈ 1 个 PR（几小时-几天）
Sub-Issue 单 Issue 内部多步追踪（如"中断快照"、"Review 反馈 R1/R2"）；与父 Issue 共用同一状态机（Linear 限制）；按需使用，非默认派生层级 Backlog / Todo / In Progress / In Review / Done 0-N / Issue

4. Project 层

4.1 状态机

[想法] → Draft → Active → Done
              ↑
        Entry Gate: 完成版本规划文档 + Issue 池派生
状态含义出口条件
Draft 你在写版本规划文档 Project Entry Gate 通过：① 5 字段全填 ② Issue 池已派生
Active 文档完成，Issue 池开始走 SOP 所有 Issue 推到 Done
Done 整个版本完成 （可选）retro 沉淀到 Obsidian

4.2 版本规划文档（5 字段）

产物路径：docs/patra/release-specs/<version>.md

工具：Claude Code + patra:release-planning skill

5 字段模板：

# <Version> 版本规划

## 1. 版本目标 / 价值
1-2 句话描述这版的核心交付价值。

## 2. 工程任务列表 / 要做的事
bulleted list，每条 = 1 个具体工程任务（按 area:be / area:fe / area:design 分流）→ 派生 1 个 Linear Issue（≈ 1 个 PR）。
跨界功能拆多个 Issue，阶段顺序用 Milestone 表达。

## 3. 范围边界 / 不做的事
明确"不做"以防 scope creep。

## 4. 关键决策 / 技术取舍
跨 Issue 的架构决策。

## 5. Done 判定
这版"完成"的客观标准。

4.3 Project Entry Gate（Draft → Active）

进入 Active 必须满足：

  1. 5 字段全部填完
  2. 第 2 字段每条工程任务 → 已生成 1 个 Linear Issue（Description 按 Magical.pm Everyday 三字段：Overview / To-Do / Acceptance Criteria），并打上 area:* 标签
  3. 跨界功能已拆为多个独立 Issue + Project Milestone 表达阶段顺序（默认 BE Ready → Design Ready → FE Ready → Released）
  4. 第 3 字段"不做的事"显式列出
  5. 第 5 字段 Done 判定可验证
Draft 红线

Draft 阶段只产出版本规划文档 + Issue 池，禁止写实现代码。在 Draft 阶段开始写代码是常见的 scope creep 来源。

5. Issue 层

5.1 状态机（Linear 默认 5 态）

Backlog → Todo → In Progress → In Review → Done
                                     │
                                     └── No-Go → 退回 In Progress
                                                 （创建 Sub-Issue「Review 反馈 R1/R2/R3」）
为何不再用六阶段状态机

v0.1 SOP 设计把"Tech Design / BE / Design / FE / Review"作为 Issue 状态，与 Linear Method 哲学冲突（Linear Method 明确反对把工程阶段塞进 Issue 状态机：Write issues, not user stories），且 v0.2 SOP Trial 实测拆 BE/FE 多 Issue 后状态机自动失效（PAP-5 永远走不到 Design/FE）。详见决策 4（重写）。

状态主导关键工具 / Skill入口条件出口条件（DoD）
Backlog — Linear Project Active 时由 release-planning skill 派生 你 pick 该 Issue 准备开始
Todo — Linear 从 Backlog pick 出来（已确定下一步要做） 你打算立刻开做（或等前置 Milestone 解锁）
In Progress 视 area:* 标签分流 area:be → patra-* skills + TDD
area:fe → openapi-typescript + Handoff
area:design → Claude Design + Handoff 		你开始动手；前置 Milestone 已通过 产物完成（PR 已开 / Design 稿已交付）；AC 自查全部满足
In Review CodeRabbit + 你 Go/No-Go CodeRabbit CLI + GitHub App PR 已开（或 Design 稿已交付） CodeRabbit 评论 100% 处理 + 自测三件套通过 + PR merged + review:pass 标签
Done — — In Review 通过 + PR merged 触发后续 Milestone / 进入下一个 Issue

Tech Design 不再是 Issue 状态——Tech Design 作为"开发活动"在 Project 启动期一次性产出（见 §6.1），产物 docs/patra/specs/<topic>-design.html 由各 area Issue 在 Description 里引用。fast-track Issue 可跳过 Tech Design 产物（见 §9.1）。

5.2 Issue 上下文字段（按 area 分布）

跨会话恢复的核心来源

Claude 新会话恢复上下文只看三处：① Linear Issue Description（Overview + To-Do + AC）② Description 里的链接 / Attachments / 标签 ③ 关联 GitHub PR + Sub-Issues（如有）。

模型变化：v0.1 SOP 假设单个 Issue 内含 BE PR + FE PR + Design URL（与"用户故事 = 1 Issue"绑定）；v0.2 起每个 Issue 是按 area:* 分流的独立工程任务，字段按 area 分布：

信息适用 Issue（area）填写时机实现方式
Acceptance Criteria 全部 Issue 创建（最迟 Todo → In Progress 前） Issue Description 里 - [ ] checklist（≥ 3 条可验证），遵循 Magical.pm Everyday 三字段模板
Tech Design 文档链接 area:be / area:fe（如 Project 有 Tech Design 产物） Issue 创建 Description 链接到 docs/patra/specs/<topic>-design.html
Design URL + Handoff token area:design Issue（产出）/ area:fe Issue（消费） Design Issue 进 In Review 时 / FE Issue 进 In Progress 时 Linear Attachments + Description（粘贴 Claude Design URL）
PR 链接 area:be / area:fe Issue In Progress 末尾（PR 创建时） GitHub Integration 自动检测 + Description 补充 PR 链接
Review Outcome 全部 In Review → Done Label Group review（review:pass / review:conditional / review:reject）+ Review Outcome 评论

5.3 Issue 标签分流

v0.2 起新增 area:* Label Group 作为开发活动分流锚点（替代旧六阶段状态机）；flow:design-first 标签删除（用 Project Milestone 表达 Design Issue 顺序）。

标签含义用途
area:be 该 Issue 产出后端实现 In Progress 时按 §6.2 BE Issue 流程
area:fe 该 Issue 产出前端实现 In Progress 时按 §6.4 FE Issue 流程
area:design 该 Issue 产出 UI/UX 设计稿 In Progress 时按 §6.3 Design Issue 流程
path:full（默认） 该 Issue 派生自 Project Tech Design 文档 默认所有非小修 Issue
path:fast-track 跳过 Project Tech Design 产出，直接创建 area:* Issue → In Progress → Review typo / UI 微调 / 纯重构 / 配置升级 / BUG fix（必须先写复现测试）
paused:<原因> Issue 暂停 配合 Sub-Issue「中断快照」使用
review:pass / review:conditional / review:reject In Review Go/No-Go 结果 Label Group review，替代 Custom Fields 中 Review Outcome 字段

6. 开发活动详细定义

章节定位变化

v0.1 SOP 把本章定为"Issue 状态阶段"——六阶段每段对应一个 Issue 状态。v0.2 重构后 Issue 状态机精简为 5 态（见 §5.1），本章重新定位为开发活动：§6.1 Tech Design 是 Project 启动期一次性产物；§6.2~§6.4 是按 area:* 分流的 Issue 处理流程（每个 area:* Issue 走自己的小节）；§6.5 是所有 Issue 进入 In Review 时的统一流程。

6.1 Tech Design 活动（Project 启动期一次性完成）

入口：Project 进 Active 后、首个 area:be / area:fe Issue 开始之前 | 主导：Claude Code（起草细化）+ 你（架构决策）

流程：

  1. 在 Claude Code 里加载 patra:brainstorming，输入 Project Description（5 字段 release spec）+ 当前 area:be / area:fe Issue 列表
  2. Claude 通过对话探索技术方案、领域模型、外部 API 调研、字段清单初稿
  3. 用 patra:writing-plans 产出实施计划
  4. 产物路径：docs/patra/specs/YYYY-MM-DD-<topic>-design.html
  5. 把细化后的 AC 拆解到各 area:* Issue 的 Description checklist
  6. 确认每个 Issue 的 path:* + area:* 标签

DoD：

  • 技术方案明确（领域模型 + 接口契约 + 数据流）
  • API 字段清单初稿（用于 area:design / area:fe Issue 参考）
  • 所有 area:* Issue 的 AC ≥ 3 条可验证，已写入 Description
  • 实施计划分步
  • path / area 标签确定

fast-track 豁免：打 path:fast-track 标签的 Issue 可跳过本活动（不产 Tech Design 文档），直接创建 area:* Issue → In Progress。

6.2 BE Issue 处理流程（area:be）

入口：area:be Issue 推到 In Progress（前置 Milestone "BE Ready" 未通过的前驱 Issue 全部 Done）| 主导：Claude Code + patra-* skill 体系；你做架构决策

流程：

  1. 加载 patra:test-driven-development（TDD 主流程，强制）
  2. 架构决策 / 创建组件 → 加载 patra-hexagonal
  3. 数据持久化 → 加载 patra-jpa
  4. 事件 / Outbox → 加载 patra-events
  5. 如发现字段变化，立即回写 Tech Design 文档字段清单（保持 SoT 同步）
  6. Bruno collection 同步更新（API 契约入 git）
  7. OpenAPI schema 自动暴露（patra-spring-boot-starter-openapi）
  8. 创建 GitHub PR；Issue Description 补充 PR 链接；Issue 推到 In Review

DoD（出 In Progress 的条件）：

  • 单元测试 + 集成测试全部 GREEN
  • AC 在 BE 层可验证（curl / Bruno 跑通）
  • Bruno collection 已更新并入 PR
  • OpenAPI schema 已 sync
  • PR 链接已补充到 Issue Description

6.3 Design Issue 处理流程（area:design）

入口：area:design Issue 推到 In Progress（默认在前置 area:be Issue Done 后；Milestone "Design Ready" 表达此顺序）| 主导：Claude Design 生成 + 你 review 微调

跳过 wireframe 阶段

AI 时代 hi-fi 成本接近零，思考流程在 Tech Design 已完成；wireframe 反而易产生"完成感幻觉"。直接生成可点击 hi-fi 原型。

流程：

  1. 打开 Claude Design，新建 design session
  2. 输入：Issue Description + Tech Design 字段清单 + BE OpenAPI schema（如已完成）
  3. Claude Design 生成可点击的 hi-fi 原型
  4. 你 review 并对话迭代到主要 user flow 可点击
  5. 发起 Handoff to Claude Code，获得 token
  6. Design URL + Handoff token 粘贴到本 Issue Attachments + Description
  7. Issue 推到 In Review（你自审 hi-fi 可点击度）→ Done 后触发 Milestone "Design Ready"

DoD（出 In Progress 的条件）：

  • 主要 user flow 在 Claude Design 中可点击
  • 每条 AC 在 UI 上能找到对应入口
  • Handoff token 已生成
  • Design URL + Handoff token 已补充到 Issue Attachments / Description

6.4 FE Issue 处理流程（area:fe）

入口：area:fe Issue 推到 In Progress（Milestone "Design Ready" 通过后，即前置 area:design Issue Done）| 主导：Claude Code 消费 Handoff + 你 review

流程：

  1. 在 Claude Code 把前置 area:design Issue 的 Handoff token 喂给 Claude，翻译到前端栈
  2. 跑 pnpm gen:bootstrap（一键 sync）：gen:types（openapi-typescript）+ gen:hooks（openapi-react-query）+ gen:zod（openapi-zod-client）
  3. 实现页面 / 组件 / Zustand 状态 / RHF + Zod 表单
  4. 用 Bruno collection 做本地联调（数据真实形态）
  5. Vitest + RTL 写关键交互单测
  6. Husky pre-commit：Biome + tsc --noEmit + Vitest 自动跑
  7. 创建 GitHub PR；Issue Description 补充 PR 链接；Issue 推到 In Review

DoD（出 In Progress 的条件）：

  • UI 还原 ≥ 95% 与 Claude Design 一致
  • 关键交互可手动跑通
  • 类型 100% 来自 openapi-typescript（零手写 API 类型）
  • Vitest + RTL 测试覆盖关键交互
  • PR 链接已补充到 Issue Description

6.5 Issue Review 流程（In Review → Done）

入口：任一 Issue 推到 In Review（PR 已开 / Design 稿已交付）| 主导：CodeRabbit（守门）+ 你做 Go/No-Go

Review 流程：

  1. PR 创建 → CodeRabbit 自动 review，发 inline 评论 + summary
  2. 你 review 评论：每条处理（修复 / "why 不改"回复 / 关闭）
  3. 自测三件套：跑 Bruno collection（API 契约验证）+ 手测 UI flow（与 AC 一一对照）+ 检查 Sentry dashboard（如已引入）
  4. Go/No-Go 决策：
    • Go → merge PR；打 review:pass 标签；Issue 推到 Done
    • No-Go → 打 review:reject 或 review:conditional 标签；Issue 退回 In Progress；创建 Sub-Issue「Review 反馈 Rn」

Review Outcome 评论格式（写在 Issue 评论 / Sub-Issue Description）：

CodeRabbit: 12 条，已处理 12
自测: Bruno ✓ | UI flow ✓ | AC 1/2/3 ✓
决策: Go @ 2026-05-19 14:30

DoD（出 In Review → Done 的条件）：

  • CodeRabbit 评论 100% 已处理（修复或解释）
  • AC 全部手测通过
  • review:pass 标签已挂上
  • 关联 GitHub PR 已 merge（area:design Issue 例外：Design 稿交付即可）

8. 横切机制

8.1 上下文管理

Claude 新会话恢复上下文只看三处：

  1. Linear Issue Description（Overview + To-Do + AC checklist）+ area:* / path:* 标签
  2. Description / Attachments 里的链接（Tech Design 文档 / Design URL / PR 链接）+ review:* 标签
  3. 关联 GitHub PR + Sub-Issues（按需：中断快照 / Review 反馈循环）+ Project Description（5 字段 release spec）+ 同 Project 内其他 Issue 状态

项目级规则由 CLAUDE.md + .claude/rules/ + skill 体系承载（已完备）。Claude 禁止在 Linear 之外另起"记忆库"；每次新会话先读 Issue 状态再决策。

8.2 中断恢复

暂停 Issue：

  1. Issue 加标签 paused:<原因>（如 paused:waiting-pubmed-api-quota）
  2. 创建 Sub-Issue「中断快照」记录：当前状态（哪一阶段）+ 下一步具体动作（≤ 3 条）+ 关键决策（选 X 而非 Y 的理由）+ 引用的代码 / 文档 / Bruno collection link
  3. 关闭 Claude Code 会话

恢复 Issue：

  1. 打开 Linear Issue，读「中断快照」Sub-Issue
  2. 把 Sub-Issue 内容粘贴到 Claude Code 新会话作为起始上下文
  3. Claude 自动恢复 → 删除 paused: 标签 + 关闭「中断快照」Sub-Issue

8.3 回路机制（No-Go 处理）

  1. Issue 状态退到出问题阶段（BE / FE / Design / Tech Design）
  2. 创建 Sub-Issue「Review 反馈 R1」（R1/R2/R3… 递增）记录每轮反馈
  3. 不允许 Skip Review：再次到 Review 要重跑三层门
升级机制

R3 仍 No-Go → 强制升级到 Tech Design 重审，从头梳理（避免在低层反复修补）；新增 Sub-Issue「Tech Design v2」记录重审原因。

8.4 三层质量门

层触发时机工具检查内容
1 pre-commit（每次 git commit） Husky + lint-staged FE: Biome + tsc --noEmit + Vitest / BE: .pre-commit-config.yaml
2 push 前（每次 git push） CodeRabbit CLI 本地快审，找语义 / 漏边界
3 PR 后 / merge 前 CodeRabbit GitHub App 自动 inline review + summary + sequence diagram

8.5 节奏（事件驱动）

不预设"周几做什么"。SOP 节奏由"Project Active / Issue 进 In Progress（按 area 分流）/ Issue 进 In Review"三类事件驱动：

触发事件你的工作模式
Project Active（首次）Claude Code + patra:brainstorming + patra:writing-plans 写 Project 级 Tech Design 文档（除非全部 Issue 都打 path:fast-track）
area:be & In ProgressClaude Code + patra-* skill + TDD
area:design & In ProgressClaude Design + Handoff
area:fe & In Progresspatra-portal + Handoff 消费 + pnpm gen:bootstrap
任一 Issue & In ReviewGitHub PR + CodeRabbit + 自测三件套

避免角色切换的策略：

  • Batch by stage：等多个 Issue 进同一阶段再统一处理，减少切换成本
  • 单 in-progress 原则：单时段只让 1 个 Issue 处于活跃实施（FE / BE 状态）
  • paused: + 中断快照：临时不做的 Issue 立即快照暂停
  • Retro 触发条件：每完成 N 个 Issue、或遇 Review 回路 ≥ R2、或 SOP 阻塞超 2 天时，写一次 retro 沉淀到 Obsidian

9. 补丁路径

9.1 path:fast-track（小修小 enhance）

触发：Issue 创建时打 path:fast-track 标签（同时仍需打 area:* 决定走哪条 §6 流程）。

精简流程：跳过 Project 级 Tech Design 文档 + 不派生 area:design Issue，直接创建 area:be 或 area:fe Issue → In Progress → In Review → Done。

适用白名单
  • typo / 措辞 / 文案
  • UI 微调（border、间距、颜色）
  • 纯重构（行为不变 + 测试 GREEN）
  • 配置 / 依赖升级
  • BUG 修复（必须先写复现测试）
fast-track 不豁免
  • CodeRabbit 双档审查
  • tsc / Vitest / 集成测试必须 GREEN
  • Review Outcome 标签仍需打

fast-track Issue 可独立存在，不必挂任何 Project（直接挂在 Linear Team / Workspace 下），适合零散小修。

9.2 Project Milestone 表达阶段顺序（替代旧 flow:design-first）

v0.2 起 flow:design-first 标签删除。Project 内 area:* Issue 的执行顺序统一用 Project Milestone 表达：

  • 默认顺序：BE Ready → Design Ready → FE Ready → Released（消费外部 API 项目 BE 先 = 字段稳定 = 回路最少）
  • Design-first 场景（UI 探索性强 / 数据完全自定义 / 对外部 API 已非常熟悉）：直接调整 Milestone 顺序为 Design Ready → BE Ready → FE Ready → Released，不需要标签开关
  • Milestone 内的 Issue 全部 Done 后该 Milestone 通过；下一 Milestone 的 Issue 解锁可以进 Todo
默认 BE 先的理由

Patra 是消费外部 API 项目，字段稳定性来自真实调用；BE 先 = 回路最少。Design-first 是 escape hatch，通过调整 Milestone 顺序实现，无需额外标签。

10. 关键决策与备选

决策 1：SOP 主轴 — Feature 全生命周期

选择"Feature 全生命周期"而非节奏切片或角色协议作为 SOP 主轴。

备选 A：节奏切片（每周一做 BE、周三做 FE…）— 否决：无外部督促的单人项目，节奏切片会制造人工压力，与"无时间压力"项目语境矛盾。
备选 B：角色协议（Claude 永远是工程师，你永远是 PM）— 否决：边界太死，无法适应"你既是 PM 又是 reviewer"的现实。

决策 2：任务卡载体 — Linear（非 GitHub Issues / Obsidian Bases）

Linear 的 UI / workflow 足够灵活，三层结构（Project / Issue / Sub-Issue）原生支持，Linear ↔ GitHub 集成成熟，PR 引用 LIN-123 自动关联。

备选 A：GitHub Issues — 否决：缺 Project 层 + Sub-Issue 层，状态机太弱。
备选 B：Obsidian Bases — 否决：离线文件系统，与 GitHub 集成需要额外桥接，复杂度高。

决策 3：Issue 上下文字段 — Description + Attachments + Label Group（非 Custom Fields）

Linear 没有"Plus plan"也没有"Custom Fields"这个功能层次（旧设计文档中的描述有误）。正确实现：AC 用 Issue Template + - [ ] checklist；Design URL / PR 链接用 Description + Attachments；Review Outcome 用 Label Group review（三个标签）。

旧误说法（已废弃）：自定义 5 个 Custom Fields — 错误：Linear 无此功能，且此方式需 API 操作，使用摩擦高。当前实现方式零摩擦，信息同样可查。

决策 4：Issue 状态机使用 Linear 默认 5 态，阶段顺序用 Milestone 表达（v0.2 重写）

Issue 状态机使用 Linear 默认 5 态（Backlog / Todo / In Progress / In Review / Done），跨界功能拆 area:be / area:fe / area:design 多个独立 Issue，阶段顺序用 Project Milestone 表达（默认 BE Ready → Design Ready → FE Ready → Released）。

决策依据（三轮调研）：

  1. Linear 官方：Linear Method 明确反对把工程阶段塞进 Issue 状态机（Write issues, not user stories）；Sub-Issue 与父 Issue 共用同一状态机，不允许独立状态机（parent-and-sub-issues）
  2. 业界主流：跨界功能拆多 Issue + Milestone 是 Reversed Digital、Climate Policy Radar 等团队的公开做法；状态机扩展最常见的就是在默认 4 态基础上加 In Review（多个工程博客一致）
  3. Solo dev 共识：Lunch Pail Labs、morgen.so 等多个 solo dev 公开材料一致建议状态机不超过 4-5 态；Sub-Issue 审慎使用

备选 A（v0.1 SOP：六阶段塞 Issue 状态机 Tech Design → BE → Design → FE → Review → Done） — 否决：与 Linear Method 哲学冲突；v0.2 SOP Trial（Project: v0.2 SOP Trial Health Vertical Slice）实测拆 BE/FE 多 Issue 后状态机自动失效（PAP-5 永远走不到 Design/FE，PAP-6 永远走不到 Tech Design/BE）。
备选 B（保留 flow:design-first 标签翻转 BE/Design 顺序） — 否决：标签表达顺序不如 Milestone 清晰；Project Milestone 是 Linear 原生支持的阶段表达机制，更直观。

决策 5：PRD skill 独立化 — patra:release-planning（非复用 brainstorming）

patra:brainstorming 已为工程 spec 特化（路径绑定 docs/patra/specs/，涵盖"架构/组件/数据流"，HARD-GATE 终点是 writing-plans）。PRD 需要不同的引导逻辑：5 字段对话引导 → 派生 Issue 池 → 终止于 Linear Project Active。强行复用违背"职责清晰"原则。

备选：Claude Design 做 PRD — 否决：Claude Design 不支持 PRD 级抽象生成（文本规划），它专注于 UI 视觉。

决策 6：三层质量门设计 — Husky → CodeRabbit CLI → CodeRabbit App

三档递进：pre-commit 快速本地拦截语法/类型错误；push 前 CLI 做语义审查；PR 后 App 做完整分析（含 sequence diagram）。总成本：CodeRabbit Pro $15/月，ROI 高。

备选：仅 CodeRabbit App — 否决：失去 pre-commit 本地快速反馈，每次要等 PR；Husky 是零成本的早期过滤器。

决策 7：节奏 — 事件驱动（非时间驱动）

SOP 节奏由 Issue 状态变化触发，不预设"周几做什么"。匹配"无时间压力"的单人绿地项目语境。

备选：固定 sprint（1 周 / 2 周） — 否决：制造人工外压，与项目特征不符；sprint 规划会议 overhead 对单人无意义。

决策 8：Sub-Issue 不进默认派生流程（v0.2 新增）

Sub-Issue 仅在单 Issue 自身需要分多步追踪时按需创建，不作为 patra:release-planning skill 的默认派生层级。典型用法：「中断快照」记录暂停状态、「Review 反馈 R1/R2/R3」记录回路反馈。

决策依据：

  1. Linear 官方限制：Sub-Issue 与父 Issue 共用同一状态机，无法承载独立"阶段"语义（parent-and-sub-issues）
  2. Solo dev 共识：Lunch Pail Labs（来源）、xavd.id（来源）、Commonwealth Creative（来源）一致建议 solo dev 默认保持 Issue 扁平结构，能不拆就不拆
  3. 业界使用习惯：典型团队 1 个 Issue 下 2-5 个 Sub-Issue，仅在"工作太大但不足以独立成 Project"时拆

判断规则：何时拆 Sub-Issue —— ① 单 Issue 工作量超过 1 个 PR / 几天还做不完；② 单 Issue 内有几个清晰可独立验证的子步骤；③ 中断快照 / Review 反馈循环记录。

备选（v0.1 SOP：Sub-Issue 作为单阶段产物的默认派生） — 否决：v0.1 把 Sub-Issue 定义为「Tech Design / BE PR / UI 稿 / FE PR / Review 反馈」六类产物，但 Linear 共状态机限制使其无法承载阶段语义；且单人开发场景下增加额外维护成本（Lola/lunchpaillabs 等 solo dev 已验证）。
