Monorepo Claude 配置三层化改造设计

📅 2026-05-19 🌿 chore/claude-config-refactor 🎯 spec draft
关键决策摘要

monorepo 化后 Claude 配置散落在 patra-api/，无 workspace 共享层、patra-portal/ 零配置。本次改造按 Claude Code 原生加载规则建立三层结构（workspace 根 / BE 子项目 / FE 子项目），cwd 决定加载哪些层。共 10 个文件变更，不动 .claude-marketplace/ 与 BE 配置主体。

1. 背景与现状

monorepo 化完成后（Patra/ 是 git 根，下含 patra-api/、patra-portal/ 两个子项目），Claude 配置呈现以下问题：

  • Patra/.claude/ 是空目录，Patra/CLAUDE.md 只有简短的 layout 表（且未提 patra-portal）—— 没有 workspace 共享层；
  • patra-api/.claude/ 与 patra-api/CLAUDE.md 配置完整，但混合了通用工作原则（绿地、TDD）与 BE 专属内容（Java/Spring/六边形）；
  • patra-api/CLAUDE.md 中"流程层 superpowers"描述已过时—— PR #36 已把 superpowers 物理内置为 patra: plugin（namespace 变更），但 CLAUDE.md/AGENTS.md 未同步更新；
  • patra-portal/ 零 Claude 配置——cwd=patra-portal 时 Claude 没有任何项目上下文。

2. 范围边界

In Scope（本次 PR 做）
  • 新建 Patra/CLAUDE.md（workspace 共享层）
  • 修订 patra-api/CLAUDE.md：删通用部分 + superpowers: → patra: 修正
  • 新建 patra-portal/CLAUDE.md + .claude/rules/ 骨架（4 个文件）
  • 删除所有 AGENTS.md（共 2 个，不维护 Codex）
Out of Scope（后续单独 PR）

  • patra-portal/.claude/skills/ / settings.json / hooks/ YAGNI
  • .github/workflows/ 提升到 monorepo 根 单独 PR
  • create-issue skill 是否提升到根（保守暂留 patra-api）

3. Claude Code 加载规则（决策依据）

本次改造方向受 Claude Code 多层加载机制约束。下表是查证 docs.claude.com 后的权威结论，非推测：

组件加载行为对设计的影响
CLAUDE.md cwd 上溯到 git 根所有层叠加；cwd 下子目录文件被读时按需加载 通用部分提到根，子项目层只写差异——叠加后语义完整
.claude/skills/ 双向发现——cwd 上溯到 git 根 + cwd 下子目录按需 子项目 skills 可不动；cwd=Patra 时编辑子目录文件能用到子目录 skills
.claude/rules/ 跟随 CLAUDE.md 子目录按需机制；paths frontmatter 进一步限制激活范围 BE rules 留在 patra-api/.claude/rules/，cwd=Patra 编辑 BE 文件时按需激活
.claude/agents/ cwd 上溯发现（子目录方向未文档化，推测不可用） agents 应放 cwd 那层或上溯路径中
.claude/settings.json 里的 hooks 硬约束：cwd 那层 settings 才触发——子目录 settings 不会在父 cwd 时激活 BE hooks 必须在 cwd=patra-api 启动时才生效
.claude-marketplace/ 非标准目录，由 extraKnownMarketplaces + enabledPlugins 指向 位置取决于哪层 settings.json 引用它
cwd 工作约定（基于硬约束）

由于 hooks 是硬约束（cwd 那层 settings 才触发），本次改造确立约定：BE 工作 cd 到 patra-api/ 启动 Claude，FE 工作 cd 到 patra-portal/ 启动 Claude。只有跨项目级任务（改 SOP / ADR / 跨项目重构）在 Patra/ 根启动。这种约定让每层配置专注于自己的领域，不需要 path matcher 滤波。

6. patra-portal/.claude/rules/ 骨架

4 个 rule 文件，每个 < 30 行，只列关键约定，具体规则后续真正开发前端时再充实（YAGNI）。

code-style.md — Biome + TS + 命名约定
  • Biome 格式化与 lint 是唯一权威，禁止与 Biome 配置冲突的手动格式化
  • TypeScript strict 模式开启：禁止 any、禁止 // @ts-ignore（用 @ts-expect-error + 注释代替）
  • 命名：组件 PascalCase、hook useXxx、纯函数 camelCase、常量 SCREAMING_SNAKE_CASE
  • 路径别名：@/ 指向 src/，不使用相对路径跨目录引用
component.md — shadcn/ui + Tailwind 习惯
  • UI primitives 优先用 shadcn/ui 安装到 src/components/ui/，禁止从 npm 直接装无 headless 实现的组件库
  • 业务组件放 src/components/<feature>/，单文件单组件原则
  • Tailwind class 顺序按 Biome 插件自动排（不手动调整）
  • 禁止 inline style，必须用 Tailwind 或 CSS variables
state.md — TanStack Query + Zustand 边界
  • 服务端状态（来自 API）一律用 TanStack Query v5：queryKey 集中管理在 src/lib/query-keys.ts
  • 客户端状态（UI 偏好、本地表单暂存等）用 Zustand：每个 store 单一职责
  • 禁止把 API 数据塞进 Zustand，禁止把 UI 状态塞进 TanStack Query
  • 表单状态用 RHF + Zod，不进 store
testing.md — Vitest + Testing Library
  • 测试位置：与被测文件并列 foo.test.tsx，或集中在 tests/（与现有结构一致）
  • 组件测试用 @testing-library/react，查询优先级：getByRole > getByLabelText > getByText，禁止 getByTestId 作为首选
  • 禁止 snapshot 测试（脆弱，无信息价值）
  • 异步交互必须 await Testing Library 的 findBy* 或 waitFor，禁止裸 setTimeout

7. 关键决策与备选

决策 1：cwd 工作模式 — 按需 cd 到子项目

选择"BE/FE 工作时 cd 到对应子项目启动 Claude Code"作为约定，monorepo 根仅用于跨项目级任务。

备选 B：总在 Patra/ 根启动 — 否决：hooks 是硬约束（子目录 settings 不会在父 cwd 时触发），要让根 cwd 也能用就得把所有 hooks 提到根 + 加 path matcher，复杂度爆炸。
备选 C：两层皆支持（hooks 在根和子项目各放一份）— 否决：违反 YAGNI，单人项目无需面面俱到。

决策 2：.claude-marketplace/ 暂留 patra-api/

patra plugin 13 个 skill 都是基于 Java/Gradle/六边形深度定制，未适配前端。提升到根会让 patra-portal 误以为可用。

备选: 立刻提升到 Patra/.claude-marketplace/ — 否决：未适配 FE 的 plugin 提升后会让 portal cwd 误调用，反而需要在 portal CLAUDE.md 写更多警告。等 plugin FE 适配的单独 PR 一起处理迁移更干净。

决策 3：删除所有 AGENTS.md，不维护 Codex

当前仅 Claude Code 使用，AGENTS.md 跟 CLAUDE.md 内容平行维护是 DRY 负担。删除两个 AGENTS.md（根 + patra-api/）。

备选 A：保留并同步维护 — 否决：增加每次改动的双倍工作量，无实际使用方。
备选 B：AGENTS.md 用 stub 指向 CLAUDE.md — 否决：增加一层间接，未来真要用 Codex 时直接重建 AGENTS.md 成本更低。

决策 4：通用提到根 + 子项目层只写差异（叠加模型）

三层 CLAUDE.md 利用 Claude Code "cwd 上溯叠加" 机制，把"绿地、TDD、Layout"等通用内容放根，子项目层只写"角色定位、技术栈、领域 skill"。

备选：每层独立完整（通用内容三层重复）— 否决：维护痛苦，三层每次修改要同步。叠加模型让每条规则唯一来源。
