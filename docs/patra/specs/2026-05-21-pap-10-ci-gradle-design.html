PAP-10: CI 全面 Gradle 化 + JaCoCo 聚合 + Codecov

📅 2026-05-21 🌿 worktree-pap-10-ci-gradle-jacoco 🎯 spec draft
关键决策摘要

把 .github/workflows/ 从已失效的 Maven 命令链改写为 Gradle，并借机一次性把测试分层重构到 JVM Test Suites（src/integrationTest/ / src/e2eTest/）。CI 单 workflow（ci.yml）单 job（build-and-test），靠 gradle/actions/setup-gradle@v4 自带 build cache 加速，不再手写 MODULE_MAP 变更检测。JaCoCo 用官方 jacoco-report-aggregation plugin 聚合 unit + IT 覆盖率，通过 codecov/codecov-action@v4 + CODECOV_TOKEN secret 上传。Required Status Check 命名 CI / build-and-test，留给 PAP-15 分支保护引用。

1. 背景与现状

当前 .github/workflows/ci.yml 和 pr-checks.yml 全部基于 Maven 命令链（mvn install / mvn test -Punit-only / cd patra-ingest/patra-ingest-boot ...），但 monorepo 已在 v0.2 阶段全量切到 Gradle，仓库内不存在任何 pom.xml；workflow 中引用的 patra-ingest/patra-ingest-boot 路径在 v0.2 reorg 之后也已不存在（实际位于 patra-api/patra-ingest/patra-ingest-boot）。两个 workflow 已**全链路失效**，触发即报错或空跑。

同时，linqibin.java-base.gradle.kts convention 已经 wire 了 JaCoCo / Spotless / SpotBugs 插件，但所有测试（*Test.java / *IT.java / *E2E.java）都挤在同一个 src/test/ 目录、共享 tasks.test，意味着 ./gradlew test 会一起拉起 Testcontainers（postgres:17、apache/rocketmq:5.3.1），任何缺 Docker 的环境直接 ERROR。

本次改造是工程基础设施基线（v0.3 Dev-Ready）的 P0 起点，PAP-12 / PAP-13 / PAP-15 都依赖本 Issue 完成后才能 wire 各自的门禁。

3. 测试分层（JVM Test Suites）

在 build-logic/src/main/kotlin/linqibin.java-base.gradle.kts convention plugin 中集中配置，让所有 java 模块自动获得三个 test suite。

testing {
    suites {
        val test by getting(JvmTestSuite::class) {
            useJUnitJupiter()
            // 保留 manual / external tag exclude
        }
        register<JvmTestSuite>("integrationTest") {
            useJUnitJupiter()
            dependencies { implementation(project()) }
            targets.configureEach {
                testTask.configure {
                    shouldRunAfter(test)
                    useJUnitPlatform { excludeTags("manual"); if (!project.hasProperty("runExternal")) excludeTags("external") }
                }
            }
        }
        register<JvmTestSuite>("e2eTest") {
            useJUnitJupiter()
            dependencies { implementation(project()) }
            targets.configureEach {
                testTask.configure {
                    shouldRunAfter(named("integrationTest"))
                    useJUnitPlatform { excludeTags("manual"); if (!project.hasProperty("runExternal")) excludeTags("external") }
                }
            }
        }
    }
}

// check 默认只依赖 test；integrationTest / e2eTest 显式调用

目录布局（每个 java 模块）：

  • src/test/java/ — unit tests（*Test.java），无 Docker 依赖
  • src/integrationTest/java/ — IT（*IT.java）+ 关联的 *ContainerInitializer.java 基础类
  • src/e2eTest/java/ — E2E（*E2E.java），完整 Spring Context + 多容器
  • src/test/resources/ — 共享 fixtures（默认 unit + IT 均可访问，按 Gradle 隐式 testRuntimeOnly 合并）
  • src/integrationTest/resources/ — 仅 IT 专属（如 Flyway test data），按需创建

SpotBugs 范围：convention 中现有 SpotBugsTask 配置仅作用于 spotbugsMain，自然不会扫到任何 test source set（决策 C：测试代码不做静态分析）。新增 integrationTest / e2eTest source set 不影响这一点。

4. JaCoCo 聚合

根 build.gradle.kts 引入 Gradle 官方 jacoco-report-aggregation plugin（Gradle 7.4+，已是 stable API），生成聚合 XML。

plugins {
    base
    `jacoco-report-aggregation`
}

dependencies {
    subprojects {
        if (plugins.hasPlugin("jacoco")) {
            jacocoAggregation(project)
        }
    }
}

reporting {
    reports {
        register<JacocoCoverageReport>("testCodeCoverageReport") {
            testSuiteName = "test"
        }
        register<JacocoCoverageReport>("integrationTestCodeCoverageReport") {
            testSuiteName = "integrationTest"
        }
    }
}

聚合输出路径（Gradle 默认）：

  • build/reports/jacoco/testCodeCoverageReport/testCodeCoverageReport.xml（unit 覆盖率聚合）
  • build/reports/jacoco/integrationTestCodeCoverageReport/integrationTestCodeCoverageReport.xml（IT 覆盖率聚合）

AC 路径微调：Issue 池原 AC 写的是 build/reports/jacoco/aggregate/jacoco.xml（单文件路径），但 Gradle 官方 jacoco-report-aggregation 按 suite 拆分，每个 suite 一份。Codecov 上传时 files: 字段同时指向两个 XML 即可，比手写 Copy task 改名简洁。本 spec 内显式覆盖 AC。

覆盖率门禁：现有 tasks.jacocoTestCoverageVerification（70% 阈值）不挂到 check，仅作为可手动调用的 task 保留（决策 D：v0.3 不设阈值门禁）。

e2eTest 覆盖率：暂不聚合（3 个 E2E 测试，覆盖率信号弱）。需要时按相同模式追加 e2eTestCodeCoverageReport。

5. CI Workflow 结构

删除 .github/workflows/pr-checks.yml，重写 .github/workflows/ci.yml 为单文件 + 单 job：

name: CI

on:
  push:
    branches: [main]
    paths-ignore: ['**.md', 'docs/**', '.gitignore', '.editorconfig', 'LICENSE']
  pull_request:
    branches: [main]
    paths-ignore: ['**.md', 'docs/**', '.gitignore', '.editorconfig', 'LICENSE']
  workflow_dispatch:

jobs:
  build-and-test:
    name: build-and-test
    runs-on: ubuntu-latest
    timeout-minutes: 25
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-java@v4
        with: { java-version: '25', distribution: 'temurin' }
      - uses: gradle/actions/setup-gradle@v4   # 自动启用 build cache + dependency cache
      - run: ./gradlew test --no-daemon        # 仅 unit；spotbugs/spotless 进 check 由 PAP-12/13 启用
      - run: ./gradlew integrationTest --no-daemon
      - run: ./gradlew e2eTest --no-daemon
      - run: ./gradlew testCodeCoverageReport integrationTestCodeCoverageReport --no-daemon
      - uses: codecov/codecov-action@v4
        with:
          token: ${{ secrets.CODECOV_TOKEN }}
          files: |
            build/reports/jacoco/testCodeCoverageReport/testCodeCoverageReport.xml
            build/reports/jacoco/integrationTestCodeCoverageReport/integrationTestCodeCoverageReport.xml
      - uses: actions/upload-artifact@v4
        if: always()
        with:
          name: reports
          path: |
            **/build/reports/
            **/build/test-results/
          retention-days: 7

关键设计点：

  • 单 job 串行：unit → IT → E2E → 聚合。同 job 内 Gradle 配置缓存/依赖缓存复用最大化；不拆 job 是为了 Required Status Check 单一稳定名字 + 简化 needs 依赖图
  • Required Status Check 命名：CI / build-and-test（workflow name + job name）。PAP-15 分支保护规则将引用此名
  • Docker：GitHub ubuntu-latest runner 自带 Docker daemon，Testcontainers 直接可用
  • setup-gradle@v4：官方 action 自带 build cache（GHA cache 存储）+ dependency cache + wrapper 校验，比手写 actions/cache 更省心
  • timeout-minutes: 25：保守值，覆盖 unit + 59 IT（Testcontainers 容器启动慢）+ 3 E2E + 聚合

6. Codecov 接入

用户侧前置动作（不在本 Issue 自动化范围）：

  1. 登录 codecov.io，添加 patra repo
  2. 从 Codecov 仓库设置页拿到 token
  3. GitHub repo Settings → Secrets and variables → Actions，添加 secret CODECOV_TOKEN

CI 自动化：codecov/codecov-action@v4 用 token 上传两份 XML，控制台见 patra 覆盖率数字即满足 AC 中 Codecov 验证条件。

不接 Codecov YAML 配置文件（如 codecov.yml 阈值规则）：v0.3 仅做可视化，不接门禁；后续需要时单独 Issue。

7. 测试文件迁移

影响面统计：

  • 62 个测试文件迁移：59 IT + 3 E2E
  • 约 15 个子模块涉及：linqibin-commons 下 4 个 starter + patra-api 下 registry/ingest/catalog/object-storage 多个子模块
  • 关联 *ContainerInitializer.java 基础类（数量待迁移时再统计，跟随 IT 走）

迁移规则：

  • git mv src/test/java/<pkg>/XxxIT.java → src/integrationTest/java/<pkg>/XxxIT.java
  • git mv src/test/java/<pkg>/XxxE2E.java → src/e2eTest/java/<pkg>/XxxE2E.java
  • git mv src/test/java/<pkg>/*ContainerInitializer.java → src/integrationTest/java/<pkg>/*ContainerInitializer.java
  • src/test/resources/ 保持原位（unit + IT 共享）；如果发现某 yml/sql 仅 IT 用 + 命名易冲突，可按需单独迁到 src/integrationTest/resources/

验证：每个被迁移模块各跑一次 ./gradlew :module:compileIntegrationTestJava :module:integrationTest --no-daemon 确认无编译错误、Docker 可用时测试绿。

关键决策与备选

决策 1：测试分层用 JVM Test Suites（方案 C），不用 @Tag 或文件名 filter

在 linqibin.java-base convention 中用 Gradle testing { suites { ... } } DSL 声明 3 个 suite，物理 + 依赖双重隔离。

备选 A（文件名 filter）：单 Test task + filter.includeTestsMatching("*IT")，0 行源码改动 — 否决：无依赖隔离，Testcontainers 依赖永远污染 unit test classpath，违背绿地项目"直接最优"原则
备选 B（JUnit @Tag）：62 个文件加 @Tag("integration") — 否决：仍无依赖隔离，且 tag 标错/漏标会静默漏跑
备选 D（手动 source set）：手写 sourceSets {} + configurations.integrationTestImplementation.extendsFrom — 否决：JVM Test Suites 是其声明式取代，少写 80% 样板

决策 2：删 MODULE_MAP 变更检测，全量跑 + Gradle build cache 加速

原 ci.yml 用 bash declare -A MODULE_MAP 手工映射文件路径到 Maven 模块，逻辑复杂且已完全失效（命名 reorg 前）。改成全量跑 ./gradlew check integrationTest e2eTest，靠 gradle/actions/setup-gradle@v4 提供的 build cache 加速增量。

备选 A（重写 MODULE_MAP path）：对齐 linqibin-commons / patra-starters / patra-api 三段命名 — 否决：单人项目反馈速度差距不显著，但 bash 逻辑长期维护成本高
备选 B（dorny/paths-filter Action）：第三方 Action 接管 path → filter 映射 — 否决：仍是"变更才跑"的复杂模型，新增依赖；Gradle build cache 已经足够智能

决策 3：Codecov 用 token + GitHub secret，不走 tokenless

Codecov v5 推荐 token 化，免 rate limit + 防止 PR 伪造上传。token 通过 GitHub repo secret CODECOV_TOKEN 注入 CI。

备选 A（tokenless）：public repo 允许 — 否决：受 rate limit 影响，v5 已开始 deprecate tokenless 模式
备选 B（不接 Codecov）：仅产出 XML artifact — 否决：与 Issue AC "Codecov 控制台见数字" 冲突

决策 4：废弃 pr-checks.yml，单一 ci.yml 单 job

原 pr-checks.yml 全部逻辑失效（mvn 命令 + 已不存在的 cd 路径），其 PR title/body 检查功能将由 PAP-14 的 CodeRabbit + PR 模板代替。

备选 A（保留 pr-checks.yml 但重写）：ci.yml 跑测试，pr-checks.yml 只做轻量 PR 信息检查 — 否决：多一套 workflow 维护成本，AI Review 已覆盖
备选 B（拆 ci.yml + pr.yml 用 workflow_call）：过度设计，单人项目不需要

决策 5：Required Status Check 命名 CI / build-and-test

workflow name: CI + job name: build-and-test，GitHub UI 显示 CI / build-and-test。该名字将被 PAP-15 分支保护规则引用，定下后改名需要同步更新分支保护配置，因此 v0.3 内必须稳定。

备选 A（CI / required）：job 命名更抽象，未来加子 check 不需改名 — 否决：v0.3 spec 原推荐 build-and-test，保持一致性
备选 B（直接用 workflow check 名）：不拆 job — 否决：jobs 必须有 name；显示效果同等

决策 6：JaCoCo 用官方 jacoco-report-aggregation，不手写聚合 task

Gradle 7.4+ 官方插件，与 JVM Test Suites 联动，按 testSuiteName 分别产出聚合报告。比手写 root-level jacocoAggregateReport task 简洁稳定。

备选 A（手写 JacocoReport task）：遍历子模块 executionData.from(...) 拼接 — 否决：官方插件做的就是这件事，重复造轮子

风险与缓解

风险等级缓解
JVM Test Suites 在 Gradle 9.5 仍标 incubating，未来 API 可能变 low API 自 Gradle 7.3 稳定 3 年无破坏性改动；社区已大规模生产使用；绿地项目重构成本可控
62 个文件 git mv 后 IDE 索引混乱 / 同名包重复扫描 mid 迁移分模块逐个进行 + 每个模块迁移后立即 ./gradlew :module:compileIntegrationTestJava 验证编译；最后整体 IDEA reimport 一次
共享 src/test/resources/ 中部分 yml/sql 仅 IT 用，IT 迁出后路径变化导致 ClassLoader 找不到 mid Gradle 默认 integrationTest 的 runtimeClasspath 继承 test，资源默认可访问；如出现冲突按需细分到 src/integrationTest/resources/
Codecov token 未配置时 CI 上传 step 失败但其他 step 已经成功 low codecov-action 设 fail_ci_if_error: false；spec 里明列 token 配置为用户侧前置动作；首次 PR 验证
Testcontainers 在 ubuntu-latest 启动 PostgreSQL/RocketMQ 超时（25 min 不够） low 本地实测 IT 套件单次跑 < 8 min；首次 CI 实测后按需调高 timeout-minutes
Gradle 9 configuration-cache 与 JVM Test Suites 兼容性 low JVM Test Suites 与 cc 严格模式兼容；测试时 --configuration-cache 显式验证
setup-gradle@v4 build cache 在 fork PR 上不可用（GitHub 安全策略禁写） low 单人项目无 fork 场景；如未来开放贡献，cache-read-only 行为自动生效
PAP-10 CI 只跑 test（不跑 check），spotbugs/spotless 暂不阻断；与 v0.3 决策 C"传统静态分析阻断 PR"的最终态有差距 mid 渐进激活：PAP-12（SpotBugs reportLevel + exclude baseline）和 PAP-13（spotlessApply 全代码 baseline）完成后，各自 PR 内追加 ./gradlew spotbugsMain / ./gradlew spotlessCheck step 到 ci.yml；PAP-15 把它们与 build-and-test 并列纳入分支保护 Required Status Checks 即可达到决策 C 的最终态
PAP-10 的 spec 决策与 Linear PAP-10 issue description 原 AC 有偏差（MODULE_MAP 删除、JaCoCo 路径变默认、job 结构由多 job 改单 job） low 实施完成后回写 Linear PAP-10 description：删 MODULE_MAP / JaCoCo 路径用 Gradle 默认两份 / job 结构改单 build-and-test。Linear 与 spec 互为 source-of-truth，避免漂移
