📅 2026-05-22 🌿 pap-18-flaky-audit 🎯 spec draft
关键决策摘要

PAP-10 已修过 7 处 flaky 症状，但代码库内 178 处 @Timeout 仍在（172 处类级、6 处 LetPub 方法级），属于"魔法数字 + 防 hang 拍脑袋"惯性写法。本次治理拆两个 PR：PR-1「护栏先行」在 linqibin.java-base.gradle.kts 三个 JvmTestSuite 各加 forkEvery = 50 + 三档全局 junit.jupiter.execution.timeout.default（test=5s / IT=30s / E2E=180s），并在 .github/PULL_REQUEST_TEMPLATE.md 补一条 flaky checklist；PR-2「逐文件清理」手工删除全部 172 处类级 @Timeout 与孤儿 import，保留 6 处 LetPub 网络调用方法级注解并加注释。AC2 已在 PAP-10 期间治理，仅作 grep 验证。AC4 = PR-2 合并后第 7 个自然日内 CI 连续 5 次全绿。

1. 背景与现状

PAP-10 首次启用 Gradle CI 后，连续两轮 CI run 暴露 7 处 pre-existing flaky test。本地 ./gradlew :module:test 单测不复现，只有 CI 冷启动 + 多模块并行 + JVM worker 复用场景下才暴露。PAP-10 PR 期间已按下症状（commit 7372ecaa9 + 1f6779bec），但根因没动：

  1. 本地与 CI 行为差异未对齐：本地 = 热路径 + 单模块 daemon 复用；CI = 冷启动 + 多模块并行 + JVM worker 跨 class 复用。
  2. @Timeout 滥用 / 魔法数字：作者用类级 @Timeout(5s) 当 hang 保护，实质成了"每个方法的硬预算"，5 秒拍脑袋。
  3. JVM worker 复用 + 全局状态泄漏：Metrics.globalRegistry 是 process-wide 单例，Gradle 默认 worker 跨 test class 复用 JVM，测试间状态污染。
  4. flaky test 无治理流程：无 retry policy / flaky tag / PR-level flaky guard。

本次审计实测结果（在 main 当前 HEAD 上）：

  • rg --type java "@Timeout" 命中 178 处声明，分布在约 150 个测试文件、5+ 模块。
  • 按值分布：@Timeout(2) / (value=2, SECONDS) 共 115 处；30 SECONDS 53 处；5 3 处；10 SECONDS 1 处；60 SECONDS 2 处（E2E）；2 MINUTES 6 处（LetPub 真实网络调用，方法级）。
  • 除 6 处 LetPub 方法级外，其余 172 处均为类级。
  • AC2 "迭代时改集合" 反模式：仅 OtelMeterRegistryBridgeTest 历史 8 处，PAP-10 已用 new ArrayList<>(getRegistries()).forEach(::remove) 治理；全代码库 grep \.forEach\s*\(.*::remove 无其他命中。
  • build-logic/src/main/kotlin/linqibin.java-base.gradle.kts 三个 JvmTestSuite 仅设 maxParallelForks，无 forkEvery（默认 0 = JVM worker 永久复用）。
  • PAP-14 已合并到 main，.github/PULL_REQUEST_TEMPLATE.md 存在，AC5 不再被阻塞。

3. PR-1 护栏：JVM 隔离 + 全局 timeout + PR checklist

PR-1 单一目标：用最小配置变更，让 CI 上的 flaky 多 4 道护栏。合并即生效，不依赖后续 AC1 大批量清理。

3.1 Gradle convention 改动

文件：build-logic/src/main/kotlin/linqibin.java-base.gradle.kts。三个 JvmTestSuite 的 testTask.configure 块各加 2 行：

// test suite
testTask.configure {
    useJUnitPlatform { /* 保持 */ }
    maxParallelForks = (Runtime.getRuntime().availableProcessors() / 2).coerceAtLeast(1)
    forkEvery = 50                                                          // 新增
    systemProperty("junit.jupiter.execution.timeout.default", "5s")         // 新增
    jvmArgs("--add-opens", "java.base/java.lang=ALL-UNNAMED")
    testLogging { events("passed", "skipped", "failed") }
}

// integrationTest suite — 同上，仅 timeout=30s
// e2eTest suite        — 同上，仅 timeout=180s

设计依据：

  • forkEvery = 50：JVM worker 跑 50 个 test class 后重启，限制 Metrics.globalRegistry 这类 process-wide 单例的累积污染窗口；Spring context 缓存仍保留 50× reuse，启动开销可控。
  • junit.jupiter.execution.timeout.default = JUnit 5 全局兜底（仅作用于 @Test / @TestFactory / @TestTemplate 方法），方法上仍可用 @Timeout 局部覆盖，LetPub 的 6 处方法级 2-min 不受影响。
  • 三档分层贴合实际：当前 @Timeout(2) 单测主流 → 5s 留 2.5× 余量；@Timeout(30 SECONDS) IT 主流 → 30s 保持；E2E @Timeout(60 SECONDS) → 180s 留充足容错。

3.2 PR template checklist

文件：.github/PULL_REQUEST_TEMPLATE.md，在 Test plan 段已有的空 - [ ] 之上或之下补一条：

- [ ] 本地 `./gradlew test` 通过 ✓ 且 CI 上无 flaky（无 step 因 flaky 重跑）

不修改 markdown 结构，仅在现有 checklist 区域内追加一行。

4.3 单文件处理标准动作

  1. 读取文件，定位类级 @Timeout(...) 注解行（位于 class 声明之上、无 @Test 紧邻的）。
  2. 删除该注解行。
  3. 检查 import org.junit.jupiter.api.Timeout;：若文件内已无其他 @Timeout（即非 LetPub 方法级保留场景），删除该 import。
  4. 检查 import java.util.concurrent.TimeUnit;：grep 该文件内 TimeUnit\. 还有其他用法（断言、Duration 等）才保留，否则删除。
  5. 运行该模块 ./gradlew :module:test 确认全绿。
  6. commit 该模块为一组（一个 task = 一个或多个 commit）。

每完成一个模块跑 ./gradlew spotlessApply，由 Spotless 的 removeUnusedImports 兜底捕获遗漏的孤儿 import。

5. AC2 反模式验证

PAP-10 已治理 OtelMeterRegistryBridgeTest 8 处 ConcurrentModificationException 风险（new ArrayList<>(getRegistries()).forEach(::remove) 拷贝快照后迭代）。本 spec 不新增代码改动，仅在 plan 验证步骤里加两条 grep，要求零命中：

# 反模式 1：forEach(::remove)
rg --type java '\.forEach\s*\(.*::remove\b' -g '!**/build/**'

# 反模式 2：for-each + list.remove(x)
rg --type java -B1 -A5 'for\s*\(.*:\s*\w+\s*\)' \
   -g '*Test*.java' -g '*IT.java' -g '*E2E.java' \
   | rg -B1 'list\.remove\('

AC2 的根因（Metrics.globalRegistry process-wide 单例 + JVM worker 复用导致跨 test class 污染）由 AC3 的 forkEvery=50 兜底，不在 AC2 范围内单独处理。

6. AC4 5 次绿统计窗口

  • 启动时刻：PR-2 合并到 main 当时。
  • 统计范围：合并日起 7 个自然日内，所有 GitHub Actions build-and-test job run（含 PR run 与 main push run）。
  • 达成标准：连续 5 次绿，无任何 step 因 flaky 被重跑或人工 retry。
  • 失败处理：
    1. 排查是否被 PR-1 全局 timeout 误杀 → 该方法局部 @Timeout 覆盖。
    2. 排查是否新增全局状态污染源 → 评估 forkEvery 调至 25 或 1。
    3. 每次失败重置计数器，从下一次绿重新计数。

7. 关键决策与备选

决策 A：AC1 治理策略 = 全删 + 全局 fallback

删除全部 172 处类级 @Timeout（保留 6 处 LetPub 方法级），在 java-base convention 设三档全局 junit.jupiter.execution.timeout.default 作为兜底。

备选 A：全删，无 fallback — 否决：真出现死循环时只靠 CI step timeout 兜底，反馈链条长。
备选 B：保守逐处评估保留 — 否决：172 处人工判断耗时大、决策易反复，且违背 AC1 倾向（默认删除）。

决策 B：forkEvery 取值 = 50

三个 JvmTestSuite 统一 forkEvery = 50。

备选 A：forkEvery = 1 — 否决：每个 test class fresh JVM，Spring Boot context init 开销 ×N，CI 时间可能 3-5×。
备选 B：forkEvery = 100 — 否决：对 Metrics.globalRegistry 单例保护不够彻底。

决策 C：AC1 清理方式 = 逐文件手工过，禁止机械化脚本

由 plan 拆 ~10 个独立 task，每个 task 一个模块，subagent-driven 执行；每个文件人工判断 import 是否孤儿、是否还有别的 TimeUnit 用法。

备选 A：写 Bash + sed 脚本批量删除 + spotlessApply 兜底 — 否决：用户明确要求"一个个过"；脚本难处理嵌套 @Timeout、Nested 类内的方法级注解、注释中的字面量等边界。
备选 B：用 IDE 全局 search & replace — 否决：跨模块批量回归风险大，无 task 粒度可追踪进度。

决策 D：全局 timeout 分三档（test=5s / IT=30s / E2E=180s）

按 JvmTestSuite 分别设 systemProperty("junit.jupiter.execution.timeout.default", ...)。

备选 A：全部统一 60s — 否决：单测 60s 才被杀，反馈环路过长。
备选 B：更宽松 test=10s / IT=60s / E2E=300s — 否决：失去早发现死循环的能力。

决策 E：PR 拆 2 个（PR-1 护栏 / PR-2 清理）

PR-1 = AC3 + AC5 + 全局 timeout（~5 行）；PR-2 = AC1 逐文件清理（~200-300 行 net delete）。

备选 A：单 PR 覆盖全部 AC — 否决：diff 大、护栏与清理耦合，护栏不能立即生效。
备选 B：AC1 按模块再拆多个 PR — 否决：过场费高，单人项目不需要这种粒度的 revert 能力；plan 内 task 拆分已足够。

决策 F：AC2 不重构 Metrics.globalRegistry

不动 MicrometerAutoConfiguration 与 OtelMeterRegistryBridgeTest 的当前实现；由 AC3 forkEvery=50 缓解跨 test class 污染。

备选 A：把 Metrics.globalRegistry 注入而非静态访问 — 否决：属架构改造，超出 PAP-18 范围；OTel Agent 仍会向 globalRegistry 注册，无法真正消除单例。
备选 B：每个 test 用 @BeforeEach 全量清理 — 否决：已实现但不彻底，且依赖每个测试作者自觉。

决策 G：6 处 LetPub 方法级 @Timeout(2 MINUTES) 保留

LetPubEnrichmentAdapterIT × 4 与 LetPubProxyConnectivityTest × 2 是真实网络调用，方法级 2-min 远超 IT 全局 30s，必须保留。每处加注释说明。

备选 A：调高 IT 全局 timeout 到 2 min — 否决：全局值膨胀影响其他无网络的 IT 反馈环路。
备选 B：把这两个测试移到 e2eTest suite — 否决：当前测试代码组织结构非本次 scope，迁移涉及 build.gradle、profile、CI workflow 联动改动。

8. 风险与缓解

风险等级缓解
全局 5s timeout 误杀某些慢单测（如有意外 IO 或 mock 慢） mid 受影响测试单独加方法级 @Timeout(value=10, SECONDS) 覆盖；plan 中 PR-1 验证步骤要求跑全量 ./gradlew test 暴露此类。
forkEvery=50 导致 CI 时间明显上升 low PR-1 合并后对比 PAP-10 历史 build time，若上升 >30% 调至 100；当前 main 单 job timeout 25 min，有充足余量。
删除类级 @Timeout 后某 test 真死循环 low 三档全局 fallback 兜底；CI workflow job-level timeout-minutes: 25 再兜底；本地 IDE 跑该测试也会卡，会被发现。
PR-2 大批量改动遗漏孤儿 import low ./gradlew spotlessApply 自动 removeUnusedImports；编译期未使用 import 会被 IDE / spotless 标记。
AC4 窗口内出现 flaky 复发，5 次绿无法达成 mid 逐次诊断：误杀 → 局部 @Timeout；状态污染 → forkEvery 调小；其他 → 单独排查并在 Implementation Notes 记录。
PR-2 改动 ~150 文件，review 成本高 low diff 全是注解 + import 删除，模式简单；按模块拆 commit；reviewer 抽样 + 信赖 CI 即可。
