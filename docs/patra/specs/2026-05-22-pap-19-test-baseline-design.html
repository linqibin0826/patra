PAP-19 测试体系基线设计 — 命名规范化 + java-test-fixtures + skill/rule 同步

📅 2026-05-22 🌿 worktree-pap-19-test-baseline 🎯 spec draft
关键决策摘要

本 spec 覆盖 PAP-19 主线 A：测试组织规范化（AC1+AC2+AC3+AC4），CI 提速主线 B（AC5/AC6/AC7/AC8a）移出本 spec、单独开新 Issue 跟踪。主线 A 三件事：① 给 16 个测试辅助类统一后缀命名（*ITBootstrap / *ITWebMvcConfig / *ITSpringConfig / *ITContainerInitializer / *Fixture）；② 启用 Gradle 标准 java-test-fixtures plugin 替换 build-logic 里两处 sourceSet hack 与 1 处跨模块 testArtifacts hack；③ 同步 6 个 skill / rule / agent 文件让 path glob 与命名规范跟上。AC4 用现有 CI 跑 3 次绿确认无回归。

1. 背景与现状

PAP-10（CI 全面 Gradle 化 + JVM Test Suites 重构）实施期暴露了与测试相关的项目层面缺口。其中 flaky bug 已由 PAP-18 独立治理完毕，CI 时长优化拆分到本 Issue 主线 B 单独跟踪。本 spec 聚焦"测试代码组织规范"维度。

1.1 现状事实（subagent 实地扫描）

build-logic 当前状态（build-logic/src/main/kotlin/linqibin.java-base.gradle.kts）：

  • 已用 JVM Test Suites DSL 注册 3 个 source set：test / integrationTest / e2eTest（PAP-10 引入）
  • 三个 suite 已统一设 maxParallelForks = (CPU / 2).coerceAtLeast(1) + forkEvery = 50（Issue 描述里"默认 maxParallelForks=1"已过期）
  • 未 apply java-test-fixtures plugin
  • L164-172 存在显式 sourceSets { named("integrationTest") { compileClasspath += sourceSets["test"].output ... } } hack —— 这是 PAP-10 为绕过 Spring Boot bootJar 禁用 jar task 导致 implementation(project()) artifact 为空所做的 workaround

测试辅助类命名分布：16 个辅助类待命名规范化，间接波及 55-60 个测试文件的 import / annotation 引用。按改动难度分：

  • 难度 A（纯改类名+文件名，仅同包内引用）：2 个 — TaskAggregateTestDataBuilder、TestHelper
  • 难度 B（改名+需验证 Spring 扫描行为，无 import 改动）：4 个 — 3 份 TestApplication（catalog/ingest/registry-infra）+ TestBatchApplication
  • 难度 C（改名+必须同步更新其他文件的 import 或 annotation 引用）：10 个 — 3 个 WebMvcConfig 类 + 5 个 ContainerInitializer + FileMetadataTestDataBuilder（跨模块）+ PubMedComputedAuthorTestDataGenerator

跨 source set 共享 fixture 的 hack 实证：

  • 同模块 sourceSet output 链式共享 hack（build-logic 全局）：PubMedComputedAuthorTestDataGenerator（catalog test → catalog integrationTest）、TestBatchApplication（batch test → batch integrationTest）
  • 跨模块 testArtifacts 自定义 configuration hack（仅一处）：patra-object-storage-domain 用自定义 testArtifacts configuration 把 test output 打 jar，patra-object-storage-app 通过 testImplementation(project(path, configuration = "testArtifacts")) 消费 FileMetadataTestDataBuilder

测试相关 skill / rule / agent 现状：

  • .claude-marketplace/patra/skills/test-driven-development/SKILL.md 测试规范表只列了 *Test，未覆盖 *IT / *E2E 命名规则
  • patra-api/.claude/agents/test-checker.md 测试位置列写死 src/test/，不区分 source set
  • patra-api/.claude/rules/testing/{integration,e2e}-test.md frontmatter paths glob 硬编码 src/test/**/*IT.java，AC2 启用 src/integrationTest source set 后 rule 立刻 silently broken
  • .claude-marketplace/patra/skills/test-driven-development/testing-anti-patterns.md 引用 src/test/java/.../testsupport/ 路径，testFixtures 落地后该建议过时
  • patra-api/.claude/skills/patra-events/resources/outbox-guide.md 引用具体 OutboxPatternE2E.java 路径，需跟随 source set 调整

1.2 痛点

  • 批量 find / grep 不能一次扫全：PAP-10 实施期用 find -name "*IT.java" -o -name "*E2E.java" 做测试代码批量迁移时漏迁了一批辅助类（Bootstrap / WebMvcConfig / Fixture），回头逐个 grep 引用关系才补齐
  • 命名缺乏区分性：TestApplication / WebMvcTestConfiguration / TestConfiguration 这类命名靠"Test"字样区分却不带后缀，与 Spring 自身注解或类名易混淆
  • 跨 source set 共享靠两个 hack：build-logic 里的 sourceSet 链式 output 共享 + patra-object-storage-domain 的 testArtifacts configuration。Gradle 自 7.x 提供 java-test-fixtures 标准 plugin 专门解决此场景，没有理由继续维持 hack
  • skill / rule path glob 写死 src/test：PAP-10 引入 src/integrationTest source set 时 rule frontmatter 漏改，IDE / 自动 attach 工具读不到 IT 文件

3. 命名约定

所有"测试辅助类"（非测试本身，而是被测试 import 或 Spring 扫描发现的支持类）统一带 IT 前缀 + 角色后缀。fixture/数据构造类不带 Test 前缀（src/testFixtures/ source set 本身已提供语义）。

3.1 角色后缀对照表

角色 目标后缀 规则说明
IT 用 Spring Boot placeholder Application（提供 @SpringBootConfiguration 给 @DataJpaTest / @SpringBootTest 找根） *ITBootstrap 类名 = {Module}ITBootstrap
@WebMvcTest 切片测试的 mock 配置根（@SpringBootConfiguration + @EnableAutoConfiguration） *ITWebMvcConfig 类名 = {Module}ITWebMvcConfig
@TestConfiguration 内嵌 bean 装配（独立文件 + inner static class 都含） *ITSpringConfig 避免与 Spring @TestConfiguration 注解同名混淆
Testcontainers 初始化器（extend 父类指定 db / bucket 名） *ITContainerInitializer 类名 = {Module}IT{Container}ContainerInitializer，区分 IT/E2E source set 用途
测试数据构造 / fixture / builder / generator *Fixture 放 src/testFixtures/java/ source set；不带 Test 前缀（source set 路径已提供语义）

4. testFixtures 落地

4.1 build-logic 改动

build-logic/src/main/kotlin/linqibin.java-base.gradle.kts 改动要点：

  1. 顶部 plugins 块 apply java-test-fixtures：与 java-library 并列声明
  2. 三个 JVM Test Suite 的 dependencies 块添加对 testFixtures(project()) 的 implementation 依赖，让 test / integrationTest / e2eTest 均能 import testFixtures 中的类
  3. 删除 L164-172 的 sourceSets hack： 
    // 整段删除（不再需要）
sourceSets {
    named("integrationTest") {
        compileClasspath += sourceSets["main"].output + sourceSets["test"].output
        runtimeClasspath += sourceSets["main"].output + sourceSets["test"].output
    }
    named("e2eTest") {
        compileClasspath += sourceSets["main"].output + sourceSets["integrationTest"].output + sourceSets["test"].output
        runtimeClasspath += sourceSets["main"].output + sourceSets["integrationTest"].output + sourceSets["test"].output
    }
}
  4. 验证 main.output 仍可被 testFixtures 看到：testFixtures source set 默认依赖 main，无需额外配置；但需确认 Spring Boot bootJar 禁用 jar task 的副作用对 testFixtures 不复现（必要时给 testFixtures 单独保留 main output 引用）

4.2 fixture 物理迁移

2 个跨 source set 共享 fixture 从 src/test/java/ 迁至 src/testFixtures/java/（保持 package 不变；遵循 YAGNI：仅"被 ≥2 个 source set / 模块 import"的 fixture 才迁，模块内私有 fixture 留在 src/test/）：

  • patra-object-storage-domain：FileMetadataTestDataBuilder → FileMetadataFixture（迁 testFixtures，跨模块共享，被 patra-object-storage-app/test/RecordUploadHandlerTest import）
  • patra-catalog-infra：PubMedComputedAuthorTestDataGenerator → PubMedAuthorFixture（迁 testFixtures，跨同模块 sourceSet 共享，被 integrationTest/AuthorItemWriterIT + test/AuthorItemReaderTest + test/PubMedComputedAuthorParserTest 引用）

另外 2 个 fixture 原地保留（仅本模块 test 内使用）：

  • patra-ingest-domain/src/test/：TaskAggregateFixture
  • patra-ingest-app/src/test/：TaskReadyCommandFixture

4.3 跨模块 testArtifacts hack 替换

patra-object-storage-domain/build.gradle.kts 删除自定义 testArtifacts configuration 与对应 jar task。patra-object-storage-app/build.gradle.kts 把 testImplementation(project(path, configuration = "testArtifacts")) 改为标准的 testImplementation(testFixtures(project(":patra-object-storage:patra-object-storage-domain")))。

4.4 IT Bootstrap 是否迁 testFixtures 的判断

不迁。3 份 *ITBootstrap（catalog/ingest/registry-infra）和 BatchITBootstrap 仍留在各自 src/integrationTest/java/ 内：

  • Bootstrap 类是 IT 专属，没有"跨 source set 共享"需求
  • 放 testFixtures 反而扩大暴露面（unit test classpath 也能扫到 @SpringBootApplication，可能干扰 @DataJpaTest 等切片测试的 Spring Boot Configuration 检测）

4.5 ContainerInitializer 是否迁 testFixtures

不迁。5 个 *ITContainerInitializer 是 IT/E2E 各自独立的实例化器（每个域有自己的 dbName / bucket name），不跨 source set 共享，留在各自 source set 内即可。父类 PostgreSQLContainerInitializer / MinIOContainerInitializer / RocketMQContainerInitializer 已在 linqibin-spring-boot-starter-test/src/main（生产 source set），通过 starter 依赖被各模块的 IT 引入，已经是合理位置。

7. 关键决策与备选

决策 1：命名风格 = *ITBootstrap / *ITWebMvcConfig / *Fixture

所有 IT 专属辅助类带 IT 前缀 + 角色后缀；Fixture 类不带 Test 前缀（source set 路径已提供语义）。

备选 A：保留 TestApplication + Fixture 加 Test 前缀（即 *TestFixture） — 否决：TestApplication 三份同名靠 package 区分，grep / find 不友好；TestFixture 与 testFixtures source set 名重复啰嗦
备选 B：不加 IT 前缀，靠 source set 路径区分（即 *Application / *WebMvcConfig） — 否决：跳转无法通过类名一眼看出 source set 归属，必须依赖 IDE

决策 2：用 Gradle 标准 java-test-fixtures 替代两类 hack

build-logic 里的 sourceSet 链式 output 共享 hack（L164-172）+ patra-object-storage-domain 的自定义 testArtifacts configuration，全部用 java-test-fixtures plugin 提供的 src/testFixtures/ source set + testFixtures(project()) 标准引用替代。

备选 A：保留 sourceSet output 共享 hack，仅清理跨模块 testArtifacts — 否决：留一个 hack 等于规范没做彻底，未来还会有人继续在 build-logic 里加新 hack
备选 B：不引入 testFixtures，把所有跨 sourceSet 共享类 inline 复制到各 source set — 否决：违背 DRY，3 处 fixture 文件 × N 份副本，维护成本极高

决策 3：主线 B（CI 提速）拆出本 spec

PAP-19 原描述含 AC5/AC6/AC7/AC8a 4 个 CI 提速相关 AC，本 spec 不覆盖，单独开新 Issue 跟踪。AC8b（maxParallelForks）已落地，无需重复处理。

备选 A：一个 spec 覆盖两条主线 — 否决：spec 篇幅过大，且主线 B 验收依赖"3 次 PR 实测对比"，与主线 A"测试组织规范"在同一 PR 里验证会互相干扰
备选 B：只做主线 B（CI 提速优先） — 否决：CI 提速依赖测试组织稳定（如 testFixtures 引入可能改变缓存命中率），先做 B 反而风险大

决策 4：rules/testing path glob 一并同步（含进 AC3）

.claude/rules/testing/{integration,e2e}-test.md frontmatter 的 paths 写死 src/test/，AC2 启用 src/integrationTest source set 后 rule 会 silently broken（IDE / 自动 attach 工具读不到 IT 文件）。含进 AC3 一起改。

备选 A：rule 文件不属于 skill / agent，不在原 Issue AC3 字面范围内 — 否决：path glob silently broken 直接影响开发者体验，不改等于留坑

8. 风险与缓解

风险等级缓解
Bootstrap 类重命名后 Spring 扫描发现机制断裂（4 个难度 B 类，无 import 但 Spring 按包扫描发现 @SpringBootApplication） mid 重命名后立即跑 ./gradlew integrationTest，关注 36 个 IT 文件（catalog 26 + ingest 6 + registry 4）+ 2 个 batch IT 是否能找到根 @SpringBootConfiguration；任何 Unable to find a @SpringBootConfiguration 异常立即修正
WebMvcConfig / ContainerInitializer 重命名漏改引用（10 个难度 C 类，需同步 15-20 处 annotation 引用 + 14 个 import） mid 用 IDE refactor rename（IntelliJ IDEA 的 Rename Class）批量改；rename 后 ./gradlew compileIntegrationTestJava compileE2eTestJava 验证无编译失败；CI 跑 3 次绿确认无运行时反射 / 字符串引用残留
