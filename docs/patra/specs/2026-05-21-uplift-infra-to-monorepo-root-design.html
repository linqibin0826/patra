把 patra-api/infra/ 上移到 monorepo 根 patra-infra/

📅 2026-05-21 🌿 worktree-issue-44-uplift-infra 🎯 spec draft
关键决策摘要

用 git mv patra-api/infra patra-infra 保留历史地把基建目录上移到 monorepo 根。让 Claude 等 AI 工具更好理解项目结构，同时给 patra-api 瘦身、聚焦业务领域。除路径变更外，不改 compose 服务内容、不改业务模块代码、不为 patra-portal 预备本地服务。.env* 在 patra-infra/docker/ 下沿用隐式入库语义（根 .gitignore 无 .env 规则即默认入库），不显式声明。本次只调整一处 Gradle 路径（otel.agent.path），其余全部是文档 / 注释字符串同步。

1. 背景与现状

评估结论见 #43，决策为方案 B：把 patra-api/infra/ 上移到 monorepo 根 Patra/patra-infra/。命名与 patra-api/ / patra-portal/ 风格统一，且根 CLAUDE.md:62 表格已预先采用 patra-infra 名称。

现状盘点（grep + ls 确认）：

  • patra-api/infra/ 含 docker/（7 个 compose yaml + alertmanager/grafana/loki/mysql-ops/otel-agent/otel-collector/postgres/prometheus/tempo 等子目录 + README.md + opentelemetry-javaagent.jar）和 scripts/（3 个 .sh + 2 个 .plist + tailscale-route-guard.sh）。
  • compose 文件全部使用同目录相对路径（如 ./postgres/init-scripts:...），grep 确认无任何 ../infra 反向引用 ⇒ yaml 文件迁移后不需要修改。
  • Gradle 引用唯一一处：patra-api/gradle.properties:51 的 otel.agent.path=infra/docker/opentelemetry-javaagent.jar，被 patra-api/build-logic/src/main/kotlin/linqibin.hexagonal-boot.gradle.kts:59 经 rootProject.projectDir.resolve(otelAgentPath) 消费（rootProject = patra-api/）。
  • .env 入库现状：patra-api/infra/docker/.env、.env.dev、.env.example 已入库（git ls-files 确认），靠 patra-api/.gitignore 的 /.env 顶层匹配语义豁免子目录。
  • patra-api/CLAUDE.md grep 确认无 infra 文本引用，不需要修改。
  • patra-api/plugins/ 与 patra-api/.claude/ 下的 infra/ 引用全是六边形架构模块名（*-infra/）或包名（infra/persistence/...），与本次迁移无关。

3. 目录迁移

核心动作：

git mv patra-api/infra patra-infra

这是单条 git mv，git 会记录为重命名而非删除+新增，git log --follow 可追溯历史。

迁移后立即做最小验证（不需要起容器）：

cd patra-infra/docker && docker compose -f docker-compose.dev.yaml config > /dev/null

预期：解析通过、无 warning（同目录相对路径不变）。

4. Gradle 路径修正

7. 关键决策与备选

决策 1：用 git mv 而非 mv + git add/rm

用 git mv patra-api/infra patra-infra 让 git 显式记录为 rename，保留 git log --follow 历史追溯能力。

备选 A：普通 mv + git add patra-infra && git rm -r patra-api/infra — 否决：依赖 git rename detection 的启发式匹配，对于多文件目录有误判风险。
备选 B：git filter-repo 重写历史 — 否决：YAGNI，单次目录上移无需重写历史；且会改写所有 commit hash，对单人项目无收益反而徒增麻烦。

决策 2：.env* 入库采用隐式语义

不在根 .gitignore 加 .env 规则、不新建 patra-infra/.gitignore、不写 !patra-infra/docker/.env* 反忽略。无规则即默认入库，与现状行为等价。

备选 A：显式在 patra-infra/.gitignore 写 !.env* — 否决：需要先在根加 .env* 全局忽略才有反忽略目标，增量两条规则，违反 YAGNI。
备选 B：全局忽略 .env、.env.* + patra-infra/docker/.gitignore 局部反忽略 — 否决：能保护 monorepo 任意位置不被意外 commit 本地 env，但单人绿地项目暂无此类风险，且增量规则跨两个文件维护成本不划算。如未来出现"在某子项目误 commit .env"事故，再升级到该方案。

决策 3：根 CLAUDE.md:62 "技术栈" 列填 Docker Compose / Bash / launchd

与 patra-api / patra-portal 行的技术栈风格一致，列出本目录实际使用的工具链。

备选 A：留空 — 否决：表格三行有两行填了，留一行空看着突兀。
备选 B：更详细如 "Docker Compose / Bash / launchd plist / Prometheus / Grafana" — 否决：罗列内部服务而非"技术栈"，与表头语义不符。

决策 4：不动 patra-api/CLAUDE.md

grep 已确认该文件无 infra 字面引用，不存在残留指向。

备选 A：添加 "基建配置已迁移到 ../patra-infra/" 的指引行 — 否决：根 CLAUDE.md:62 已经在工作区目录表里列了 patra-infra，AI 工具读根 CLAUDE.md 就能找到，不需要在子项目 CLAUDE.md 重复。

风险等级缓解
漏改某处文档 / 注释里的 infra/docker、infra/scripts 字符串引用 mid 迁移完成后 grep 兜底：grep -rn "infra/docker\|infra/scripts" patra-api/ patra-infra/ docs/ CLAUDE.md *.md，预期仅在 patra-codex/.claude 的六边形架构文档里命中（六边形模块名，无关）
./gradlew check 找不到 OTel agent jar，报错或测试场景跑不起来 mid 迁移后立即跑 cd patra-api && ./gradlew check；如失败，检查 build-logic 是否还有其它 hardcode 的 infra/ 引用
compose 启动时 volume 找不到（理论可能性） low 已 grep 确认所有 compose volume 路径都是 ./xxx 同目录形式，无 ../ 反向引用；docker compose config 解析通过即视为安全
PR squash-merge 后本地 main 与 origin/main divergent low 本次所有 commit 都在 worktree 分支（按 memory 的 brainstorming-branch-first feedback）。merge 后 git fetch && git checkout main && git reset --hard origin/main 即可
git rename detection 失败、历史不连续 low 用 git mv 而非 mv，git 强制记录为 rename。验证：git log --follow patra-infra/docker/README.md 应能看到 patra-api/infra/docker/README.md 时代的历史
