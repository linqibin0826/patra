PAP-12 + PAP-13: SpotBugs / Spotless Convention Plugins 抽出 + CI 接入

📅 2026-05-22 🌿 worktree-pap-12-spotbugs 🎯 spec draft
关键决策摘要

把 SpotBugs 与 Spotless 两套配置从 linqibin.java-base.gradle.kts 抽出为独立的 linqibin.spotbugs.gradle.kts 与 linqibin.spotless.gradle.kts convention plugins，仍由 java-base 通过 id("...") 引入；SpotBugs 参数对齐 PAP-12 AC（effort=MAX / reportLevel=LOW / ignoreFailures=false / 显式禁用 spotbugsTest、spotbugsIntegrationTest、spotbugsE2eTest / toolVersion=4.9.3）；Spotless 保留 googleJavaFormat 默认 2 空格缩进与 .editorconfig 对齐，PAP-13 AC 的"4 空格"为口误待校正。CI 把 ./gradlew test 替换为 ./gradlew check 一次性纳入 spotbugsMain + spotlessCheck，并额外上传名为 spotbugs-reports 的专门 artifact。PAP-12 与 PAP-13 合并为同一 PR。

1. 背景与现状

v0.3 Dev-Ready 工程基建版的目标是让 Patra monorepo 进入"业务功能开发开箱即敲"状态，关键决策 B 把静态分析工具栈最小化为 SpotBugs（抓逻辑 bug）+ Spotless（抓格式），关键决策 C 进一步规定 spotbugsMain 与 spotlessCheck 失败必须阻断 PR（spotbugsTest 不阻断、测试代码不做静态分析）。

PAP-10（CI 全面 Gradle 化）已在 main 落地：.github/workflows/ci.yml 单 job build-and-test 跑通 test / integrationTest / e2eTest + JaCoCo 聚合 + Codecov 上传，但 step 内显式注释 "暂不调 ./gradlew check，因 spotbugsMain / spotlessCheck 需 PAP-12/13 完成 baseline 后才能阻断 CI"。

SpotBugs 与 Spotless 在 linqibin.java-base.gradle.kts（222 行总入口）中已部分接入：插件 apply 与基础参数都在，spotbugs-exclude.xml（282 行）也已存在并填好 Lombok / MapStruct / DO/DTO/VO / Spring / 错误处理 / JPA 等场景的预排除；build-logic/build.gradle.kts 已显式引入 SpotBugs + Spotless plugin 依赖，libs.versions.toml 锁定 spotbugs-plugin = 6.4.8、spotbugs = 4.9.3、spotless = 7.0.4、google-java-format = 1.29.0、ktlint = 1.5.0。

但配置还有 7 处差异不满足 PAP-12 AC：文件位置不独立 / reportLevel 仍是 MEDIUM / 未显式声明 ignoreFailures 与 toolVersion / spotbugsTest 未禁用 / 报告配置用 configureEach 影响所有 SpotBugs task / java-base 直接 apply 上游插件而非引入本地 convention plugin。Spotless 也有类似情况：targetExclude 未排除生成代码、未显式 importOrder()、kotlinGradle target 未覆盖 build-logic/。

PAP-12 与 PAP-13 改造路径高度重叠（"build-logic 抽 convention plugin → CI 接入 → 基线化"三件套），且 release-spec 第 274 行明示 #3 #4 可并行，因此本次合并为同一 spec / 同一 plan / 同一 PR 落地。

3. SpotBugs Convention Plugin

3.1 文件落点

新增 build-logic/src/main/kotlin/linqibin.spotbugs.gradle.kts，由 linqibin.java-base 通过 id("linqibin.spotbugs") 引入（决策 1）。所有 Java 模块仍通过现有的 java-base 链路自动继承 SpotBugs 配置，无需逐模块 apply。

3.3 任务范围（仅 main，禁用 test 系列）

决策 C 与 PAP-12 AC 双重要求"不分析 test 代码"。当前 java-base 用 tasks.withType<SpotBugsTask>().configureEach { ... } 影响所有 SpotBugs task（含 spotbugsTest / spotbugsIntegrationTest / spotbugsE2eTest）。本 plugin 改为：

  1. 显式 tasks.named("spotbugsTest") { enabled = false }
  2. 显式 tasks.named("spotbugsIntegrationTest") { enabled = false }（由 JVM Test Suites 自动生成）
  3. 显式 tasks.named("spotbugsE2eTest") { enabled = false }（同上）
  4. 报告配置只针对 spotbugsMain：tasks.named<SpotBugsTask>("spotbugsMain") { reports { create("html"){required=true}; create("xml"){required=true} } }

这样 ./gradlew check 触发的 SpotBugs 任务只剩 spotbugsMain，报告路径在每个模块的 build/reports/spotbugs/main.html / main.xml。

4.1 文件落点

4.2 Java 格式化规则

  • googleJavaFormat(libs.findVersion("google-java-format").get().requiredVersion) — 锁定 1.29.0，默认 2 空格缩进（与 .editorconfig 一致）
  • importOrder() — 默认 import 顺序（PAP-13 AC "import 顺序" 项）
  • removeUnusedImports() — 移除未使用 import
  • trimTrailingWhitespace() — 去尾空白
  • endWithNewline() — EOF 换行
  • target("src/**/*.java") — 仅源码
  • targetExclude("**/build/**", "**/generated/**") — 排除生成代码（MapStruct *MapperImpl.java / Lombok 拓展产物）

4.3 Gradle 脚本格式化规则

  • kotlinGradle：target("*.gradle.kts", "build-logic/**/*.gradle.kts")
  • ktlint(libs.findVersion("ktlint").get().requiredVersion) — 锁定 1.5.0
  • trimTrailingWhitespace() / endWithNewline()

4.4 缩进对齐说明（重要）

PAP-13 release-spec AC 字面写"缩进 4 空格"，但项目 .editorconfig（已存在）第 16 行规定 indent_size = 2，linqibin.java-base 当前 googleJavaFormat（默认 2 空格）也已与之一致。三者要求一致才能避免编辑器与 Spotless 之间的"对手戏"，因此本 spec 决议 统一为 2 空格，PAP-13 AC 视为口误，待 release-spec 同步修正（见 §8）。

6. spotbugs-exclude.xml 处置策略

6.1 AC 要求

"沿用根目录 spotbugs-exclude.xml"——不重写 282 行的现有内容，仅根据本次 baseline 跑出的报告按需增删条目。

6.2 Baseline 决策树

实施期执行 ./gradlew spotbugsMain --no-daemon 跑出第一份完整报告后：

  1. 统计告警总数 N
  2. N ≤ 30：逐条决策路径（默认）
    • 真 bug → 修源码，独立 commit 引用本 spec
    • 误报 → spotbugs-exclude.xml 加 <Match> 条目，必带 XML 注释说明"为什么是误报"（不接受裸 exclude）
  3. N > 30：暂停上报数字，由用户选 fallback
    • 选项 A：继续硬清零（耗时大但最干净）
    • 选项 B：降级 baseline mode（把当前所有告警 exclude，新代码零起步；存量告警记 TODO 留待后续）

6.3 不接受的做法

  • 批量 exclude 没有注释（违反"每条决策"原则）
  • 用 ignoreFailures = true 绕过失败（违反 AC + 决策 C）
  • 调高 reportLevel 隐藏低置信度告警（违反 AC）

7. CI Workflow 改造（ci.yml）

7.1 Step 替换

当前 .github/workflows/ci.yml 第 45-47 行：

9. 关键决策与备选

决策 1：抽出独立 convention plugin，java-base 内部引入

新增 linqibin.spotbugs.gradle.kts + linqibin.spotless.gradle.kts，linqibin.java-base 用 id("linqibin.spotbugs") / id("linqibin.spotless") 引入；所有 Java 模块沿现有 java-base 链路自动继承，无需逐模块 apply。

备选 A：每个 Java 模块的 build.gradle.kts 各自 apply — 否决：30+ 模块 30 多行重复代码，未来调参易遗漏。
备选 B：不抽出，仅在 java-base 内修参数 — 否决：违反 PAP-12 AC 字面"新增 convention plugin"，且 java-base 文件已 222 行职责过载。

决策 2：PAP-12 + PAP-13 合并为同一 PR

同一份 spec / 同一份 plan / 同一次 CI 改造 / 同一份基线提交。PAP-13 Linear 关闭并指向本 PR。

备选 A：拆分两个 PR — 否决：两者改造路径 80% 重叠（build-logic 抽插件 + CI 改 step + 基线化），分两次做有重复劳动；release-spec 第 274 行明示 #3 #4 可并行；单人 + 绿地项目 PR 体量不是问题。

决策 3：Spotless 缩进保留 2 空格

保留现有 googleJavaFormat 默认 2 空格，与 .editorconfig 第 16 行 indent_size = 2 一致。PAP-13 release-spec AC 的"4 空格"判定为口误，待 §8 同步修正。

备选 A：改为 4 空格（googleJavaFormat.aosp() 或 palantir）— 否决：需同步改 .editorconfig（与编辑器对手戏风险）；spotlessApply 一次性重写所有源码格式（PR 噪音最大）；AOSP 风格业界资源较少。

决策 4：CI 上传专门的 spotbugs-reports artifact，与通用 reports artifact 并存

保留 PAP-10 已配置的通用 reports artifact（含全部 build/reports/），额外新增 spotbugs-reports artifact（仅 **/build/reports/spotbugs/**）。

备选 A：仅保留通用 reports artifact + 加注释 — 否决：reviewer 下载大 zip 后需手动找路径，体验差。
备选 B：拆分通用 reports artifact 为多个细化 artifact — 否决：与 PAP-10 刚调好的结构冲突，动作过大。

决策 5：CI 用 ./gradlew check 一次性接入 spotbugsMain + spotlessCheck

把 ./gradlew test step 替换为 ./gradlew check，让 Gradle 内置 lifecycle 同时驱动 test + spotbugsMain + spotlessCheck。

备选 A：在 test step 后单独加 spotbugsMain step（不动 test）— 否决：PAP-13 合并 scope 后 spotlessCheck 也要进 CI，要么再加 step（3 个 step）要么后续再改一次 ci.yml，不一次到位。
备选 B：合并为 ./gradlew test spotbugsMain spotlessCheck 单 step — 否决：失败语义混在一起，CI 日志定位错误略麻烦；与 check 语义等价但更绕。

决策 6：spotbugs-exclude.xml 沿用现有，按 baseline 增删；告警量 > 30 时上报降级

不重写现有 282 行 exclude 文件，跑 spotbugsMain 后逐条决策；告警量超 30 暂停上报让用户选 fallback。

备选 A：跑前先 audit 现有 exclude 是否仍合理（清理 Maven 时代残留）— 否决：违反 AC "沿用"字面；可作为后续独立任务（如 v0.4 quality 维护版）。
备选 B：直接 baseline mode（不区分数量，把当前所有告警 exclude）— 否决：违反 AC "每条决策"原则；存量告警长期累积无人 review。

10. 风险与缓解

风险等级缓解
SpotBugs reportLevel=LOW + effort=MAX 跑出大量告警（> 30） high §6.2 决策树：> 30 暂停上报，由用户选硬清零 / baseline mode 降级
spotlessApply 改动量过大，PR 噪音淹没真实改动 mid 实施期独立 commit："chore(spotless): apply baseline formatting"；PR 描述明示"spotless baseline 在 commit X，业务变更在 commit Y"；CodeRabbit 评论可按文件 collapse
SpotBugs toolVersion=4.9.3 与 plugin 6.4.8 不兼容 low SpotBugs Gradle plugin 6.x 官方支持 4.7+ 核心；当前 libs.versions.toml 已并存这两个版本，CI 中 sanity-run 即可验证；如不兼容回退 toolVersion 至插件捆绑默认
JVM Test Suites 生成的 spotbugsIntegrationTest / spotbugsE2eTest task 名变化导致 tasks.named() 找不到 mid 本地先 ./gradlew tasks --all | grep spotbugs 验证 task 名；如名称不一致改用 tasks.matching { it.name.startsWith("spotbugs") && it.name != "spotbugsMain" }.configureEach { enabled = false }
spotbugs-exclude.xml 现有规则与 LOW confidence 新告警相互抵消，掩盖真问题 mid baseline 跑两次：首次保留现有 exclude 跑出 N；如 N 异常少（< 5）二次清空 exclude 跑 N'，对比 N' - N 看现有 exclude 是否过宽
./gradlew check 在 CI 比 ./gradlew test 慢（加跑 spotbugs + spotless） low spotbugs effort=MAX 在 patra 模块体量下预计单模块 10–30s，总体 ~2 min；可接受。如显著超时，CI 用 --parallel
合并 scope 后 PR 体量增加导致 review 时间延长 low commit 序列严格分层：(1) build-logic 抽 plugin (2) java-base 改造 (3) spotlessApply 基线 (4) spotbugsMain baseline (5) CI 改造 (6) Linear 同步；reviewer 可按 commit 单独 review
Spotless targetExclude("**/generated/**") 路径不匹配 MapStruct 实际输出 low MapStruct 默认输出在 build/generated/sources/annotationProcessor/，**/build/** 已覆盖；generated/ 是冗余保险，无副作用
