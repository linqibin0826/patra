PAP-24 CI 提速 — 容器预拉取 + Job 拆分 + Configuration Cache + Unit Sharding 设计

📅 2026-05-22 🌿 worktree-pap-24-ci-speedup 🎯 spec draft
关键决策摘要

把 PAP-24 拆为 3 个 PR 收敛实施：PR1 基础设施层（删 --no-daemon + cc 加密 key + docker pull 并发隐藏 + 镜像版本一致性校验）；PR2 Job 拓扑重构（拆 unit / IT / e2e / coverage 四 job + required-check 改为 needs 依赖 + 引入 jacoco-report-aggregation plugin）；PR3 Unit Sharding（unit-tests 改 4-shard matrix）。不做调研判定收益不确定的动作：fork 数翻倍（AC5b 推迟至后续 Issue，先 build scan 取 baseline）。Required Status Check（required-check）顺便修补为真正依赖测试 job，让 PAP-15 分支保护从「事实摆设」变为强制生效。目标：CI 总时长 ~8 min → ~3-4 min。

1. 背景与现状

PAP-19 brainstorming 阶段拆分的「CI 提速主线 B」在 PR #77 合并主线 A 时漏拆 Issue、被一并误关。本 spec 承载 PAP-24 补齐这部分，并新增针对 unit test 真测试时长的专项优化（用户反馈 unit 是当前最大瓶颈）。

当前 CI 单 job build-and-test 串行跑 check → integrationTest → e2eTest → JaCoCo aggregate → Codecov upload，PAP-10 baseline 实测稳定 ~7-8 min，可分解为「冷启动 3:05」+「真测试 4:30」两大段。

探索阶段对 PAP-24 描述的 3 处事实修正：

PAP-24 假设实际含义
Required Status Check 是 CI / build-and-test 实际是 required-check（1 min dummy job, 无 needs:，独立 echo），build-and-test 不是必需检查 job 拓扑可自由重组；分支保护实际处于「摆设」状态，本 spec 顺便修补
需要在 gradle.properties 启用 cc org.gradle.configuration-cache=true 已开（含 problems=warn），parallel / caching / daemon 也已开 AC4 唯一动作变为给 setup-gradle@v4 加 cache-encryption-key 让 cc cache 跨 run 持久化
RocketMQ 用 GenericContainer / Testcontainers 模块 实际用 docker-compose（docker-compose-rocketmq.yml），双容器 NameServer + Broker docker pull 预热仍然有效；镜像版本散落于 Java 源码 + 两份 compose 文件共 5 处

调研对 PAP-24 计划的 5 处反驳：

  • gradle/actions/setup-gradle@v4 自带 cc cache 支持，不需要叠加 actions/cache@v4；只要传 cache-encryption-key 即生效
  • JaCoCo on-the-fly instrumentation 真实开销 <10%（官方文档），「20-30% 节省」混淆了 jacocoTestReport 报告生成时间
  • 业界 unit test sharding 甜点为 3-4 shard，单 shard 冷启动 ~30-60s 叠加；原计划 7 shard 过细
  • Gradle 9 + setup-gradle@v4 下 --no-daemon 是反模式（daemon 复用 JVM warmup 反而更快）
  • maxParallelForks = availableProcessors() 收益不确定（JVM startup 200-500ms/进程），需 build scan baseline 才能决策

2. 范围边界

In Scope
  • PR1：删 --no-daemon、setup-gradle 加 cache-encryption-key、docker pull 与编译并发、镜像版本一致性校验 step
  • PR2：拆 4 个 job（unit / IT / e2e / coverage-and-upload）、required-check 改 needs 依赖 + alls-green、引入 jacoco-report-aggregation convention plugin
  • PR3：unit-tests 改 4-shard matrix（commons-and-starters / registry / ingest / catalog-and-misc），fail-fast: false + cache-read-only: true
  • 每个 PR 描述贴 3 个连续 run 实测耗时（AC3）
  • Required Status Check（required-check）从 dummy 改为真正依赖测试 job，让分支保护强制生效
Out of Scope
  • AC5b fork 数翻倍（maxParallelForks = availableProcessors()）— 调研判定收益不确定，需要 build scan baseline 后另开 Issue
  • JUnit 5 intra-JVM 并行（parallel.enabled）— 风险高于收益，与 fork 并行是不同层级，与 PAP-18 治理范围不重叠
  • Spring Context 跨 IT 共享（需逐个审计 @DirtiesContext / @MockitoBean）— 另开 Issue
  • 镜像版本集中到 libs.versions.toml + 代码读取改造 — 选择更轻量的 CI 一致性校验方案替代
  • AppCDS / JDK 启动调优、Develocity remote cache、补 unit test、提升测试覆盖率

3. 目标与收益预期

阶段CI 总时长关键路径主要收益来源
当前 baseline (PAP-10) ~8 min 单 job 串行 check → IT → e2e → coverage —
PR1 后 ~6 min 单 job 串行，但冷启动压缩 cc cache hit 省 ~25-30s + docker pull 完全藏入编译时间 ~30-50s + 去掉 --no-daemon 让 daemon 复用
PR2 后 ~5 min max(unit, IT, e2e) + coverage 3 job 并行；unit job 关 jacocoTestReport 报告生成省 5-15%
PR3 后 ~3-4 min max(shard×4, IT, e2e) + coverage unit 4-shard 并行；理想下单 shard ~1.5-2 min

实测达标判据：每个 PR merge 前在 PR description 贴 3 个连续 run 数据；若任一 PR 实测显示净负收益，按对应 PR 的回滚条件回退。

4. 最终态 Job 拓扑（PR3 落地后）

PR/push 触发
  │
  ├─ unit-tests (matrix shard × 4, parallel, fail-fast: false)
  │   ├─ shard: commons-and-starters
  │   │     :linqibin-commons:* + :patra-starters:*
  │   ├─ shard: registry
  │   │     :patra-api:patra-registry:*
  │   ├─ shard: ingest
  │   │     :patra-api:patra-ingest:*
  │   └─ shard: catalog-and-misc
  │         :patra-api:patra-catalog:* + gateway-boot + common:* + object-storage:*
  │   每 shard: ./gradlew <modules>:test -x jacocoTestReport
  │   setup-gradle: cache-read-only: true (避免竞争写)
  │   upload-artifact: jacoco-exec-${{ matrix.shard.name }}
  │
  ├─ integration-tests (single job)
  │   Pre-pull postgres:17 & apache/rocketmq:5.3.1 并发
  │   ./gradlew integrationTest -x integrationTestCodeCoverageReport
  │   upload-artifact: jacoco-exec-integration
  │
  ├─ e2e-tests (single job)
  │   Pre-pull 同上
  │   ./gradlew e2eTest
  │
  └─ coverage-and-upload (needs: unit-tests, integration-tests)
      download-artifact: jacoco-exec-*
      ./gradlew jacocoAggregatedReport
      Codecov upload

汇总 gate:
  required-check (needs: ALL above, if: always())
    使用 re-actors/alls-green@release/v1 校验所有依赖 success/skipped
    任一非 success 则失败 → branch protection 真正生效

5. PR1 — 基础设施层（低风险高收益打底）

目标：在不改 job 拓扑的前提下，把冷启动 3:05 大段压缩到 ~1:30。

改动：

  • .github/workflows/ci.yml：删所有 --no-daemon（4 处：check / integrationTest / e2eTest / jacoco aggregate）
  • .github/workflows/ci.yml：setup-gradle@v4 增加 cache-encryption-key: ${{ secrets.GRADLE_ENCRYPTION_KEY }}，让 cc cache 跨 run 持久化
  • .github/workflows/ci.yml：新增 step 「Pre-pull container images」，置于 setup-gradle 之后、check 之前；用 docker pull postgres:17 & docker pull apache/rocketmq:5.3.1 & wait。GHA step 串行执行但 step 内部并发，pull 总时长 ~40-50s。由于 check 阶段 Gradle 编译占主导（~3 min），pull 整体藏在编译路径下
  • .github/workflows/ci.yml：新增 step 「Verify container image versions consistency」。workflow 顶部 env: 声明 POSTGRES_IMAGE: postgres:17 / ROCKETMQ_IMAGE: apache/rocketmq:5.3.1；step 用 grep 校验 4 个文件中的镜像声明与 env 一致：
    • linqibin-commons/linqibin-spring-boot-starter-test/src/main/java/dev/linqibin/starter/test/container/initializer/PostgreSQLContainerInitializer.java（POSTGRESQL_IMAGE 常量）
    • linqibin-commons/linqibin-spring-boot-starter-test/src/main/resources/docker-compose-rocketmq.yml（2 个 image: 行）
    • patra-api/patra-ingest/patra-ingest-boot/src/test/resources/docker-compose-rocketmq.yml（2 个 image: 行）
    不一致则 fail，强制 PR 同时更新 4 处

GitHub repo secret 一次性手工准备：

  1. 本地生成密钥：openssl rand -base64 16
  2. GitHub repo → Settings → Secrets and variables → Actions → New repository secret，名 GRADLE_ENCRYPTION_KEY，值为上一步输出
  3. 此动作在 PR1 description 里给出明确指令；PR1 不依赖此 secret 也能 merge（缺失时 cc cache 不生效，但 CI 不会失败，因为 setup-gradle 容忍缺失）

不在 PR1 范围：job 拆分、sharding、JaCoCo 配置改动、build-logic 改动。

预期实测：~8 min → ~6 min。

回滚条件：若 cc cache 持续降级（CI 日志反复出现 "Calculating task graph as no cached configuration is available"），从 setup-gradle 移除 cache-encryption-key 临时回退；gradle.properties 中的 org.gradle.configuration-cache=true 不动（本地构建仍受益）。

6. PR2 — Job 拓扑重构 + JaCoCo Aggregation

目标：从单 job 串行升级为 3 job 并行，引入 coverage aggregation 框架，required-check 从 dummy 升级为真正 gate。

改动：

  1. 新建 convention plugin build-logic/src/main/kotlin/linqibin.jacoco-aggregation.gradle.kts：
    • 应用 jacoco-report-aggregation plugin
    • 定义新 task jacocoAggregatedReport：扫描 **/build/jacoco/*.exec（含 download-artifact 解出来的位置）、聚合所有项目的 classes/sources、输出 XML + HTML 到 build/reports/jacoco/aggregated/
    • 沿用现有 exclude：net/sf/jsqlparser/**
    • 挂载到根项目（settings.gradle.kts 已有 includeBuild build-logic，无需额外引用）
    单一职责 convention plugin，命名风格与现有 linqibin.spotless / linqibin.spotbugs 一致。
  2. .github/workflows/ci.yml 重构 jobs：
    • unit-tests（先单 job）：./gradlew check -x jacocoTestReport + upload-artifact jacoco-exec-unit 含 **/build/jacoco/test.exec
    • integration-tests：含 PR1 的 docker pull + ./gradlew integrationTest -x integrationTestCodeCoverageReport + upload-artifact jacoco-exec-integration
    • e2e-tests：含 docker pull + ./gradlew e2eTest（不上传 coverage artifact，沿用现状）
    • coverage-and-upload（needs: [unit-tests, integration-tests]）：download-artifact jacoco-exec-* → ./gradlew jacocoAggregatedReport → Codecov upload
    • required-check 改为 needs: [unit-tests, integration-tests, e2e-tests, coverage-and-upload] + if: always() + 使用 re-actors/alls-green@release/v1 action 校验所有依赖 success/skipped；任一非 success 则失败
    • reports / spotbugs-reports artifact 上传从汇总改为每 job if: always() 各自上传
  3. build-logic/src/main/kotlin/linqibin.java-base.gradle.kts 微调：jacocoTestReport task 保持现状（本地 ./gradlew check 仍会顺带跑），CI 通过 -x 跳过

Spike：PR2 merge 前在本地验证 jacoco-report-aggregation plugin 对 Java 25 + Spring Boot 4 + 30+ 模块结构的兼容性：

  • 本地跑 ./gradlew test integrationTest jacocoAggregatedReport
  • 确认输出 XML 路径、格式符合 Codecov 期望（codecov-action 自动识别 jacoco XML 格式即可）
  • 确认 exclude pattern net/sf/jsqlparser/** 生效

Spike 失败 fallback：coverage-and-upload job 改为重跑全量 ./gradlew testCodeCoverageReport integrationTestCodeCoverageReport（即「双跑」模式），与原 ci.yml 行为一致，仅 coverage job 与 unit/IT 并行。

预期实测：~6 min → ~5 min（关键路径 = max(unit, IT, e2e) + coverage，并行度从 1 升到 3）。

回滚条件：

  • required-check 与 branch protection 联动异常（PR 永远不能 merge 或永远绿）→ 临时把 required-check 改回独立 dummy job，分支保护规则 contexts 不动
  • jacoco-report-aggregation 在 CI 环境失败 → 走 fallback 双跑模式

7. PR3 — Unit Sharding

目标：把 unit-tests 单 job 改为 4-shard matrix，压缩 unit 关键路径从 ~3 min 到 ~1.5-2 min。

改动：

  1. .github/workflows/ci.yml 改 unit-tests job：
    • 添加 strategy.matrix.shard，4 个 shard 各含 name 和 modules（直接写 Gradle 命令参数，不引入辅助脚本）：
    strategy:
  fail-fast: false
  matrix:
    shard:
      - name: commons-and-starters
        gradle-tasks: >-
          :linqibin-commons:linqibin-commons-core:test
          :linqibin-commons:linqibin-commons-storage:test
          :linqibin-commons:linqibin-spring-boot-starter-core:test
          :linqibin-commons:linqibin-spring-boot-starter-web:test
          :linqibin-commons:linqibin-spring-boot-starter-jpa:test
          [...剩余 starters...]
      - name: registry
        gradle-tasks: >-
          :patra-api:patra-registry:patra-registry-domain:test
          :patra-api:patra-registry:patra-registry-app:test
          :patra-api:patra-registry:patra-registry-infra:test
          [...]
      - name: ingest
        gradle-tasks: >-
          :patra-api:patra-ingest:patra-ingest-domain:test
          [...]
      - name: catalog-and-misc
        gradle-tasks: >-
          :patra-api:patra-catalog:patra-catalog-domain:test
          [...] + gateway-boot + common:* + object-storage:*
    实际命令展开到所有模块的 :test task，单 PR 一次性维护清单，模块新增/删除时改一处
  2. setup-gradle@v4 在 shard job 中加 cache-read-only: true（避免 N 个 shard 竞争写 cache 导致污染）
  3. upload-artifact 名称加 shard 区分：jacoco-exec-${{ matrix.shard.name }}
  4. coverage-and-upload job 用 glob pattern download：pattern: jacoco-exec-* + merge-multiple: true

预期实测：~5 min → ~3-4 min。

回滚条件：

  • 实测显示单 shard 时长 > 单 IT 时长（说明 unit 不再是瓶颈，sharding 收益小于复杂度成本）→ 回退到 PR2 形态的单 unit job
  • matrix 任一 shard 连续 3 个 run cache miss 率 > 50% → 排查 cache key 设计，必要时调整 shard 划分
  • 4 shard 模块清单遗漏某模块（全量 unit test 覆盖断层）→ 在 spec 检视阶段用 ./gradlew projects 输出对照清单

8. AC3 实测与回滚机制

每个 PR 在 merge 前必须在 PR description 贴对照表：

Run触发总时长关键路径分解
baselinemain 上最近 3 run（PR1 前）——
本 PR run 1-3本 PR 连续 3 个 push——
结论——是否达预期、是否触发回滚

数据来源：GitHub Actions UI 各 job 的 wall-clock；关键路径分解从最长 job 取值。若某 PR 实测净负收益，按对应 PR 节中的回滚条件处理；不达预期但仍正收益 → 接受，写明原因。

9. 关键决策与备选

决策 1：顺便修补 Required Status Check 配置错误

探索发现实际 Required Status Check 是 1 min dummy required-check job（无 needs:，独立 echo），build-and-test 不是必需检查。即 PAP-15 分支保护实际处于「摆设」状态——任何 PR 即使测试全挂仍可 merge。本 spec 在 PR2 把 required-check 改为 needs: [unit-tests, integration-tests, e2e-tests, coverage-and-upload] + if: always() + alls-green，让分支保护真正强制生效。

备选 A：本 Issue 只做 CI 提速，分支保护 bug 另开 Issue — 否决：拆 job 拓扑后 required-check 无论如何都要改，分两个 Issue 会反复改 ci.yml
备选 B：本 Issue 改 ci.yml 但不动 branch protection rule — 已实质等同于「顺便修补」，因为 branch protection contexts 仍指向 required-check，仅是不明确改动 PAP-15 范围

决策 2：unit test 加速选 4-shard sharding，不动 fork 数

调研显示 AC5a sharding 与 AC5b fork 翻倍性质不同：sharding 业界 3-4 shard 是甜点（单 shard 冷启动 ~30-60s 叠加），4 shard 起步保守稳健；fork 翻倍调研判定「方向对但收益不确定」（JVM startup 200-500ms/进程，可能净负）。本 spec 做 sharding 不做 fork 调整，AC5b 推迟至后续 Issue，先收集本 Issue 落地后的 build scan baseline 再决策。

备选 A：两者都做，fork=availableProcessors 收集数据后决定保留 — 否决：变量太多，AC3 实测难以归因；推迟 AC5b 损失最多 1 min 收益，可接受
备选 B：只做 fork 不做 sharding — 否决：不解决 unit 真测试 4:30 主瓶颈，目标无法达成
备选 C：都不做 — 否决：与本 Issue 目标不符

决策 3：JaCoCo 用 shard 关 report + aggregation merge .exec 模式

shard job 加 -x jacocoTestReport（agent 仍记录 .exec，跳过报告生成）→ upload-artifact .exec → coverage job 用 Gradle 原生 jacoco-report-aggregation plugin 合并出报告。无重复跑测试，总算力最省。

备选 A：shard 完全关 JaCoCo agent（-Djacoco.skip=true）+ coverage job 重跑全量 unit/IT — 否决：「双跑」浪费算力且拉长总 CI 时长；agent 开销 <10% 不值得完全关
备选 B：shard 关 report，coverage job 重跑 testCodeCoverageReport — 否决：仍要跑全量 unit test，本质同备选 A

决策 4：镜像版本治理用 CI yaml 一致性校验，不改 Java/compose 源

workflow 顶部 env: 声明镜像版本，新增 step 用 grep 校验 4 个文件（Java 常量 + 2 份 compose）镜像版本与 env 一致；不一致则 fail。零构建逆依赖，添加一重防护。

备选 A：集中到 gradle.properties + system property + processResources 占位符替换 — 否决：构建逻辑复杂度过高，仅为镜像版本治理引入跨层耦合，YAGNI
备选 B：仅 README 加修改 checklist — 否决：依赖人能纪律，无强制力

决策 5：PR 拆分用 3 PR 收敛而非 5 PR 完全拆分

PR1（基础设施层 4 个高内聚动作）+ PR2（Job 拓扑重构含 JaCoCo aggregation，两者必然耦合）+ PR3（最复杂结构变更，单独便于回滚和实测调优）。

备选 A：5 PR 完全拆分（AC1 / AC4 / AC2 / AC5c / AC5a 各一 PR） — 否决：PR2/PR3/PR4 都改 ci.yml，merge 冲突频繁；收益归因清晰度提升不足以抵消 churn
备选 B：一锅端单 PR — 否决：变量太多，AC3 实测每个 AC 收益归因困难；CI yaml 重写出问题难定位

10. 风险与缓解

风险等级缓解
cc cache 持续降级（CI 日志反复 "Calculating task graph"），收益不达预期 mid PR1 实测验证；若降级，从 setup-gradle 移除 cache-encryption-key 临时回退；排查是否 build-logic 中存在 cc 不兼容代码（如 task action 中访问 project）
jacoco-report-aggregation plugin 与 Java 25 / Spring Boot 4 / 30+ 模块结构不兼容 mid PR2 spike 本地先验证；fallback 走「双跑」模式（coverage job 重跑全量带 JaCoCo），与原 ci.yml 行为一致
alls-green 与 branch protection 联动异常（PR 永远不能 merge / 永远绿） mid PR2 在测试 PR 上验证 alls-green 行为后再 merge；异常时临时把 required-check 改回独立 dummy，分支保护 contexts 不动
4 shard 模块清单遗漏某模块，全量 unit test 覆盖断层 mid PR3 spike：用 ./gradlew projects 输出对照 shard 清单；CI 加 assertion step 检查所有模块都被某 shard 覆盖（可选）
shard 后单 shard 冷启动叠加吃掉收益，CI 总时长反而增加 low AC3 实测 3 个连续 run 验证；不达预期回退到 PR2 形态单 unit job；4 shard 而非 7 shard 已主动控制风险
matrix shard 缓存竞争导致 cache miss 率高 low cache-read-only: true 已在设计中包含；监控 setup-gradle 输出的 cache hit/miss 日志
GitHub Free plan 单仓 20 concurrent jobs 上限 low 4 unit shard + IT + e2e + coverage + required-check = 8 jobs，远低于 20；多 PR 同时跑时 GitHub 自动排队
Docker pull 网络抖动 / Docker Hub rate limit low pull 失败 fail-fast；监控 PR1 后 docker pull step 失败率；必要时切换 GHCR mirror 或 docker login
