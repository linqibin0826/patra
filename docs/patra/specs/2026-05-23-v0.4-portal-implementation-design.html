关键决策摘要

本 spec 覆盖 v0.4 Portal Foundation 剩余 6 个 FE Issue（PAP-28 / PAP-30 / PAP-31 / PAP-32 / PAP-33 / PAP-34）在一个 PR内合并落地的实现路径。核心决策：① Tailwind-first 样式策略（handoff 作为视觉真相，所有 CSS class 改写为 Tailwind utility，禁用 CSS Modules / inline style / @apply）；② 两份 handoff 拆分职责（design system zip = shadcn 主题覆盖 + 设计原则真相；portal handoff zip = 6 区块首页落地真相）；③ 按层拆 6 个 commit（tokens / mock / 组件 / 装配 / 响应式 / e2e），squash-merge 时压成 1 个；④ 关键节点 fan-out subagent（Stage 1 token+mock 并行；Stage 2 组件库按区块分 3 个 subagent；Stage 3 主对话亲自做 page 装配 / 响应式 / e2e）；⑤ 忠实 handoff fidelity，与 release-spec 文档冲突时以 handoff 为准（区块名 TopicCloud / Journals、mock 32 topic / 6 journal / 14 paper、响应式 4 档 880/720/640/1200）。

3.2 RSC / Client 边界规则

默认 Server Component；仅当组件需要 useState / useEffect / 事件处理时加 'use client'。本 PR 共 4 个 client component：

  • TopNav：汉堡菜单 open/close state + Sheet 触发
  • Composer：active tab + input value + onSubmit handler（Hero 的 RSC 子组件）

其余 6 个 portal 组件（Hero / TopicCloud / Journals / JournalCoverCard / PaperCard / AISummaryBadge / Footer）+ page.tsx 全部 RSC。

4.2 Subagent fan-out 计划

遵循 memory rule "大批量同构工作 fan out subagents"。三个 Stage：

Stage 1（并行 2 个 subagent → 合在 C1+C2 提交）：
  SA-tokens (C1)   ─┐
                    ├─→ 主对话 review → commit C1 + C2
  SA-mock (C2)     ─┘

Stage 2（并行 3 个 subagent → 合在 C3 提交）：
  SA-nav     (TopNav + Footer + AISummaryBadge)              ─┐
  SA-hero    (Hero + Composer)                                ├─→ 主对话 review → commit C3
  SA-feed    (TopicCloud + Journals + JournalCoverCard
              + ExploreFeed + PaperCard)                      ─┘

Stage 3（主对话亲自做，3 个独立 commit）：
  主对话: page.tsx 装配 (C4) → 响应式调试 (C5) → e2e + CI (C6)

Stage 3 不 fan-out 的理由：page 装配 / 响应式 / e2e 都需要看着浏览器或 dev server 反复调，subagent 没有视觉反馈，不如主对话 + 真实 dev server 高效；且这三步彼此强依赖（page 完才能响应式，响应式完才能 e2e 验证）。

4.3 Subagent 的 TDD 约束

每个 SA 必须按红-绿-重构循环：

  1. 先写 .test.tsx，列 3-5 个用例 → pnpm test 红
  2. 写最小组件实现 → 测试转绿
  3. 重构样式 / 抽 prop → 保持绿
  4. SA 回传：组件文件 + 测试文件 + 测试通过证据（pnpm test 输出片段）

测试查询规则（来自 .claude/rules/testing.md）：getByRole > getByLabelText > getByPlaceholderText > getByText > ~~getByTestId~~（禁止作为首选）。禁止 snapshot 测试。禁止断言实现细节。

5.3 类型设计决策

  • 数组用 readonly Topic[] 而非 as const——后者字面量类型过窄，组件 props 不好用
  • topicTier() 与 TOPIC_CLOUD 同文件（topics.ts），因为它是基于阈值数据的派生函数
  • PaperSource 用字面量联合而非 string——防止笔误
  • minutesAgo? 可选：只 recent tab 部分 paper 有
  • noUncheckedIndexedAccess 已开 → 访问 JOURNALS[0] 得 Journal | undefined，组件用 .map() 避免

9. 关键决策与备选

决策 1：Tailwind-first 样式策略，禁用 CSS Modules

handoff bundle 是设计真相（视觉、信息层级、token 数值），不是代码真相。所有 CSS class 改写为 Tailwind utility（含 before: / after: / aspect-[3/4] / text-[clamp()] arbitrary value / snap-x snap-mandatory）。少量全局基础（body 字体 / ::selection / @keyframes patra-live-pulse）留 globals.css 的 @layer base。

备选 A：CSS-first（搬 homepage.css 原样，组件用 className="composer-tab" 引用）— 否决：与 portal 现有 Tailwind utility + shadcn 风格混搭，违反 .claude/rules/code-style.md "Class 顺序由 Biome 插件自动排" 与 "禁止 inline style"
备选 B：混合（tokens + 局部 .module.css）— 否决：两套体系并存的认知成本；CSS Modules 与 Biome / Tailwind 工具链协同薄弱

决策 2：6 个 Issue 单 PR 合并 + 按层拆 6 个 commit

所有 6 个 FE Issue（PAP-28/30/31/32/33/34）在 feat/v0.4-portal-foundation 单分支单 PR 内合并。PR 内按层拆 6 个 commit（tokens / mock / 组件 / 装配 / 响应式 / e2e），squash-merge 时压成 1 个；中间状态便于 review 与回滚。

备选 A：每 Issue 单独 PR — 否决：用户明确要求合并到一个 PR；单 Issue PR 之间依赖链强，串行 review 反而慢
备选 B：按 Issue 拆 commit（每 commit 对应一个 Linear issue 号）— 备选可行，但层次切分更贴近"代码改动维度"而非"Issue 管理维度"，且 squash-merge 后等价

决策 3：handoff 为 source of truth，与 release-spec 冲突时以 handoff 为准

release-spec 描述与 hi-fi 实际产出存在 3 处偏差（区块名 / mock 量级 / 响应式断点），统一以 handoff 为准。release-spec 文档本身不修改（作为版本规划历史保留）。Linear PAP-31/32/33 标题已同步跟齐 handoff。

备选 A：release-spec 为准（按 8 topic / 5 journal / 768 单断点严格落地）— 否决：违背 PAP-29 hi-fi 已交付事实；视觉与数据规模会失真
备选 B：混合（视觉细节按 handoff，量级按 release-spec）— 否决：量级砍半后 TopicCloud 5 级 tier 与 ExploreFeed 三 tab 都填不满，破坏视觉密度

决策 4：关键节点 fan-out subagent（3 Stage）

Stage 1：token (C1) + mock (C2) 并行 2 个 SA。Stage 2：公共组件库 (C3) 按区块分组 3 个 SA（nav/hero/feed）。Stage 3：主对话亲自做 page 装配 (C4) / 响应式 (C5) / e2e (C6)，因这三步需要看着 dev server 反复调，subagent 无视觉反馈。

备选 A：完全串行 — 否决：违反 memory rule "大批量同构工作 fan out subagents"
备选 B：全 fan out（6 issue 各一个 SA）— 否决：PAP-32/33/34 强依赖前序产出，SA 互等反而慢

决策 5：补装 shadcn 6 个原语，不多装

补装 input / card / badge / tabs / sheet / sonner 共 6 个，达到 v0.4 最小集。dropdown / dialog / tooltip 不装（v0.4 用不到，YAGNI）。

备选 A：多装 3 个预热（dropdown/dialog/tooltip） — 否决：违反 YAGNI 与 CLAUDE.md "禁止以"如果时间允许""
备选 B：少装（不装 badge + 不装 sonner） — 否决：badge 直接对应 design system 的 chip-badge 规范；sonner 是 v0.4 spec 明确要求的 toast 库

决策 6：Server Component 优先，仅 4 个 client component

page.tsx + 6 个静态组件全部 RSC。只有 TopNav（汉堡 state）/ Composer（tab+value state）/ ExploreFeed（active tab）/ Toaster（sonner 自身要求）4 个用 'use client'。

备选 A：page.tsx 整体 'use client' 一刀切 — 否决：违反 patra-portal/CLAUDE.md "Server Components 优先"
备选 B：更细粒度（TopicCloud 也 client，便于交互动画） — 否决：v0.4 不实现 topic 点击跳转，TopicCloud 是纯展示

决策 7：Playwright e2e 非阻断式（continue-on-error: true）

v0.4 加 e2e job 到 portal-ci，但 continue-on-error: true，不进入 aggregator 的 needs 列表。理由：webServer 启动 + dev build 在 CI 不稳定（pnpm dev 冷启可能超 60s）；先观察 1-2 个 PR 稳定度。v0.5 视情况切换阻断式。

备选 A：直接阻断（required check）— 否决：CI 冷启时间不稳定，可能让 PR 卡住
备选 B：不加 e2e 到 CI，只本地跑 — 否决：违反 PAP-34 AC "CI 中 e2e job 跑通"

决策 8：响应式 4 档断点而非"880/720 双断点"

PAP-33 标题写"880/720 双断点"是简化叙述。忠实 handoff hi-fi 实际用了 4 档（1200 / 880 / 720 / 640）。L1=1200 仅 Journals 6→3 列；L4=640 仅 Hero composer 紧凑。AC 验证矩阵覆盖 4 个视口（375/800/1024/1440）。

备选 A：严格 880/720 双档 — 否决：删除 1200 后 Journals 在 880-1200 段仍 6 列挤；删除 640 后 mobile composer 字号不收窄
备选 B：用 Tailwind 默认 md/lg/xl 断点 — 否决：handoff 视觉切换点是 880 而非 768，错位会破坏 hi-fi 视觉

10. 风险与缓解

风险等级缓解
shadcn 标准 slot 与 Patra tokens 命名空间冲突（如 --radius 同时被 shadcn 和 Patra 使用） mid 在 @theme inline 内明确指定优先级（Patra 的 --radius-lg 通过 --radius: var(--radius-lg) 桥接给 shadcn）；tokens.css 内的 --font-sans 重命名为 --patra-font-sans
Tailwind v4 arbitrary breakpoint (max-[880px]:) Biome 误报 unknown-selector low biome.json 已配 tailwindDirectives: true；实施前在 SA-tokens commit 内跑 pnpm lint 验证 arbitrary 写法被识别
next/font 5 字体首屏加载体积膨胀（5 个 woff2 文件 + Noto SC 子集大） mid 所有字体设 display: "swap"；Noto SC 限定 weight=["400","500","600","700"]（不加载 italic）；首屏验证 LCP 仍可接受；后续如确需优化，v0.5 自托管 + subset
RSC / Client 边界设错（TopNav 等含 state 的组件忘加 'use client'） low Next.js 编译期会报错（useState in server component），typecheck 阶段拦截；Vitest 单测在 jsdom 环境下也会暴露
handoff 的 32 个 topic / 14 个 paper mock 数据移植笔误（中文标点 / hex 颜色 / URL） mid SA-mock 用脚本化方式从 handoff data.jsx 提取（regex + 手工转换），不手抄；测试用 JSON.stringify diff 校验
shadcn <Sheet> 默认动画 / 主题色与 Patra warm paper 不匹配 low 所有 shadcn 颜色由 --background / --popover / --border 等 slot 驱动，已在 globals.css 重写为 Patra tokens；动画时长用 Patra --dur-fast/normal/slow 替换默认
Playwright webServer 在 CI 冷启 > 60s，e2e job 超时 mid webServer.timeout: 120000；e2e job 本版 continue-on-error: true，超时不阻断 PR；记录冷启时间，v0.5 切换为 pnpm build && pnpm start 跑 e2e 减少冷启
Stage 2 三个 subagent 并行写组件时，src/components/portal/ 同目录可能冲突 low 每个 SA 分到不同组件文件（按 §4.2 分组），无重叠；主对话 review 时 git status 验证无意外文件
tokens.css 内仍含 .patra { ... } 语义元素样式（hairline / h1-h4 / .eyebrow），与 Tailwind utility 双轨 mid tokens.css 仅保留 :root 内的 CSS 变量定义，删除 .patra 命名空间内所有语义元素样式（h1/h2/.eyebrow/.reading 等），改由 Tailwind utility 在组件内表达；保留 tokens.css 顶部注释说明此调整
Geist 字体在 layout.tsx 移除后，可能 portal-ci 历史 smoke test 引用 font-sans 触发回归 low 现有 smoke.test.tsx 只测 h1 文本存在，不测字体；C1 commit 后跑一遍 pnpm test 验证现有测试全绿
