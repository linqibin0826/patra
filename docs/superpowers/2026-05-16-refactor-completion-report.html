01
The Brief

一句抱怨，是十八个小时的起点。

单人项目、绿地代码库、零历史包袱 — 仍然有人在意未来。

Conversation start
2026·05·15
Initial commit:
f552d2205
"我发现这个产品没有按我的组织进行命名。直接用了 com.patra。
这样我以后开发别的产品的时候就很尴尬。" — Linqibin, project owner

Patra 是医学出版物数据平台 — 采集、解析、存储来自 PubMed / EPMC / Crossref 等 10+ 外部数据源的文献数据。微服务 + 六边形 + DDD + 事件驱动， Java 25 / Spring Boot 4.0.1 / Gradle 9.2.1。21 个 Gradle 模块，4 个微服务（registry / ingest / catalog / gateway）+ 13 个 Spring Boot starter + 4 个 patra-common-* + build-logic。

项目顶层包名 com.patra 跳过了 Java 反向域名约定的"组织"层（com.{org}.{product}）。用户拥有个人域名 linqibin.dev， 希望所有自有项目共用 dev.linqibin.* 命名空间，建立长期可演化的产品矩阵。

Goal 把 com.patra 重命名为 dev.linqibin.patra （业务侧），同时把通用基础设施（CQRS、DDD 基类、错误处理、JSON、对象存储等）从 patra-common-* 抽到独立的 linqibin-commons-* 模块下（dev.linqibin.commons.*），为未来产品复用做准备。

Refactor-Caused Failures

"编译通过 ≠ 行为正确" — 4 个真重构 bug。

主重构完成后跑 ./gradlew test，暴露 25 个失败。诊断后 4 类问题，分别 4 个独立 commit 修复。

A linqibin-commons-core slf4j compileOnly 导致 test runtime 缺失
b5270bf85
影响 17 tests — RemoteCallExceptionTest + RemoteErrorHelperTest 全部嵌套 class
症状 java.lang.NoClassDefFoundError: Could not initialize class
dev.linqibin.commons.error.remote.RemoteCallException
Caused by: NoClassDefFoundError: org/slf4j/LoggerFactory
根因 Phase 2.1 implementer 照搬 patra-common-core 的 compileOnly("org.slf4j:slf4j-api")。原模块的 test classpath 通过某 transitive 桥拿到 slf4j；新模块没了那个桥。@Slf4j 注解的类静态初始化时找不到 LoggerFactory。
修复 
  compileOnly("org.slf4j:slf4j-api")
+ testRuntimeOnly("org.slf4j:slf4j-api")
B ArchUnit 白名单缺 dev.linqibin.commons..
cfd080e1a
影响 2 tests — IngestArchitectureTest + CatalogArchitectureTest 共 95 个 ArchUnit 违规（52 + 43）
症状 Class <dev.linqibin.patra.ingest.domain.model.aggregate.PlanAggregate>
extends class <dev.linqibin.commons.domain.AggregateRoot>
Architecture Violation [Rule 'Domain 层仅允许依赖...']
根因 Post-review Issue 6 把 HexagonalArchitectureRules 参数化（commit 5e088ccf2）。测试调用方只传了 "dev.linqibin.patra.common.."，遗漏 "dev.linqibin.commons.." — Domain 层用 AggregateRoot 等通用类被误判违规。
修复 
  new HexagonalArchitectureRules(
      "dev.linqibin.patra.ingest",
      List.of("dev.linqibin.patra.common..",
              "dev.linqibin.commons.."))
C StorageEndpointIT 双 bug：路径错位 + @Valid 重定义
9f4ecbde9
影响 7 tests — StorageEndpointIT 全部
症状 Status expected:<200 OK> but was:<404 NOT_FOUND>
(修完路径后)
HV000151: A method overriding another method ...
根因 ① URL 路径错配：测试 URL /internal/，Controller 实际 /_internal/（带 _ 前缀的内部 API 约定）— 某 subagent 在 sed rename 时把字符串字面量误改。
② @Valid 重定义：StorageEndpointImpl.recordUpload 上有 @Override @Valid，但接口 StorageEndpoint 没有 — Hibernate Validator HV000151 禁止 override 时重定义 parameter constraint。
修复 测试 URL 全部加回 _ 前缀 · @Valid 上移至 interface，impl 删除重定义
D PublicationDataReadyEventTest null vs [] 期望错位
f22853e0a
影响 2 tests — BuilderTests.shouldSupportNullFields + ToStringTests.shouldHandleNullFields
症状 expected: null but was: []
expected: "storageKeys=null" but was: "storageKeys=[]"
根因 PublicationDataReadyEvent Record 紧凑构造器按 code-style 规范做防御性拷贝 storageKeys != null ? List.copyOf(...) : List.of() — null 被归一为空集合。测试期望 null 是 by-design 错误。

Note on Cold-start Cascading Flaky

./gradlew clean test --parallel 第一次跑时偶发 cascading flaky — Gradle parallel test worker 在 cold-start 大量并发时 JVM 类加载 / jar 读取竞争触发 Spring 的 ApplicationContext failure threshold (1) exceeded。所有失败测试 isolated rerun 或热缓存跑 100% 通过。这是 Gradle infrastructure 行为，非代码 bug。修法是设 maxParallelForks = 1-2 或加 test retry 插件 — 独立的 test infra 优化，不在本次范围。
