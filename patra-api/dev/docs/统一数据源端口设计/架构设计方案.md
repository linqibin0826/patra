## 1. 问题背景

### 1.1 当前问题

Patra Ingest 模块中存在两个用于访问 PubMed 数据源的端口：

```
计划阶段：PubmedSearchPort（初步代码，未整体设计）
  └─ preparePlanMetadata(query, params, config)
     → PlanMetadata(count, webEnv, queryKey)

执行阶段：DataSourcePort（规范化设计）
  └─ fetchData(context, dataType, typeRef, batch)
     → DataFetchResult<T>
```

**核心问题**：
1. **过度抽象**：PubmedSearchPort 仅提取 3 个字段，为此专门定义端口和适配器
2. **架构不一致**：计划和执行使用不同的抽象层次
3. **可扩展性问题**：未来每个数据源都需要定义专门的 SearchPort
4. **职责割裂**：计划和执行本质上是紧密相关的操作

### 1.2 框架层能力

`patra-spring-boot-starter-provenance` 已提供完整的 PubMed 封装：

```java
// 低层次 API 客户端
PubMedClient
├─ esearch() → ESearchResponse (count, webEnv, queryKey, idList)
├─ efetch() → EFetchResponse
└─ epost() → EPostResponse

// 高层次数据提供者
DataSourceProvider
└─ fetchData() → ProviderResult<T>
```

### 1.3 设计目标

作为绿地项目，我们追求：
- ✅ **最优架构**：不考虑成本，追求技术卓越
- ✅ **长期可维护性**：清晰的职责边界和依赖关系
- ✅ **高度可扩展性**：支持未来多个数据源（EPMC、DOAJ、DrugBank）
- ✅ **类型安全**：编译时检查，避免运行时错误

## 2. 现状分析

### 2.1 当前架构图

```
计划阶段：
PubmedBatchPlanner → PubmedSearchPort ← PubmedSearchAdapter → PubMedClient

执行阶段：
GenericBatchExecutor → DataSourcePort ← DataSourceAdapter → DataSourceProvider
```

### 2.2 代码结构

```
patra-ingest/
├── domain/
│   └── port/
│       ├── PubmedSearchPort.java        # ❌ PubMed 特定
│       └── DataSourcePort.java          # ✅ 通用抽象
├── app/
│   └── strategy/planner/
│       └── PubmedBatchPlanner.java      # 使用 PubmedSearchPort
└── infra/
    └── integration/
        ├── pubmed/
        │   └── PubmedSearchAdapter.java # ❌ 薄适配器，仅转发
        └── datasource/
            └── DataSourceAdapter.java   # ✅ 使用 DataSourceProvider
```

### 2.3 问题诊断

#### 问题 1：薄适配器无价值

```java
// PubmedSearchAdapter 的核心逻辑
@Override
public PlanMetadata preparePlanMetadata(...) {
    ESearchRequest request = buildESearchRequest(params);
    ESearchResponse response = pubMedClient.esearch(request, config);
    return extractPlanMetadata(response);  // 仅提取 3 个字段
}
```

**分析**：
- 适配器仅做字段提取，无转换逻辑
- 没有添加实质性价值
- 增加调用链路和维护成本

#### 问题 2：架构不一致

```
计划阶段：PubmedSearchPort（PubMed 特定）
执行阶段：DataSourcePort（数据源通用）

→ 两者访问同一数据源，却使用不同抽象层次
```

#### 问题 3：可扩展性问题

```java
// 如果为 EPMC、DOAJ 也做计划，会导致：
EpmcSearchPort
DoajSearchPort
DrugBankSearchPort
...

→ 端口接口爆炸
```

## 3. 架构师评审过程

为了找到最优方案，我们邀请了 3 位架构专家并行进行设计评审。

### 3.1 架构师 A：分层抽象模式

**核心观点**：计划端口和执行端口应该独立，使用分层继承体系。

#### 方案设计

```java
// 基础抽象
public interface IngestionPlanPort {
    IngestionPlan createPlan(QueryCriteria criteria);
}

// 特定扩展
public interface PubmedPlanPort extends IngestionPlanPort {
    @Override
    PubmedIngestionPlan createPlan(QueryCriteria criteria);
}
```

#### 评估

**优点**：
- ✅ 职责分离清晰（SRP）
- ✅ 类型安全
- ✅ 易于独立演进

**缺点**：
- ❌ 接口数量多
- ❌ 增加复杂度
- ❌ 计划和执行割裂

**结论**：🔶 可行，但可以进一步优化

---

### 3.2 架构师 B：统一端口模式

**核心观点**：计划和执行应统一为单一端口，符合高内聚原则。

#### 方案设计

```java
public interface DataSourcePort {

    // 新增：准备计划元数据
    PlanMetadata preparePlan(ExecutionContext context, DataType dataType);

    // 已有：获取实际数据
    <T> DataFetchResult<T> fetchData(...);

    // 已有：能力查询
    boolean supports(String provenanceCode, DataType dataType);
    Set<DataType> getSupportedTypes(String provenanceCode);
}
```

#### 元数据继承体系

```java
// 基类
public abstract class PlanMetadata {
    private final String dataSourceType;
    private final int totalCount;
    private final Instant plannedAt;

    public abstract boolean hasSessionToken();
}

// PubMed 特定
public class PubmedPlanMetadata extends PlanMetadata {
    private final String webEnv;
    private final String queryKey;
}

// EPMC 特定
public class EpmcPlanMetadata extends PlanMetadata {
    private final String cursorMark;
}

// DOAJ 特定
public class DoajPlanMetadata extends PlanMetadata {
    private final String scrollId;
    private final int pageSize;
}
```

#### 评估

**优点**：
- ✅ 高内聚 - 相关操作在同一端口
- ✅ 简化依赖 - 领域层只需依赖一个端口
- ✅ 统一抽象 - 所有数据源使用相同接口模式
- ✅ 清晰边界 - 一个端口代表完整的"数据源访问"能力
- ✅ 类型安全 - 使用继承体系而非 Map

**缺点**：
- 无明显缺点

**结论**：✅ 强烈推荐

---

### 3.3 架构师 C：激进简化模式

**核心观点**：删除所有薄适配器，Application 层直接使用 starter-provenance 组件。

#### 方案设计

```java
@Component
public class IngestionPlanningOrchestrator {
    // 直接注入框架提供的 Provider
    private final DataSourceProvider dataSourceProvider;

    public IngestionPlan createPlan(PlanRequest request) {
        // 直接使用，无需适配器
        SearchResult searchResult = dataSourceProvider.search(...);
        return IngestionPlan.create(...);
    }
}
```

#### 核心观点

- 删除 PubmedSearchPort 和 DataSourcePort
- Application 层直接使用 DataSourceProvider
- 只在需要真正适配时创建适配器
- Domain 层可以依赖稳定的数据模型

#### 评估

**优点**：
- ✅ 最简洁 - 减少 50% 调用链路
- ✅ 最直接 - 无中间层
- ✅ 务实 - 不为架构而架构

**缺点**：
- ⚠️ 可能违反六边形架构原则
- ⚠️ Domain 层依赖框架（但如果框架稳定，可接受）
- ⚠️ 失去适配器层的协议转换价值

**结论**：🔶 激进但有启发性

## 4. 最终设计方案

### 4.1 方案名称

**统一务实架构** - 融合架构师 B 和 C 的优点

### 4.2 核心设计

#### 设计原则

1. **统一端口**（来自架构师 B）
   - 计划和执行统一为 DataSourcePort
   - 添加 preparePlan() 方法

2. **继承体系**（来自架构师 A 和 B）
   - PlanMetadata 抽象基类
   - 各数据源实现子类

3. **明确责任边界**（关键原则）
   - **Framework 层**：提供通用的数据源访问能力
     * DataSourceProvider 接口只接收通用参数 `(query, params, config)`
     * 不涉及 Ingest 特定概念（如 ExecutionContext）
     * 返回通用的 PlanMetadata（可在多个模块复用）
   - **Infrastructure 层**：协议转换和参数提取
     * DataSourceAdapter 提取 ExecutionContext 中的通用参数
     * 调用 Framework 层的 DataSourceProvider
     * 转换异常和结果格式
   - **Application 层**：业务逻辑编排
     * 批次生成逻辑保留在 UnifiedBatchPlanner
     * 使用 PlanMetadata 生成批次计划

4. **务实实现**（来自架构师 C）
   - Infrastructure 层直接使用 starter-provenance 的 DataSourceProvider
   - 通过 Map 自动注入所有 Provider 实现
   - 不创建无价值的薄适配器

#### 架构图

```
┌─────────────────────────────────────────────────┐
│          Application 层（Ingest 业务逻辑）        │
│  ┌──────────────────┐  ┌──────────────────┐    │
│  │UnifiedBatchPlanner│  │GenericBatchExec  │    │
│  │（批次生成逻辑）    │  │                  │    │
│  └────────┬──────────┘  └────────┬─────────┘    │
└───────────┼──────────────────────┼──────────────┘
            │                      │
            └──────────┬───────────┘
                       │ 依赖（Ingest 特定接口）
            ┌──────────▼──────────┐
            │   Domain 层（Ingest）│
            │ ┌──────────────────┐│
            │ │ DataSourcePort   ││ ← Ingest 端口
            │ │  - preparePlan() ││   （接收 ExecutionContext）
            │ │  - fetchData()   ││
            │ └──────────────────┘│
            │ ┌──────────────────┐│
            │ │  PlanMetadata    ││ ← 继承体系（通用）
            │ │  ├─ PubmedPlan   ││
            │ │  ├─ EpmcPlan     ││
            │ │  └─ DoajPlan     ││
            │ └──────────────────┘│
            └──────────▲──────────┘
                       │ 实现（参数转换）
            ┌──────────┴──────────┐
            │  Infrastructure 层   │
            │ ┌──────────────────┐│
            │ │DataSourceAdapter ││ ← 参数提取器
            │ │（提取通用参数）   ││   ExecutionContext →
            │ └────────┬─────────┘│   (query, params, config)
            └──────────┼──────────┘
                       │ 使用（通用参数调用）
            ┌──────────▼──────────┐
            │ Framework 层（starter-provenance）│
            │ ┌──────────────────┐│
            │ │DataSourceProvider││ ← Provider 接口
            │ │（PubMed/EPMC 实现）│   （只接收通用参数）
            │ │  - preparePlan() ││   (query, params, config)
            │ │  - fetchData()   ││

            │ ┌──────────────────┐│
```

## 5. 核心优势

### 5.1 高内聚低耦合

```java
// ✅ 优势 1：计划和执行在同一端口
DataSourcePort port = ...;
PlanMetadata plan = port.preparePlan(context, dataType);  // 计划
DataFetchResult<T> data = port.fetchData(context, ...);   // 执行

// 符合"数据源访问"的完整业务概念
```

### 5.2 类型安全

```java
// ✅ 优势 2：使用继承体系，编译时检查
PlanMetadata plan = dataSourcePort.preparePlan(context, dataType);

if (plan instanceof PubmedPlanMetadata pubmedPlan) {
    // IDE 支持代码补全
    String webEnv = pubmedPlan.webEnv();
    String queryKey = pubmedPlan.queryKey();
    // 编译时类型安全
}

// ❌ 对比：使用 Map 的方案
Map<String, Object> metadata = plan.getMetadata();
String webEnv = (String) metadata.get("webEnv");  // 运行时类型转换
```

### 5.3 优秀的扩展性

```java
// ✅ 优势 3：新增数据源只需 3 步
// Step 1: 创建 PlanMetadata 子类
public class NewSourcePlanMetadata extends PlanMetadata {
    private final String specificToken;
}

// Step 2: 在 DataSourceAdapter 中添加方法
private PlanMetadata prepareNewSourcePlan(ExecutionContext context) {
    // ...
}

// Step 3: 在 UnifiedBatchPlanner 中添加批次生成逻辑
private List<Batch> generateNewSourceBatches(NewSourcePlanMetadata plan) {
    // ...
}

// 无需定义新的端口接口！
```

### 5.4 清晰的架构分层

```java
// ✅ 优势 4：依赖关系清晰
Application 层 → Domain 层（DataSourcePort 接口）
                      ↑
Infrastructure 层（DataSourceAdapter 实现）
                      ↓
Framework 层（PubMedClient、DataSourceProvider）

// 遵守依赖倒置原则
// 保持 Domain 层纯净
```

### 5.5 务实的实现

```java
// ✅ 优势 5：直接使用 starter-provenance，无过度封装
@Component
public class DataSourceAdapter implements DataSourcePort {

    private final PubMedClient pubMedClient;  // 直接注入

    @Override
    public PlanMetadata preparePlan(...) {
        // 直接调用，无中间层
        ESearchResponse response = pubMedClient.esearch(request, config);
        return new PubmedPlanMetadata(...);
    }
}
```

## 6. 架构决策

### 6.1 关键决策

#### 决策 1：为什么统一端口？

**理由**：
- 计划和执行是紧密相关的操作，应该高内聚
- "数据源访问"是一个完整的领域概念
- 统一后简化依赖关系

**证据**：
- 计划阶段获取的 WebEnv 在执行阶段使用
- 两者访问同一个数据源
- 执行依赖计划的结果

#### 决策 2：为什么保留适配器层？

**理由**：
- 提供协议转换价值（Domain 模型 ↔ Framework 模型）
- 提供错误处理和异常转换
- 提供日志和监控能力
- 保持 Domain 层纯净

**对比**：
```java
// 如果删除适配器层：
@Component
public class UnifiedBatchPlanner {
    private final PubMedClient pubMedClient;  // Domain 层依赖框架

    public BatchPlan plan(ExecutionContext ctx) {
        // Domain 层直接处理框架模型
        ESearchResponse response = pubMedClient.esearch(...);
        // 需要在 Domain 层处理框架异常
    }
}

// 保留适配器层：
@Component
public class UnifiedBatchPlanner {
    private final DataSourcePort dataSourcePort;  // Domain 层依赖抽象

    public BatchPlan plan(ExecutionContext ctx) {
        // 使用 Domain 模型
        PlanMetadata plan = dataSourcePort.preparePlan(ctx, dataType);
        // 框架异常已被适配器转换为领域异常
    }
}
```

#### 决策 3：为什么使用继承而非 Map？

**理由**：
- 类型安全，编译时检查
- IDE 友好，支持代码补全和重构
- 避免运行时类型转换错误
- 表达清晰的类型关系

**对比**：
```java
// ❌ 使用 Map
public class PlanMetadata {
    private Map<String, Object> metadata;
}
// 问题：运行时类型转换、无类型检查、容易出错

// ✅ 使用继承
public abstract class PlanMetadata { }
public class PubmedPlanMetadata extends PlanMetadata {
    private final String webEnv;
    private final String queryKey;
}
// 优点：编译时检查、类型安全、IDE 支持
```

#### 决策 4：为什么在 Framework 层定义 DataSourceProvider？

**理由**：
- 保持 Framework 层的独立性和可复用性
- DataSourceProvider 是通用的数据源访问接口，不应耦合 Ingest 业务
- 允许其他模块（如未来的 Export 模块）复用相同的数据源能力

**参数设计原则**：
```java
// ❌ 错误：Framework 层接收 Ingest 特定参数
DataSourceProvider.preparePlan(ExecutionContext context)

// ✅ 正确：Framework 层只接收通用参数
DataSourceProvider.preparePlan(String query, JsonNode params, ProvenanceConfig config)
```

**责任分离**：
- Framework 层：数据源访问（技术能力）
- Infrastructure 层：参数转换（协议适配）
- Application 层：批次生成（业务逻辑）

#### 决策 5：为什么批次生成逻辑在 Application 层？

**理由**：
- 批次生成是 Ingest 模块特定的业务逻辑
- 不同业务场景的批次策略不同（如优先级调度、窗口限制）
- Framework 层不应该包含业务逻辑

**对比**：
```java
// ❌ 错误：在 Framework 层生成批次
DataSourceProvider.generateBatches(PlanMetadata plan, ExecutionContext ctx)
// 问题：Framework 层耦合了 Ingest 的批次生成逻辑

// ✅ 正确：在 Application 层生成批次
UnifiedBatchPlanner.generateBatches(PlanMetadata plan, ExecutionContext ctx)
// 优点：业务逻辑保留在业务层
```

#### 决策 6：为什么使用策略模式而非 instanceof？

**理由**：
- 符合开闭原则（OCP）：新增数据源无需修改现有代码
- 避免代码膨胀：10+数据源时不会导致单类1000+行
- 职责清晰：每个策略类独立负责一种数据源
- 与 Map 自动注入哲学一致：保持架构哲学的统一性

**对比**：
```java
// ❌ 不推荐：使用 instanceof（违反 OCP）
private List<Batch> generateBatches(PlanMetadata plan, ExecutionContext ctx) {
    if (plan instanceof PubmedPlanMetadata pubmedPlan) {
        return generatePubmedBatches(pubmedPlan, ctx);
    } else if (plan instanceof EpmcPlanMetadata epmcPlan) {
        return generateEpmcBatches(epmcPlan, ctx);
    }
    // 每次新增数据源都要添加分支
}

// ✅ 推荐：使用策略模式（符合 OCP）
private List<Batch> generateBatches(PlanMetadata plan, ExecutionContext ctx) {
    BatchGenerationStrategy strategy = strategyMap.get(plan.getClass());
    return strategy.generateBatches(plan, ctx);
    // 新增数据源无需修改此方法
}
```

**设计模式说明**：
- **策略模式（Strategy Pattern）**：定义一系列算法，把它们一个个封装起来，并且使它们可以相互替换
- **Map 自动注入**：Spring 自动注册所有 `BatchGenerationStrategy` 实现，运行时动态选择
- **开闭原则**：对扩展开放（新增策略类），对修改关闭（无需改 Planner）

#### 决策 7：PlanMetadata 应该放在哪一层？

**理由**：
- PlanMetadata 是**跨模块的通用领域模型**，不仅限于 Ingest 使用
- 可能被 Ingest、Export、Synchronization 等多个模块复用
- 应该与 `CanonicalLiterature`、`DataType` 等通用模型放在一起

**建议位置**：`patra-common-model` 模块

**模块依赖关系**：
```
patra-common-model（通用领域模型）
├─ PlanMetadata（抽象基类）
├─ PubmedPlanMetadata
├─ EpmcPlanMetadata
└─ DoajPlanMetadata

patra-ingest-domain（Ingest 特定端口）
├─ DataSourcePort（引用 PlanMetadata）
└─ ExecutionContext

patra-spring-boot-starter-provenance（Framework 层）
└─ DataSourceProvider（引用 PlanMetadata）
```

**依赖方向验证**：
- ✅ patra-ingest-domain → patra-common-model（允许）
- ✅ patra-spring-boot-starter-provenance → patra-common-model（允许）
- ✅ 两者都依赖通用模型，无循环依赖

#### 决策 8：为什么 Framework 层使用 TypeReference？

**理由**：
- 保持泛型信息：`Class<T>` 会丢失泛型信息，`TypeReference<T>` 可以保留
- 统一接口签名：Domain 层和 Framework 层使用相同的类型参数
- 消除类型转换：避免 Infrastructure 层的 `@SuppressWarnings("unchecked")`
- 类型安全：编译时检查，避免运行时 ClassCastException

**对比**：
```java
// ❌ 不推荐：使用 Class<T>
<T> ProviderResult<T> fetchData(..., Class<T> targetClass);
// 问题：Infrastructure 层需要类型转换
@SuppressWarnings("unchecked")
Class<T> targetClass = (Class<T>) typeRef.getRawType();

// ✅ 推荐：使用 TypeReference<T>
<T> ProviderResult<T> fetchData(..., TypeReference<T> typeRef);
// 优点：类型一致，无需转换
```

**TypeReference 说明**：
- 来自 `patra-common-type` 模块
- 使用技巧：通过匿名子类保留泛型信息
- 示例：`new TypeReference<List<String>>() {}`

---

#### 决策 9：消除硬编码类型列表（架构师评审改进 v4.1）

**问题背景**：
v4.0 版本中，`UnifiedBatchPlanner.findSupportedPlanType()` 方法硬编码了已知的 PlanMetadata 子类列表，违反了开闭原则。

```java
// ❌ v4.0 版本：硬编码类型列表
private Class<? extends PlanMetadata> findSupportedPlanType(BatchGenerationStrategy strategy) {
    List<Class<? extends PlanMetadata>> knownTypes = List.of(
            PubmedPlanMetadata.class,  // 硬编码
            EpmcPlanMetadata.class,    // 硬编码
            DoajPlanMetadata.class     // 硬编码
    );
    // 每次新增数据源都要修改这里
}
```

**三位架构师评审意见**：
- **架构师 A**："策略类型推断需要硬编码已知类型列表，建议使用注解或在接口中添加方法"
- **架构师 B**："硬编码问题违反 OCP，应通过注解或配置发现类型"
- **架构师 C**："致命问题（扩展性阻塞）- 10+ 数据源时将成为严重的维护负担"

**解决方案**：
在 `BatchGenerationStrategy` 接口中添加 `getSupportedType()` 方法，让策略类自己声明支持的类型。

```java
// ✅ v4.1 改进：策略接口新增方法
public interface BatchGenerationStrategy {
    /**
     * 获取策略支持的计划元数据类型
     *
     * @return 支持的 PlanMetadata 类型
     */
    Class<? extends PlanMetadata> getSupportedType();

    List<Batch> generateBatches(PlanMetadata plan, ExecutionContext ctx);
}

// ✅ v4.1 改进：策略实现
@Component
public class PubmedBatchGenerationStrategy implements BatchGenerationStrategy {
    @Override
    public Class<? extends PlanMetadata> getSupportedType() {
        return PubmedPlanMetadata.class;  // 策略自己声明类型
    }
}

// ✅ v4.1 改进：UnifiedBatchPlanner 自动发现
private Map<Class<? extends PlanMetadata>, BatchGenerationStrategy> buildStrategyMap(
        List<BatchGenerationStrategy> strategies) {
    Map<Class<? extends PlanMetadata>, BatchGenerationStrategy> map = new HashMap<>();

    for (BatchGenerationStrategy strategy : strategies) {
        Class<? extends PlanMetadata> supportedType = strategy.getSupportedType();
        map.put(supportedType, strategy);  // 无需硬编码
    }

    return map;
}
```

**收益分析**：
1. **完全符合 OCP**：新增数据源零修改
2. **消除维护负担**：无需维护类型列表
3. **类型安全**：编译时检查类型匹配
4. **自动发现**：Spring 启动时自动构建策略 Map
5. **防御性检查**：可以检测重复策略、null 类型等异常情况

**对比表**：

| 维度 | v4.0（硬编码） | v4.1（getSupportedType） |
|------|---------------|------------------------|
| **OCP 合规** | ❌ 违反 | ✅ 完全符合 |
| **新增数据源** | ❌ 必须修改 UnifiedBatchPlanner | ✅ 零修改 |
| **维护成本** | ❌ 高（易遗漏） | ✅ 低（自动发现） |
| **类型安全** | ⚠️ 运行时检查 | ✅ 编译时检查 |
| **10+ 数据源** | ❌ 列表过长 | ✅ 不受影响 |

**实施要点**：
- 在策略接口中添加 `getSupportedType()` 方法
- 所有策略实现类都必须实现此方法
- `UnifiedBatchPlanner` 启动时自动构建策略 Map
- 添加防御性检查（null 检查、重复检测）

### 6.2 权衡分析

| 选择 | 优点 | 缺点 | 决策 |
|------|------|------|------|
| 统一端口 | 高内聚、简化依赖 | 接口稍微复杂 | ✅ 采纳 - 优点远大于缺点 |
| 保留适配器 | 协议转换、错误处理 | 增加一层调用 | ✅ 采纳 - 提供实质价值 |
| 使用继承 | 类型安全、IDE 友好 | 需要定义子类 | ✅ 采纳 - 长期可维护性 |
| 直接用框架 | 简化代码、减少层次 | Infrastructure 依赖框架 | ✅ 采纳 - 框架稳定 |

### 6.3 架构原则验证

#### ✅ 符合六边形架构

- Domain 层定义端口（DataSourcePort）
- Infrastructure 层实现端口（DataSourceAdapter）
- 依赖方向正确（向内流动）
- Domain 层保持纯净（无框架依赖）

#### ✅ 符合 SOLID 原则

- **SRP**：DataSourcePort 负责完整的"数据源访问"能力
- **OCP**：通过继承扩展 PlanMetadata，无需修改接口
- **LSP**：所有 PlanMetadata 子类可替换基类
- **ISP**：接口职责清晰，不冗余
- **DIP**：Application 层依赖抽象，不依赖实现

#### ✅ 符合 DDD 原则

- 端口定义在 Domain 层
- 使用领域语言（PlanMetadata、DataSourcePort）
- 适配器处理技术细节
