Claude 配置 workspace 上移 v0.1

📅 2026-05-21 🌿 main 🎯 spec draft
关键决策摘要

把 patra-api/.claude/ 下 4 项 workspace 通用配置（patra plugin marketplace、plugin 启用声明、send-stop-notification hook、create-issue skill）上移到 Patra/.claude/；另把 Patra 的一句产品定义补到 Patra/CLAUDE.md。BE 专属内容（Java/Gradle hooks、六边形 rules、layers/testing/tech、patra-* 领域 skill、test-checker agent、project-info.md 实现细节）全部留在 patra-api。不引入 Patra/.claude/rules/ 目录（YAGNI）。Stop hook 借助 settings.json 叠加机制无缝衔接，不修改任何脚本逻辑。同时取消原本设想的「FE 适配」工作 —— 基于 LLM 自动跨技术栈映射能力，FE 与 BE 共用同一份 patra plugin 即可，顺手删除 portal/CLAUDE.md 的 FE 警示段（见 §7 决策 6）。

1. 背景与现状

上一次「monorepo Claude 配置三层化改造」（PR #37，commit 14134d4de）确立了 workspace 共享层 / BE 子项目层 / FE 子项目层的分层模型，并按 cwd 工作模式约定按需 cd。但当时为快速落地，把多项实际属于 workspace 的配置（patra plugin 实体与启用、跨语言的停止通知 hook、语言无关的 create-issue skill）暂留在 patra-api/.claude/ 下，commit message 也明确标注「.claude-marketplace/ 暂留 patra-api/（plugin 未适配 FE）」。

这次延续上次的方向：把真正的 workspace 共享配置抽到 Patra/.claude/，让 patra-api/.claude/ 只保留 BE 专属。诉求来自用户：「主要的配置还都是在 patra-api 下面，需要拆出整个项目相关的，然后迁移或改造到 Patra 下面」。

5. settings.json 拆分

Patra/.claude/settings.json（新增 hooks 段 + 扩充 enabledPlugins）：

{
  "hooks": {
    "Stop": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "$CLAUDE_PROJECT_DIR/.claude/hooks/send-stop-notification.sh"
          }
        ]
      }
    ]
  },
  "enabledPlugins": {
    "linear@claude-plugins-official": true,
    "context7@claude-plugins-official": true,
    "frontend-design@claude-plugins-official": true,
    "codex@openai-codex": true,
    "patra@patra-local": true
  }
}

patra-api/.claude/settings.json（移除 send-stop-notification 与 enabledPlugins，PostToolUse 与剩余 Stop 不变）：

{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write|MultiEdit",
        "hooks": [
          {"type": "command", "command": "$CLAUDE_PROJECT_DIR/.claude/hooks/mark-java-changes.sh"}
        ]
      }
    ],
    "Stop": [
      {
        "hooks": [
          {"type": "command", "command": "$CLAUDE_PROJECT_DIR/.claude/hooks/gradle-compile-check.sh"},
          {"type": "command", "command": "$CLAUDE_PROJECT_DIR/.claude/hooks/trigger-build-resolver-java.sh"}
        ]
      }
    ]
  }
}

叠加语义：当 cwd 在 patra-api 时，Claude Code 同时加载 Patra/.claude/settings.json 与 patra-api/.claude/settings.json。Stop 事件会依次触发三个 hook：workspace 层的 send-stop-notification.sh + patra-api 层的 gradle-compile-check.sh + trigger-build-resolver-java.sh。$CLAUDE_PROJECT_DIR 在所有 hook 里都解析为 cwd（即 patra-api/），所以 send-stop-notification.sh 的 basename "$PROJECT_ROOT" 仍输出「patra-api」—— 行为零变化。

7. 关键决策与备选

决策 1：patra plugin 仅在 workspace 启用，FE 不主动调用

把 patra@patra-local 从 patra-api 提到 workspace 启用。无论 cwd 是 patra-api 还是 patra-portal，plugin 都被加载，但 patra-portal/CLAUDE.md 仍保留「不应调用 patra: 流程 skill」的现状声明（plugin 内的 brainstorming/TDD 等示例仍是 Java 语境）。未来 FE 适配完成只需删除那段警示。

备选 A：BE/FE 同时启用并立刻删 FE 警示 — 否决：plugin 内 Java 示例未改，会误导 FE 工作流。
备选 B：plugin 保持在 patra-api 不上移 — 否决：违背「拆 workspace 通用配置」的初衷，FE/workspace 视角无法引用同一份 plugin 实体。

决策 2：project-info.md 整份留在 patra-api

初判时考虑过整文件上移到 Patra/.claude/rules/project-info.md。审查后发现文件 90% 是 BE 实现细节（技术栈 Java/Spring Boot/JPA/Gradle、4 个微服务清单、六边形 6 层模块结构、11 个 starter、build-logic Convention Plugins），只有顶部一句业务定义属于 workspace。整份上移会污染 workspace 视图；只上移一句话又不值得建独立文件。

备选 A：整份上移 — 否决：BE 工程细节出现在 workspace 视图里，与 patra-portal cwd 语义错位。
备选 B：拆成两份（workspace 一份业务概览 + BE 一份工程细节） — 否决：维护双份易漂移，且业务概览仅一句话不值得独立文件。

决策 3：不引入 Patra/.claude/rules/ 目录

沿 YAGNI 原则。本次 workspace 层没有需要单独 md 承载的规则；唯一的产品定义直接合并进 Patra/CLAUDE.md 顶部。未来真有 workspace 级规则需求再建目录。

备选 A：预建 Patra/.claude/rules/ 作为占位目录 — 否决：空目录无意义，未来需要时再建零成本。

决策 4：create-issue skill 临时路径改名 /tmp/patra-api → /tmp/patra

skill 上移到 workspace 后，SKILL.md 中 3 处 /tmp/patra-api/ 引用同步改为 /tmp/patra/。语义中性化，避免 portal cwd 调用此 skill 时临时文件落到带 BE 命名的目录。

备选 A：保留 /tmp/patra-api/ — 否决：路径与 skill 实际位置脱节，portal/workspace cwd 调用时语义错位。

决策 5：用 git mv 而非物理 mv + add

Patra/ 和 patra-api/ 同一 git repo，git mv 保留历史血缘，便于后续 git log/blame 追溯（patra plugin marketplace 内含大量 skill 文件，历史很有价值）。

备选 A：cp + rm + add — 否决：丢失 git 历史。

决策 6：取消 FE 适配工作，BE/FE 共用同一份 patra plugin

brainstorming 期间深入对比了上游 obra/superpowers（200k star，今日仍在更新）的语言策略：上游本身就是技术栈混合（TDD 用 TypeScript、debugging 用 shell + macOS 工具链），并不强绑后端。本地 patra fork 当前的「Java 化 + 中文化」是 fork 时的本地化决策。

原本设想给 FE 单独适配（双语示例 / cwd 动态分支 / 去技术栈化）。深入后发现：现代 Claude/Opus 已经能自动跨技术栈映射（看 Java assertThat() 能反向映射到 TS expect()，反之亦然），方法论部分（红绿重构 / 四阶段调试 / brainstorming 9 步 / code review 流程）100% 通用。FE 读 Java 示例仅有「轻度 cognitive friction」，不到「不能用」的程度。

结论：FE 适配不必做。本次 spec 顺手删除 patra-portal/CLAUDE.md 的「⚠️ Skill 加载现状」警示段，让 FE 与 BE 同等享用 plugin。如果未来 FE 工作中映射开销显著超预期，再单开「plugin 去技术栈化」PR（回归上游风格 + 保留中文翻译，patra-* 领域 skill 不动）。

备选 A：立刻做 FE 双语示例 — 否决：skill 长度膨胀 ~50%，且 LLM 自动映射已能承担同样作用，是「无效定制」。
备选 B：立刻做去技术栈化（回归上游 + 中文） — 否决：工作量大且本次 PR 应聚焦位置上移，不应耦合两类改造；如未来真有痛点再独立 PR 解决。
备选 C：保留 portal/CLAUDE.md 警示段不删 — 否决：警示存在的前提是「FE 调用会出问题」，与本决策得出的「能跨栈映射」结论矛盾。

8. 风险与缓解

风险等级缓解
Claude Code 是否会从 patra-api（cwd）上溯查找 Patra/.claude-marketplace/patra/？若不上溯，patra: 命名空间 skill 在 BE cwd 下会消失。 high 实施后第一步：在 patra-api cwd 启动新 Claude Code 会话，检查 /skill 列表里 patra:brainstorming 等是否仍可见。若不可见 → 回滚或在 patra-api/.claude-marketplace 留符号链接 ln -s ../../.claude-marketplace/patra patra。
Stop hook 叠加顺序：workspace 层与 BE 层的 Stop hooks 是否按预期都触发？ mid 实施后手动 trigger Stop 事件（结束一次会话），观察 macOS 通知是否弹出 + gradle 编译检查是否运行。Claude Code 官方文档说明 settings 叠加规则为「合并而非覆盖」，预期都触发。
create-issue skill 中 /tmp/patra-api/ 漏改 low 实施时用 grep -n patra-api SKILL.md 兜底校验 0 匹配。
patra-api/CLAUDE.md 中「流程层 patra:（plugin namespace，13 个方法论 skill）」描述上移后失准 low 命名空间与 skill 数量不随物理位置变化，描述仍准确。实施时复读一次确认。
marketplace 路径变动导致 settings.json 中 patra@patra-local 的「patra-local」名称不再有效 mid 「patra-local」是 marketplace 名（来自 marketplace 元数据中的 name 字段，而非物理路径），物理路径变化不影响命名。实施后通过 risk 1 的 skill 列表检查同时验证。
FE 实际使用 patra: 流程 skill 时，Java 示例造成的 cognitive friction 超预期，影响产出质量 low 用一段时间观察 FE 工作流（brainstorming / writing-plans / TDD 等）的实际产出；若痛点显著则独立开「plugin 去技术栈化」PR（回归上游风格 + 中文翻译），patra-* 领域 skill 不动。本次先不做。
