1. 背景与现状

本次 PR 已完成 monorepo Claude 配置三层化改造。改造过程中产出了一份 spec（2026-05-19-claude-config-refactor-design.html）和一份 plan（2026-05-19-claude-config-refactor.html）。用户审阅这两份文档后，识别出当前 skill 的几个问题：

问题根因
样式不统一 html-output-guide.md 明文 "风格自由：根据主题适配，不强制全 plugin 一致 CSS"——AI 每次重新设计 CSS，~200 行 inline 内容跨文档可能漂移
费 token / 慢 AI 每次从零写 CSS、布局、组件样式——稳定的视觉表达本应一次定型
无偏离日志机制 实施期 AI 遇到 spec/plan 未规定的决策只能写到 commit message 或散落在代码注释——缺乏集中、结构化的"偏离台账"
章节顺序不稳定 guide 只列"必须包含的语义元素"，未定顺序——AI 每次自行安排

3. 三栏布局设计

spec 和 plan 都采用同一三栏布局：

TOC220 px
sticky
内容min 0, 1fr
章节正文
Notes320 px
sticky
scroll-y
  • CSS Grid: grid-template-columns: 220px minmax(0, 1fr) 320px; gap: 24px;
  • 左右栏 sticky: position: sticky; top: 24px;——滚动中栏时不动
  • 响应式: @media (max-width: 1024px) 折叠为单栏——TOC 移到顶部 <details>，右栏移到底部
  • 右栏 max-height: max-height: calc(100vh - 48px); overflow-y: auto;——note 多时内部滚动不影响中栏

5. 视觉设计原则（委托给 frontend-design skill）

设计权委托

本次改造不在 spec 里硬编码具体颜色 / 字体 / 组件样式。模板的视觉风格由 /frontend-design:frontend-design skill 在实施期产出，确保符合下方 3 个原则。这避免我（设计能力存疑的工程师）拍脑袋定的样式被固化进模板。

5.1 三大视觉原则

原则含义
简约 视觉元素克制——必要的层次差异、不滥用色彩；优先字体粗细 / 间距区分而非颜色堆砌
易读 长文档阅读舒适——合适行高 / 段距 / 字号 / 对比度；代码块清晰；表格扫读快
易改 CSS 变量集中在 :root，调一处就改全局；命名语义化（不出现 --c-1 这种位置依赖名）；新增组件少量代码即可

5.2 结构性约束（spec 强制定义，frontend-design 不可违反）

  • 支持三栏布局（§3 定义的 220 / fit / 320 grid + sticky + 响应式断点 1024px）
  • 4 类 note pill 必须视觉上可区分（DECISION / CHANGE / TRADEOFF / OTHER），具体配色由 frontend-design 决定
  • task card 的 data-status 状态机（pending / in-progress / done / blocked / skipped）必须有显著视觉区分
  • HTML 单文件自包含——CSS / JS inline，禁止外部依赖（字体可引 Google Fonts）
  • spec 与 plan 两份模板使用同一套 CSS 变量定义（手工同步保持一致，CSS 变量名约定一致）

5.3 实施流程

  1. plan 第一个 task：调用 /frontend-design:frontend-design skill，输入 3 大原则 + 5 项结构性约束 + 章节骨架（spec 7 / plan 5），让它产出 base CSS
  2. 把 frontend-design 产出的 CSS 嵌入 spec-template.html 和 plan-template.html
  3. 两份模板的 CSS 内容保持完全一致（除模板专属选择器外，如 .task / .step 只在 plan 模板）

7. 关键决策与备选

决策 1：完全固定 CSS + 布局（vs 中等刚性 / 默认+override）

模板严格化路线：CSS、配色、字体、布局全部固定，AI 只填占位内容。

备选 B：仅固定骨架，CSS 可适配主题 — 否决：仍有风格漂移空间。
备选 C：默认固定 + 允许主题 override — 否决：违反 YAGNI（单人项目无多主题需求），且增加 reviewer 检查负担。

决策 2：spec / plan 右栏各自独立（vs 仅 plan / 合并独立文件）

spec.html 自己的右栏记 writing-plans 阶段发现的 spec 不足；plan.html 自己的右栏记 executing-plans 阶段发现的 plan 不足。各管各的，对应正确层级。

备选 B：只 plan 加右栏 — 否决：丢掉 spec 阶段在 plan 中被质疑的信息。
备选 C：合并为独立 deviation-log.html — 否决：额外文件、引用复杂、跨文档维护成本高。

决策 3：4 类 note 显式区分（vs 不分类）

每条 entry 一个 pill 标签（DECISION/CHANGE/TRADEOFF/OTHER）+ 文档末 stats bar 汇总计数。

备选 B：不分类 — 否决：失去快速打分能力（用户扫一眼就想知道"有几个 decision"）。

决策 4：模板放 template/ 子目录（vs 同级平铺）

每个 skill 加 template/ 子目录。理由："规范一些"（用户原话），未来可能加多个模板（如 ADR template）。

备选：同 SKILL.md 平铺 — 否决：扩展性差，且 brainstorming 已有 scripts/ 子目录先例。

决策 5：本 PR 不改 executing-plans / subagent-driven-development

追加 note 的强制规则只在模板 hint 文字 + reviewer prompt 检查项里约定，对应 skill 的硬编码强制规则留到后续 PR。

备选：本 PR 一并改 — 否决：PR 范围已经较大（10+ 文件），再加 2 个 skill 改造增加 review 负担。模板 hint 文字对当前阶段已足够。

决策 6：视觉风格委托给 frontend-design skill（vs 我直接指定）

不在 spec 里硬编码 CSS 配色 / 字体 / 具体组件样式。模板视觉由 /frontend-design:frontend-design skill 产出，spec 只规定 3 大原则（简约 / 易读 / 易改）和 5 项结构性约束。

备选 A：我直接在 spec 里指定 indigo + Inter 全套（即 v0.1 写法）— 否决：我（工程师）的视觉判断不可靠，固化进 spec 会拖累后续调优。
备选 B：不调用 frontend-design，让 AI 在写模板时自由设计 — 否决：又回到"自由发挥"老路，无法保证质量。

8. 风险与缓解

风险等级缓解
模板过于严格限制未来主题适配 mid CSS 用变量集中在 :root，调一处即可全局改色 / 间距 / 圆角
AI 实施期忘记追加 note（因为 executing-plans skill 这次没硬编码强制规则） mid 模板右栏 hint 文字明确约定 + reviewer prompt 加检查项；后续 PR 在 skill 加硬规则
已生成的 2 份 spec/plan 不符合新模板 low 历史快照不动；新模板生效后才约束新文档
三栏在窄屏（< 1024px）需折叠，可能影响阅读 low 响应式断点显式定义；移动端 TOC 改 <details>，右栏移到底部
