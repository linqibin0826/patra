# 日志规范（Logging Convention）

> 状态：草案（Draft） — 适用于 Papertrace 全平台所有微服务与组件。模块级说明（若有）不得与本规范冲突；若需扩展，需在对应模块文档中引用本规范并仅新增，不修改通用约束。

## 1. 目标与设计理念
- 可检索：统一结构化前缀 + 语义稳定字段，方便 grep / Loki / ES 聚合
- 可观测：关键生命周期、状态转移、错误链路有且仅有一条清晰日志
- 低噪声：热点高频路径仅 DEBUG；INFO 留给低频“完成态”与运维关键节点
- 成本敏感：避免字符串拼接浪费；使用参数化占位符；不序列化巨大对象
- 可演进：前缀体系可增量扩展，不破坏既有查询面板

## 2. 日志级别使用准则
| 级别 | 适用场景 | 必要性判断 | 典型示例 |
|------|-----------|-----------|-----------|
| ERROR | 业务不可恢复错误 / 数据不一致风险 / 外部依赖调用失败需要介入 | 是否需要报警或人工确认 | 外部服务超时、持久化失败、幂等被破坏 |
| WARN  | 可自愈或被忽略的异常路径 / 潜在风险 / 数据缺失 | 是否需要排查但不必立刻报警 | 重试即将耗尽、发现多默认项、降级生效 |
| INFO  | 低频、重要的状态完成/关键里程碑 | 是否有助于审计或 SLA 追踪 | 作业调度落地、配置刷新完成、批次闭合 |
| DEBUG | 高频内部状态转移 / 诊断粒度 | 是否用于调试且可通过日志级别动态开启 | Upsert 分支、发布开始/成功、租约竞争 |
| TRACE | 暂不启用（默认关闭） | 需要代码级跟踪时临时打开 | N/A |

> 禁止在业务正常分支打印 ERROR；异常捕获后转换为业务语义（如校验失败）应降级为 WARN/INFO/DEBUG。

## 3. 统一前缀格式（简化后）
为降低日志前缀长度、提升可读性，从 v0.2.0 起统一前缀仅保留两段：
```
[模块][层]
```
说明：
- 模块（原“平台”）：微服务或可执行子系统名称，例如：`INGEST`、`REGISTRY`、`GATEWAY`、`COMMON`。
- 层（Layer）：对应六边形/分层位置：`ADAPTER` | `APP` | `DOMAIN` | `INFRA` ｜ `API`.


命名建议（非强制）：
1. 仅保证前缀格式 `[模块][层]`；其余正文用简短自然语义描述
2. 无需固定的 `domain/feature/stage`、键值顺序或数量限制；能读懂即可
3. 只有在后续定位确实需要时才在正文末尾补充少量信息（例如 taskId=123）；不做强制结构化
4. 不为了“规整”而填充占位键值；保持最小可读
5. 失败时可在正文附上错误码/摘要（如 MQ_TIMEOUT / conflict），长堆栈仍交由日志系统折叠

## 4. 参数与上下文
- 禁止打印：密码、令牌、个人隐私、密钥（若必须出现，截断：前 6 + '***'）
- 大对象（>2KB）禁止直接 `toString()`；如需输出仅保留摘要：`payloadSize=... payloadHash=...`
- 失败日志补充 `errCode` / `errMsg`（截断/摘要）避免刷屏；长堆栈由日志聚合工具折叠查看

## 5. 频率控制策略（参考）
| 场景 | 策略 |
|------|------|
| 高频循环（批量处理内） | 外层聚合统计 INFO，一条任务粒度 DEBUG（可选采样） |
| 重试 / 退避 | 仅首次与最终 FAIL 打 ERROR，中间 WARN + 递增计数 |
| 竞争锁/租约 | 成功 ACQUIRE 打 DEBUG；失败不刷屏（可按周期采样） |
| 轮询空结果 | 不打印或每 N 次打印 1 条 DEBUG（N >= 100） |

## 6. 结构化与 MDC（参考）
可使用 logback pattern 注入：`traceId`, `spanId`, `corrId`（或团队自定义）。
- 入口层可放置拦截器统一初始化 `traceId`
- 业务链路 ID（`planId` / `taskId` 等）仅在确有跨日志关联需要时添加

## 7. 错误与异常处理（参考）
- 可恢复 / 业务校验类结果无需打印 ERROR
- 需要堆栈时保留一次即可，避免同一异常多点重复刷屏
- 外部重试机制（调度 / MQ）已有统一策略时可减少人工循环日志

## 8. 审计与安全（参考）
- 对用户变更、权限调整等可视情况独立出 `AUDIT` 类 logger（如确有审计需求）
- 生产环境 TRACE 通常关闭；DEBUG 按需临时启用

## 9. 引入新日志的自查（可选）
在提交前可快速自问（不满足也不强制修改，仅作质量提醒）：
1. 是否复用了既有 `[模块][层]` 而不是新增别名？
2. 高频路径是否避免了不必要的 INFO/ERROR？
3. 是否无敏感数据泄露（或已脱敏）？
4. 如果后续需要串联链路，是否已包含关键 ID（taskId / planId 等）？

## 10. 最小实践示例（参考）
只保留两段前缀 + 自然语义正文：
```
[INGEST][APP] Task publish started (taskId=123)
[INGEST][APP] Task publish success (taskId=123 elapsed=45ms)
```
失败示例：
```
[INGEST][APP] Task publish fail (taskId=123 err=MQ_TIMEOUT sendTimeout)
```
说明：括号内信息按需增减；若无需检索可直接写自然语句，例如：
```
[INGEST][APP] Task publish started
```

## 11. 与指标/追踪的边界（参考）
- 日志：离散事件（qualitative）
- 指标：数值聚合（quantitative） → QPS、耗时、失败率
- 追踪：跨服务调用链 → 延迟拆解
三者互补，不重复：不要同时在同一循环内打印每条记录的耗时（应用 timer metric）
