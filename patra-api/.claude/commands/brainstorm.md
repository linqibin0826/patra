### 阶段 1：代码探索（Code Archaeology）- 🔴 REQUIRED

**目标**：系统化搜索相关代码，建立事实基础。

#### 搜索策略（根据话题类型选择）

**如果是组件设计话题**（例如："设计标签管理组件"）：
1. ✓ 搜索相关领域概念的现有实现
   - 工具：`search_for_pattern`
   - 关键词：提取话题中的核心概念（如 tag, label, category）
   - 范围：domain 层的所有包

2. ✓ 查找相关模块的聚合和实体
   - 工具：`get_symbols_overview` + `find_symbol`
   - 路径：`patra-{module}/patra-{module}-domain/src/main/java/`
   - 重点：Aggregate、Entity、Value Object、Domain Event

3. ✓ 查找类似的 Orchestrator
   - 工具：`find_symbol` with `substring_matching: true`
   - 模式：`*Orchestrator`

4. ✓ 查找相关的 Repository 实现
   - 工具：`search_for_pattern`
   - 模式：`interface *Repository`, `class *RepositoryImpl`

5. ✓ 检查 patra-common 中的可复用组件
   - 路径：`patra-common/src/main/java/`

**如果是功能实现话题**（例如："批量导入文献"）：
1. ✓ 搜索现有的批量处理逻辑
   - 关键词：`Batch`, `Import`, `Bulk`, `Job`, `Scheduled`

2. ✓ 查找 Adapter 层的实现
   - 路径：`patra-{module}/patra-{module}-adapter/`
   - 重点：Controllers, Jobs, Listeners

3. ✓ 查看错误处理模式
   - 搜索：`exception` 包，`@ExceptionHandler`

4. ✓ 查找事务管理模式
   - 搜索：`@Transactional`

**如果是技术方案话题**（例如："如何实现版本控制"）：
1. ✓ 搜索现有的类似技术实现
   - 关键词：相关技术概念（version, history, audit, snapshot）

2. ✓ 查看已有的技术依赖
   - 工具：Read `pom.xml` 文件

3. ✓ 查找相关的 DO（Data Object）结构
   - 路径：infra 层的 `dataobject` 包

4. ✓ 检查现有的 starter 配置
   - 路径：`patra-spring-boot-starter-*/`

**如果是架构决策话题**（例如："服务间依赖关系"）：
1. ✓ 查看模块间的依赖关系
   - 工具：Read 各模块的 `pom.xml`

2. ✓ 查找 API 定义
   - 路径：`patra-{module}/patra-{module}-api/`

3. ✓ 查看 Port 接口定义
   - 搜索：domain 层的 `port` 包

4. ✓ 检查跨模块的调用模式
   - 工具：`find_referencing_symbols`

### 阶段 2：深度分析（Deep Analysis）- 🔴 REQUIRED

**使用 `sequential-thinking` 工具进行系统化推理**。

基于阶段 1 找到的**实际代码**，分析以下方面：

#### 分析框架

**Step 1: 现有实现评估**
- 现有代码的优点是什么？
- 存在哪些不足或技术债？
- 为什么要这样设计？背后的考量是什么？

**Step 2: 架构契合度验证**
- 现有代码是否遵循了 Hexagonal Architecture？
- 是否违反了依赖方向？
- Domain 层是否保持纯净？

**Step 3: 可复用模式提取**
- 哪些模式可以直接复用？
- 需要做哪些调整？
- 有哪些反模式需要避免？

**Step 4: 新设计方向**
- 基于现有代码，新设计应该如何融入？
- 需要新增哪些组件？
- 需要修改哪些现有代码？

**Step 5: 风险和权衡**
- 技术风险：性能、安全、扩展性
- 实现成本 vs 维护成本
- 短期收益 vs 长期影响

**Step 6: 多方案对比**（如果适用）
- 方案 A：...（优缺点、适用场景）
- 方案 B：...（优缺点、适用场景）
- 推荐方案及理由

### 阶段 3：协作讨论（Collaborative Discussion）

基于前两个阶段的**事实基础**，开始苏格拉底式的引导讨论。

#### 讨论原则

**✅ 应该这样做**：
- 引用具体代码："`ProvenanceOrchestrator` (patra-registry-app/.../ProvenanceOrchestrator.java:45) 使用了..."
- 对比现有实现："在现有的批量处理中，`BatchPlanOrchestrator` 采用了...，我们可以..."
- 提出引导性问题："我注意到现有代码使用了 Strategy 模式，你觉得新组件是否也需要？"
- 挑战假设："你提到需要缓存，但从 `LiteratureRepositoryImpl` 的实现看，现有的查询性能如何？"
- 渐进式深入："让我们先确定这个组件属于哪个层，然后再讨论实现细节"

**❌ 避免这样做**：
- 凭空臆测："通常这种情况会用..."（除非有项目代码支持）
- 引用不存在的代码："可以参考 XxxService..."（如果代码库中没有）
- 过度抽象："根据 DDD 原则..."（必须结合具体代码说明）

#### 互动模式

根据话题类型，使用 `AskUserQuestion` 工具提出 2-4 个关键问题：

**组件设计话题**：
- 边界和职责："基于现有的 [具体聚合名称]，这个组件的核心职责是什么？"
- 层次归属："它应该放在哪个层？为什么？"
- 依赖关系："它需要依赖哪些现有组件？"

**功能实现话题**：
- 核心场景："MVP 应该包含哪些功能？"
- 复用策略："现有的 [具体组件] 可以复用吗？"
- 错误处理："参考 [具体实现]，错误处理策略是什么？"

**技术方案话题**：
- 方案选择："我找到了 A、B 两种现有实现，你更倾向哪种？"
- 约束确认："技术栈、性能、成本方面有什么约束？"

**架构决策话题**：
- 影响评估："这个改动会影响哪些模块？"
- 迁移路径："如何从现状迁移到目标状态？"
