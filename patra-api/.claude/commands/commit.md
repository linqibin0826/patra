## Git 变更上下文

**当前分支：**
!`git branch --show-current`

**仓库状态：**
!`git status --short`

**未暂存的变更详情：**
!`git diff --stat`

**已暂存的变更详情：**
!`git diff --cached --stat`

**完整变更内容预览（未暂存 + 已暂存）：**
!`git diff HEAD`

---

## 任务目标

分析本地所有未提交的变更（包括未暂存和已暂存），智能拆分为多个逻辑相关的提交，并为每个提交生成规范的中文提交消息。

### 核心原则

1. **小步提交**：每个提交只包含一个逻辑变更单元，便于代码审查和版本回溯
2. **关联性分组**：相关的文件变更归为一个提交，不相关的拆分为多个提交
3. **规范消息**：遵循 Conventional Commits 规范，使用中文描述
4. **禁止推送**：🚨 **绝对禁止执行 `git push`**，只在本地创建提交

## Conventional Commits 规范

### 消息格式

```
<type>(<scope>): <简短描述>

<详细说明（可选）>

<footer（可选）>
```

### Type 类型（必须使用英文）

| Type | 说明 | 示例场景 |
|------|------|----------|
| `feat` | 新功能 | 添加新的 API 端点、新的领域服务 |
| `fix` | Bug 修复 | 修复空指针异常、逻辑错误 |
| `refactor` | 重构 | 优化代码结构、重命名、提取方法 |
| `docs` | 文档 | 更新 README、添加 JavaDoc |
| `test` | 测试 | 添加/修改测试用例 |
| `style` | 格式 | 代码格式化、缩进调整（不影响逻辑） |
| `chore` | 杂务 | 更新依赖、配置文件变更 |
| `perf` | 性能 | 性能优化 |
| `ci` | CI/CD | 流水线配置变更 |
| `build` | 构建 | Maven/Gradle 配置变更 |

### Scope 作用域（使用模块名或功能名）

根据本项目结构，推荐的 scope：
- 微服务模块：`ingest`、`registry`、`catalog`、`gateway`
- 架构层级：`domain`、`app`、`infra`、`adapter`、`api`
- Starter 模块：`starter-web`、`starter-jpa`、`starter-rest-client` 等
- Common 模块：`common-core`、`common-model`
- 功能领域：`auth`、`batch`、`storage`、`provenance` 等

### 消息编写规范

1. **简短描述**（第一行）：
   - 使用中文
   - 不超过 50 个字符
   - 使用动词开头：添加、修复、优化、重构、更新、移除
   - 不加句号

2. **详细说明**（正文）：
   - 使用中文
   - 解释「为什么」做这个变更，而非「做了什么」
   - 每行不超过 72 个字符
   - 与简短描述之间空一行

3. **Footer**（可选）：
   - `BREAKING CHANGE:` 破坏性变更说明
   - `Closes: #issue` 关联 Issue
   - `Refs: #pr` 关联 PR

### 示例

```
feat(ingest): 添加 PubMed 文献批量导入功能

实现基于 Spring Batch 的批量导入机制，支持断点续传。
采用分页拉取策略降低内存占用，每批次处理 1000 条记录。

Closes: #42
```

```
fix(starter-jpa): 修复乐观锁版本号未正确递增的问题

批量更新时未触发版本递增，原因是 Hibernate 的批量操作
绕过了 @Version 检查。改用 EntityManager.merge() 策略解决此问题。
```

```
refactor(catalog): 将 JournalService 拆分为领域服务和命令处理器

遵循六边形架构原则，将业务逻辑从 Application 层下沉到 Domain 层。
JournalDomainService 负责核心业务规则，JournalHandler 负责命令处理。
```

## 执行流程

### 第一步：分析变更

1. 检查是否有未提交的变更（未暂存 + 已暂存）
2. 如果没有变更，输出提示并结束
3. 分析每个变更文件的：
   - 所属模块（从路径推断）
   - 变更类型（新增/修改/删除）
   - 变更内容的性质（功能/修复/重构/文档等）

### 第二步：智能分组

根据以下规则将变更文件分组为多个提交：

**同一提交的判断标准：**
- 同一功能特性的相关文件（如 Controller + Service + Repository）
- 同一 Bug 修复涉及的所有文件
- 同一重构操作的所有文件
- 同一模块内紧密关联的变更

**拆分为不同提交的判断标准：**
- 不同模块的独立变更
- 不同类型的变更（如功能开发与文档更新）
- 逻辑上独立的多个修复
- 测试代码与实现代码可以分开（如果逻辑独立）

**单个提交的文件数量建议：**
- 理想：1-5 个文件
- 可接受：6-10 个文件
- 超过 10 个文件应考虑是否可以进一步拆分

### 第三步：生成提交计划

输出提交计划供用户确认：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📋 提交计划
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

共检测到 N 个变更文件，建议拆分为 M 个提交：

📦 提交 1/M
   类型: feat(scope)
   描述: xxx
   文件:
   - path/to/file1.java (新增)
   - path/to/file2.java (修改)

📦 提交 2/M
   类型: fix(scope)
   描述: xxx
   文件:
   - path/to/file3.java (修改)

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
❓ 是否按此计划执行提交？(Y/n/调整)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### 第四步：执行提交

**用户确认后**，按顺序执行每个提交：

1. 先 `git reset HEAD` 取消所有暂存（如果有）
2. 对于每个提交：
   a. `git add <files>` 暂存该提交的文件
   b. `git commit -m "$(cat <<'EOF'\n<message>\nEOF\n)"` 创建提交
   c. 输出提交成功信息

### 第五步：输出结果

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
✅ 提交完成
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

成功创建 M 个提交：

1. abc1234 feat(ingest): 添加 PubMed 文献批量导入功能
2. def5678 fix(starter-jpa): 修复乐观锁版本号未正确递增的问题

📊 统计：
   - 新增文件: X
   - 修改文件: Y
   - 删除文件: Z

⚠️ 提示：提交仅在本地，如需推送请手动执行 git push
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## 特殊情况处理

### 情况 1：只有一个逻辑变更

直接生成单个提交，无需拆分。

### 情况 2：变更过于复杂难以自动分组

询问用户偏好：
```
检测到变更涉及多个模块且关联性不明确，请选择处理方式：
1. 合并为单个提交
2. 按模块拆分
3. 按变更类型拆分
4. 手动指定分组
```

### 情况 3：存在未追踪的新文件

提示用户确认是否需要包含这些文件：
```
检测到以下未追踪的新文件，是否加入提交？
- path/to/new-file.java
```

### 情况 4：存在冲突或异常

如果执行过程中出现错误：
1. 输出错误信息
2. 回滚已执行的操作（如果可能）
3. 提供修复建议

---

## 禁止行为

1. 🚨 **禁止执行 `git push`** — 任何形式的推送操作都被禁止
2. 禁止提交敏感文件（`.env`、`credentials.json` 等）
3. 禁止生成空提交
4. 禁止在用户未确认前执行提交
5. 禁止修改已推送到远程的提交历史
