# 📖 通用数据采集任务执行流程

> 完整描述从 MQ 消费到游标推进的通用执行流程
---

## 🎯 这个流程在做什么？

这是一个**通用的外部数据源采集任务执行引擎**。无论是从医学文献库（PubMed）、欧洲医学中心（EPMC）、学术交叉引用库（Crossref），还是其他任何数据源采集数据，都遵循这套统一的执行流程。

**核心目标**：
- **可靠采集**：确保数据不丢、不重、可追溯
- **断点续传**：任务中断后能从断点继续，不从头再来
- **进度追踪**：记录已采集到哪里（游标/水位线），下次从那里接着采
- **弹性容错**：部分失败不影响整体，支持重试和降级

---

## 🔄 完整流程（8个步骤）

### **阶段一：准备阶段（步骤 0-4）** - 确保任务能安全启动

#### **步骤 0：任务就绪通知**

**触发方式**：调度器（定时任务）或上游系统通过消息队列（MQ）发送任务就绪消息

**消息内容**：
- 任务 ID：这是哪个任务
- 幂等键：防止重复执行的唯一标识

**幂等检查**：先看任务是否已成功（状态 = `SUCCEEDED` 且幂等键匹配），如果是直接跳过

#### **步骤 1：争抢执行权（CAS 抢占租约）**

**问题场景**：多个工作节点（服务器）可能同时收到同一条消息（MQ 重复投递、多副本消费等）

**解决方案**：用数据库的 CAS（Compare-And-Set）机制"抢锁"
- 在任务表里写入租约信息：租约持有者 = `节点ID:随机标识`，租约到期时间 = `当前时间 + 60秒`
- 抢锁 SQL：`UPDATE task SET lease_owner=?, leased_until=? WHERE id=? AND (lease_owner IS NULL OR leased_until < NOW())`
- **抢到**：继续执行
- **没抢到**：优雅退出，不报错（别人在执行，我不重复干）

#### **步骤 2：建立执行会话（初始化上下文）**

**在一个数据库事务内完成**：
1. 把任务状态改为 `RUNNING`（正在执行）
2. 创建执行记录（`TaskRun`）：
   - 记录这是第几次尝试（attemptNo = 上次 attemptNo + 1）
   - 绑定执行窗口（时间范围）
   - 绑定追踪标识（correlationId、schedulerRunId，用于分布式追踪）
3. 启动**心跳续租机制**：
   - 每隔 20 秒（租约时长的 1/3）主动续租
   - 告诉系统"我还活着，继续锁定任务"
   - 如果续租失败（网络故障、节点崩溃），其他节点可以接管

#### **步骤 3：还原配置快照（加载执行参数）**

**为什么要快照？**：采集任务可能是几天前创建的，期间配置可能被修改。用快照确保"按创建时的配置执行"，保证幂等性

**加载内容**：
1. **数据源配置**（provenance_config_snapshot）：
   - API 基础地址（base_url）
   - HTTP 超时、重试、代理设置
   - 分页模式（PAGE_NUMBER / CURSOR / TOKEN）、每页大小
   - 批处理策略（批量拉取时每批多少条）
2. **查询表达式**（expr_snapshot）：
   - 搜索条件（关键词、时间范围、字段过滤等）
   - 表达式哈希值（用于校验完整性）

**哈希校验**：计算配置的哈希值，与计划时保存的哈希对比，防止配置被篡改

**心跳维护**：刷新心跳时间戳；如果这步耗时超过租约时长的 1/3，主动续租

#### **步骤 4：编译查询表达式**

把保存的表达式（JSON 格式）编译成**目标数据源能理解的查询参数**


### **阶段二：执行阶段（步骤 5-8）** - 实际采集与数据处理

#### **步骤 5：规划批次（Generate RunBatch Plan）**

这一步的目标是**把一个大任务拆成多个小批次**，方便并发、重试、进度追踪。

**拆分策略（根据数据源和操作类型选择）**：

| 数据源 | 拆分方式 | 说明 |
|--------|---------|------|
| **PubMed** | **分页拆分** | 先调用 `esearch` API 获取总数（比如 5000 条），服务端缓存查询结果（WebEnv）；根据配置的分页大小（比如 500 条/页）拆成 10 批 |
| **EPMC** | **游标拆分** | 调用初始查询获取第一页 + 游标（cursor token），每批用上一批返回的 cursor 继续拉取，直到 cursor 为空 |

**执行流程**：
1. 调用 `BatchPlannerRegistry.getPlanner(provenanceCode)` 获取对应数据源的规划器
2. 规划器调用初始 API（探测总量或获取游标）
3. 根据配置计算需要多少批：
   - 批次数 = min(总量 ÷ 每批大小, 最大批次限制)
   - 比如：5000 条数据，每批 500 条 → 10 批；但如果配置了 `maxBatchesPerExecution=5`，只生成 5 批
4. 批量插入批次记录到数据库（`ing_task_run_batch`）：
   - 每批都是 `PENDING` 状态
   - 每批有唯一的幂等键（比如 `SHA256(runId + ":" + batchNo)`）
   - 记录批次参数（起始位置、游标 token、时间范围等）

#### **步骤 6：逐批执行（Execute RunBatch）**

这是**核心采集逻辑**，按顺序或并发执行每个批次。

**执行流程（针对每一批）**：
1. **幂等检查**：这批是否已经 `SUCCEEDED`？是 → 跳过
2. **更新状态**：批次状态改为 `RUNNING`（开始执行）
3. **调用数据源 API**：
   - PubMed：`efetch` API + WebEnv + 分页参数（retstart/retmax）
   - EPMC：`search` API + cursor token
4. **解析响应**：把 XML / JSON 响应解析成标准化的领域对象（比如 `Article` 实体）
5. **数据压缩**：用 gzip 压缩原始数据（节省存储空间，压缩比通常 5-10 倍）
6. **上传对象存储（先暂未，后续统一接入）**：
   - 存储路径：`{bucket}/{provenanceCode}/{year}/{month}/{runId}/batch_{batchNo}.json.gz`
   - 示例：`papertrace-ingest/pubmed/2024/01/run_12345/batch_003.json.gz`
7. **写入 Outbox（事务保证）**：
   - 在**同一个事务**内：
     - 插入 Outbox 消息：`{channel: "ingest.data.pubmed", payload: {storagePath: "...", recordCount: 500}}`
     - 更新批次状态为 `SUCCEEDED`，记录统计信息（记录数、文件大小、最大时间戳等）
   - 事务保证：要么数据和消息都写入成功，要么都失败回滚
8. **异常处理**：
   - API 调用失败：标记批次为 `FAILED`，记录错误信息，**继续执行下一批**（部分失败容忍）
   - MinIO 上传失败：重试 3 次（指数退避），仍失败则标记 `FAILED`
   - 数据库异常：抛出异常，任务整体失败，等待 MQ 重试

#### **步骤 7：推进游标（Advance Cursor）**

**游标的作用**：记录"已经采集到哪里了"，下次从那里接着采，避免漏数据或重复采集。

**推进时机**：整个任务的所有批次执行完成后（不是每批完成就推进，避免部分失败污染游标）

**推进策略（根据操作类型选择）**：

| 操作类型 | 游标类型 | 推进逻辑 | 示例 |
|---------|---------|---------|------|
| **HARVEST**<br/>（增量采集） | **时间型** | 从所有成功批次的统计信息（stats）中提取最大的时间戳 | 最新发表时间 `2024-01-15 23:59:59` |
| **HARVEST**<br/>（增量采集） | **ID 型** | 从所有成功批次中提取最大的记录 ID | 最大 PubMed ID `39876543` |
| **BACKFILL**<br/>（回填补采） | **窗口型** | 记录已完成的时间窗口范围 | 已补采 `2023-12-01 ~ 2023-12-31` |
| **UPDATE**<br/>（更新已有数据） | **可能不推进** | 根据业务决定是否更新游标 | 重新采集指定 ID 列表，不改变游标 |

**执行流程**：
1. 判断是否需要推进（根据 `operation_type`）
2. 从所有 `SUCCEEDED` 批次的 `stats` 字段聚合：
   - 提取每批的最大值（时间戳 / ID）
   - 计算全局最大值
3. 更新游标表（`ing_cursor`）：
   - 使用**乐观锁**（version 字段）更新当前水位
   - 记录推进后的值、规范化时间（normalized_instant）
4. 插入游标事件（`ing_cursor_event`）：
   - 记录推进历史（从哪到哪、操作类型、关联的 run_id）
   - 带幂等键，防止重复推进

#### **步骤 8：更新执行状态（Finalize Execution）**

任务收尾，记录执行结果。

**统计聚合**：
- 总批次数、成功批次数、失败批次数
- 总采集记录数、总文件大小
- 执行耗时

**判断最终状态**：
- **全部成功** → `SUCCEEDED`
- **有失败** → `FAILED`（可能触发重试）
- **部分成功** → `PARTIAL`（部分数据已采集，游标已推进，失败部分需人工介入）

**更新数据库（在一个事务内）**：
1. 更新执行记录（`ing_task_run`）：
   - 状态、统计信息（stats JSON）、完成时间、错误信息（如果失败）
2. 更新任务本身（`ing_task`）：
   - 状态、完成时间
   - 清空租约（`lease_owner = NULL`, `leased_until = NULL`）
   - 如果失败，更新重试计数和错误码

**停止心跳**：取消心跳定时任务，释放资源

## 🛡️ 通用容错机制

### **1. 多层幂等保障**
- **任务级别**：幂等键（idempotentKey）唯一约束，已成功的任务不会重复执行
- **批次级别**：每批有独立的幂等键，支持断点续传（已成功的批次自动跳过）
- **游标级别**：游标事件有幂等键，防止重复推进水位线
- **消息级别**：Outbox 的 (channel, dedup_key) 唯一约束，防止重复发送

### **2. 分布式租约机制**
- **CAS 抢占**：防止多节点并发执行同一任务
- **心跳续租**：执行过程中定期续租，维持租约有效性
- **租约超时接管**：节点崩溃时，租约到期后其他节点可以接管任务

### **3. 部分失败容忍**
- 批次级别隔离：某批失败不影响其他批次
- 最终状态判断：根据成功率判断任务状态（SUCCEEDED / FAILED / PARTIAL）
- 失败重试：
  - MQ 层重试（消费失败触发 binder 重试）
  - API 层重试（HTTP 客户端的重试策略，从 `retry_config` 读取）
  - 存储层重试（MinIO 上传 3 次指数退避）

### **4. 断点续传支持**
- 所有批次预先落库（PENDING 状态）
- 任务重启后从 PENDING 或 FAILED 批次继续
- 已成功批次通过幂等检查自动跳过

### **5. 可观测性**
- 完整的追踪链路：`correlationId`（消息级别）、`traceId`（调用链追踪）
- 详细的执行日志：每步操作都记录 taskId、runId、batchNo、owner 等关键信息
- 统计信息持久化：执行耗时、记录数、文件大小等存入 `stats` JSON 字段

## 📦 通用数据流转路径

### 1. 配置数据流
```
配置快照（Plan 冻结） → 解析为执行参数 → 传给 BatchPlanner/Executor
```

### 2. 表达式数据流
```
表达式快照（JSON） → 编译器 → 目标数据源的查询参数
```

### 3. 原始数据流
```
数据源 API 响应 → 解析器 → 领域对象 → 压缩 → MinIO → Outbox 轻量引用 → MQ 发布 → 下游消费
```

### 4. 游标数据流
```
批次统计信息（stats） → 聚合计算 → 全局最大值 → 更新水位线 → 记录推进事件
```

## 🎯 通用设计亮点

### 1. **策略模式（Strategy Pattern）**
- `BatchPlannerRegistry`：根据 `provenanceCode` 选择不同的批次规划器
- `CursorStrategy`：根据 `operationType` 选择不同的游标推进策略

### 2. **适配器模式（Adapter Pattern）**
- `ProvenanceClient` 抽象接口，不同数据源有不同实现（`PubMedClient`、`EpmcClient`、`CrossrefClient`）
- 屏蔽数据源差异，统一调用方式

### 3. **配置驱动**
- 所有数据源特性（分页模式、批处理策略、重试次数等）都从配置读取
- 新增数据源只需：①实现 `ProvenanceClient` ②实现 `BatchPlanner` ③配置注册

### 4. **存算分离**
- 原始数据存对象存储（MinIO），不占数据库空间
- Outbox 只存路径引用，保持轻量
- 为未来数据湖架构铺路

### 5. **事件溯源（Event Sourcing）**
- 游标推进事件（`ing_cursor_event`）记录完整历史
- 可追溯每次水位线的变化，支持审计和回溯

---
**核心思想**：通过配置驱动和策略模式，用一套流程适配多种数据源，保证可靠性、可追溯性和可扩展性 🚀
