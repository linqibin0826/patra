# Patra 本地提交

用于分析当前工作区改动、按逻辑拆分提交，并在获得用户明确授权后执行本地 `git commit`。

## Arguments

用户调用这个命令时传入的参数为：`$ARGUMENTS`

如果参数里包含额外的提交目标、scope 倾向或拆分说明，把这些信息作为分组建议的优先约束；如果参数为空，就完全基于当前工作区状态分析。

## Preflight

执行任何提交分析前，先做这些检查；任一关键检查失败时停止并给出可执行建议。

1. 确认当前目录是 Git 仓库：
   - 运行 `git rev-parse --is-inside-work-tree`
   - 如果失败，直接说明当前目录不是 Git 仓库，不能执行本命令
2. 读取当前分支与工作区状态：
   - `git branch --show-current`
   - `git status --short`
   - `git diff --stat`
   - `git diff --cached --stat`
3. 检查危险状态：
   - 如果存在冲突文件、rebase/merge 未完成、空工作区、空暂存区且无未暂存改动，需要先说明状态，再决定是否继续
   - 如果发现敏感文件、环境文件、证书、密钥、构建产物，明确提示默认不应提交
4. 明确边界：
   - 始终禁止 `git push`
   - 没有用户明确授权时，只能输出提交拆分计划和提交消息建议，不能执行 `git commit`

## Plan

在执行任何写操作前，先向用户说明本次准备怎么处理：

1. 工作区是否可视为单一逻辑提交
2. 如果不能，准备拆成几组提交，以及每组的大致边界
3. 是否只做分析，还是用户已经明确要求执行本地提交
4. 如需提交，说明会按组暂存并逐个本地提交，但不会 push

如果用户尚未明确授权提交，只输出计划并等待确认，不要继续执行 `git add` 或 `git commit`。

## Commands

### 1. 收集改动上下文

至少运行以下只读命令，并基于结果判断提交边界：

```bash
git branch --show-current
git status --short
git diff --stat
git diff --cached --stat
git diff HEAD
```

当改动较大时，可以补充：

```bash
git diff --name-only
git diff --cached --name-only
```

### 2. 生成提交分组计划

按“一个可审查逻辑单元”拆分，而不是机械按目录拆。优先遵循以下规则：

- 同一功能链路的实现与测试优先放在同一提交
- 同一缺陷修复涉及的实现与测试优先放在同一提交
- 独立文档改动与独立重构应拆开
- 无关改动不要混入同一提交

每组都要给出：

- 提交类型
- scope
- 中文简短描述
- 涉及文件列表
- 是否建议纳入本次提交

提交类型、scope 和中文消息格式参考 `commands/references/commit-message-guide.md`。

输出格式使用下面这个模板：

```text
提交 1/2
类型: feat(app)
描述: 添加期刊导入命令处理流程
文件:
- path/to/file1
- path/to/file2
```

### 3. 仅在获得明确授权后执行本地提交

用户明确要求“提交”“执行 commit”“按这个计划提交”之后，才能执行下面的操作：

1. 必要时清理错误暂存，但不要破坏用户未授权放弃的改动
2. 只暂存当前这一组提交需要的文件
3. 执行本地 `git commit`
4. 继续下一组，直到所有已确认分组完成

执行时遵守这些限制：

- 禁止 `git push`
- 禁止创建空提交
- 禁止把敏感文件、冲突文件、无关改动带进提交
- 若发现用户工作区在处理中发生变化，先重新展示计划再继续

## Verification

无论是只分析还是实际提交，最后都要验证结果。

1. 只分析模式：
   - 确认已经给出完整分组计划
   - 确认没有误执行 `git commit`
2. 实际提交模式：
   - 每次提交后运行 `git show --stat --oneline HEAD`
   - 必要时运行 `git status --short`，确认剩余改动与计划一致
   - 汇总每个新提交的短哈希与标题

如果任何提交结果与计划不一致，明确说明偏差，不要假装成功。

## Summary

结果输出保持简洁，至少包含：

```text
## Result
- Action: 仅分析提交计划 | 已执行本地提交
- Status: success | partial | failed
- Details: 分组数量、是否已提交、提交哈希或阻塞原因
```

## Next Steps

根据结果给出下一步建议：

- 仅完成分析：请用户确认是否按计划执行本地提交
- 已完成本地提交：提示可以继续开发或手动 push（只提醒，不代执行）
- 因状态异常中止：指出先解决冲突、清理敏感文件或整理工作区后再重试
