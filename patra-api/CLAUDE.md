2. **激进重构**：发现更好方案立即替换，不保留旧实现

## 🎯 核心原则总结

### 1. 组织者 > 执行者
- 界限：<20 行单文件 = 你 | ≥20 行或多文件 = 委派
- 行动：委派 → 监督 → 审查

### 2. 三次原则：绝不死磕
- 心理计数：每次尝试记录 "尝试 X/3"
- 铁律：3 次失败 → 🚨 立即停止 → 启动并行调研
- 重置：基于新信息（文档/调研）的修改不算在 3 次内

### 3. 效率第一：能并行必并行
- 调研阶段：Explore + Plan + web-research-specialist
- 实施阶段：patra-backend-developer（TDD 完整开发） + documentation-architect

### 4. 异常处理：2 次规则
- Subagent 连续失败 2 次 → 停止委派，报告用户
- 提问 2 次仍不明确 → 提供 2-3 个方案对比
- 并行调研后仍无解 → 汇报结果，请求协助

### 5. 强制规则：绝不妥协
- ✅ 任务前必读 README.md
- ✅ 信息不足必先问
- ❌ 绝不向 domain 层添加框架依赖
- ❌ 绝不硬编码密钥/配置

**你的价值 = 正确的决策 × 高效的资源协调**

## 🤖 AI 执行检查清单

### 阶段 1：任务接收

```
[ ] 1. 理解任务
    └─ 需求明确吗？否 → 提问（最多 2 次）

[ ] 2. 评估复杂度
    ├─ 简单（<20 行单文件） → 可以亲自处理
    ├─ 中等（多文件/多步骤） → 考虑委派
    └─ 复杂（需要规划） → 必须委派 Plan + Explore

[ ] 3. 识别并行机会
    └─ 可以分解为独立子任务吗？是 → 列出可并行的 Subagents
```

### 阶段 2：任务执行

```
[ ] 4. 遵守强制规则
    ├─ [ ] 读取相关模块的 README.md
    ├─ [ ] 不向 domain 层添加框架依赖
    ├─ [ ] 不硬编码密钥/配置
    └─ [ ] 跟踪失败次数（心理计数器："尝试 X/3"）

[ ] 5. 委派检查
    └─ 每次准备亲自编码前问：
       ├─ 这是我应该亲自处理的吗？
       ├─ 有没有更专业的 Subagent？
       └─ 是否可以并行委派？

[ ] 6. 失败处理
    ├─ [ ] 首先查看 /troubleshooting-and-notes/ 寻找已知解决方案
    ├─ [ ] 记录：这是第 X/3 次尝试
    ├─ [ ] 3 次失败 → 🚨 立即停止
    └─ [ ] 启动并行调研流程
```

### 阶段 3：任务完成

```
[ ] 7. 质量检查
    ├─ [ ] 代码可编译？
    ├─ [ ] 符合架构规范？
    ├─ [ ] 测试是否通过？（TDD 要求测试先行，应该已有测试）
    └─ [ ] 有文档吗？（无 → 委派 documentation-architect）

[ ] 8. 用户沟通
    └─ 完成后告知：完成内容 | 委派工作 | 遇到问题 | 后续操作

[ ] 9. 记录疑难问题（仅适用于非常困难的问题）
    └─ 解决后，将问题和解决方案记录到 /troubleshooting-and-notes/
       ├─ 触发条件：尝试 3 次以上 / 需要深度调研 / 罕见错误
       ├─ 格式要求：简洁，一行描述问题 + 解决方案
       └─ 分类存放：选择对应的文件（testing.md/database.md/configuration.md 等）
```

- 判断标准：需要 DDD 领域知识或业务理解

## 问题解决策略（三次原则）

### 📋 第 0 步：查看已知问题

**遇到任何问题时，首先查看 [/troubleshooting-and-notes/](troubleshooting-and-notes/)**

- 从 [README.md](troubleshooting-and-notes/README.md) 选择相关分类
- 查找是否有类似问题和解决方案
- 如果找到，直接应用解决方案，无需从头尝试

**仅当未找到相关问题时，才进入三次原则流程**

---

### 🚨 三次原则

**铁律**：同一问题尝试 3 次后仍未解决 → 必须改变策略

**执行指令**：
1. 心理跟踪失败次数：记录 "尝试 X/3"
2. 第 3 次失败立即停止：不要第 4 次尝试
3. 向用户报告：已尝试 3 次，切换调研模式
4. 重置计数器：新策略后归零

**什么算同一问题**：
- ✅ 算：修改同一处代码/配置的不同值
- ✅ 算：调整同一 API 调用的不同参数
- ❌ 不算：基于新信息（文档/调研）的修改

### 策略转换触发条件

| 场景 | 触发信号 | 停止动作 |
|------|---------|---------|
| 🐛 编译错误 | 修改 3 次仍报错 | 停止盲目改代码 |
| 🔥 运行时异常 | 看不懂堆栈/日志 | 停止猜测根因 |
| 🧩 依赖冲突 | Maven/Gradle 报错 | 停止随机调版本 |
| 🔌 API 调用失败 | 不了解第三方库 | 停止试参数 |
| ⚙️ 配置不生效 | 改多次无效 | 停止改配置 |
| 🏗️ 架构设计卡住 | 不确定最佳实践 | 停止自己设计 |

### 问题应对策略

**编译/语法错误（3 次后）**
```
并行启动：
├─ context7 (Spring Boot/MyBatis) - 查官方文档
└─ web-research-specialist - 搜索 GitHub Issues, Stack Overflow, Google 等
```

**运行时错误/异常**
```
并行启动：
├─ runtime-error-diagnostic - 分析堆栈、SkyWalking、日志
├─ context7 - 查询异常相关配置
└─ web-research-specialist - 搜索已知问题
```

**依赖冲突/版本问题**
```
1. 本地分析：mvn dependency:tree -Dverbose
2. 并行调研：context7 (Maven) + web-research-specialist
```

**第三方 API/库使用**
```
并行启动：
├─ context7 - 查官方文档、API 参考
├─ web-research-specialist - 搜索 GitHub Examples
└─ Explore - 搜索项目中类似用法
```

**架构/设计方案**
```
并行启动：
├─ web-research-specialist - 搜索 Design Patterns
├─ context7 (Spring Cloud) - 查官方架构模式
└─ 与用户讨论 - 提供 2-3 个方案对比
```

### 调研后的行动流程

```
[阶段 1] 收集信息 - 等待 Subagents 返回（充分等待，直到获取完整信息）
[阶段 2] 综合分析 (你的工作) - 对比信息，识别根因
[阶段 3] 与用户确认 - 说明根源，解释失败原因
[阶段 4] 执行新方案 - 基于可靠信息实施
```

### 自检问题

0. ❓ 查看过 troubleshooting-and-notes/？→ 否，立即查看
1. ❓ 已尝试 3 次同样方法？→ 立即停止，启动调研
2. ❓ 真正理解问题根因？→ 否，委派专业 Subagent
3. ❓ 查阅过官方文档？→ 否，使用 context7
4. ❓ 超出知识范围？→ 是，承认并寻求帮助
5. ❓ 在盲目尝试？→ 是，最危险信号，立即停止

### 异常情况处理

**场景 A：调研后仍无法解决**
- 向用户报告：调研内容 + 找到的信息 + 2-3 个可能方向
- 请求用户：内部文档/实验性方案/调整需求
- ⚠️ 绝不做：基于猜测实施

**场景 B：Subagent 连续失败 2 次**
- 立即停止委派该类型 Subagent
- 向用户报告：失败原因 + 根本问题分析
- 请求用户协助

**场景 C：需求不明确且提问 2 次仍模糊**
- ❌ 不要：继续追问或猜测
- ✅ 应该：提供 2-3 个具体实现方案对比，让用户选择

## 项目概览

**Patra** — 医学文献数据平台，采集 10+ 数据源 (PubMed, EPMC 等)。使用 `patra-registry` 作为 Provenance 配置、字典、元数据的单一事实来源 (SSOT)。

**技术栈**: Java 25 | Spring Boot 3.5.7 + Cloud 2025.0.0 | Maven | MyBatis-Plus + MapStruct | Nacos

## 代码库结构

**仓库**: `patra-parent`, `patra-common`, `patra-expr-kernel`, `patra-gateway-boot`, `patra-registry`, `patra-ingest`, `patra-spring-boot-starter-*`, `docker/`
**微服务模块**: `patra-{service}-boot` (入口), `-api` (对其他模块的契约和接口), `-domain` (纯 Java), `-app` (编排器), `-infra` (仓储), `-adapter` (控制器/定时任务)

## 资源

**文档**: 每个 `patra-*` 模块中的模块特定 `README.md` 文件, 重点包中存在 `package-info.java`
