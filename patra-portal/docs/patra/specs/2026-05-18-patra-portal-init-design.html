关键决策摘要

新建 Patra/patra-portal 作为 Patra 的 C 端前端，采用 Next.js 15 + React 19 + TypeScript strict + Tailwind v4 + shadcn/ui + TanStack Query + Zustand。

v1 范围 仅交付脚手架：一个落地页 + /api/health + 一个 smoke 测试 + 全套 dev/build/test/lint/typecheck 命令。不实现任何业务页面、不接 patra-gateway-boot API、不预置鉴权/i18n/Docker/CI。

独立 git 仓库（已 git init），暂不绑远端。命名 patra-portal 与已归档的 patra-web 区分，避免混淆。

1. 背景

Patra 是医学出版物数据平台，后端由四个微服务构成：patra-registry（SSOT 注册中心）、patra-ingest（采集）、patra-catalog（目录索引）、patra-gateway-boot（API 网关）。

历史前端 patra-web（Nuxt 4 + @nuxt/ui 4）仅为 demo，已于 2026-05-18 归档至 ~/Projects/Archive/patra-web。本设计为全新 C 端前端 patra-portal 的 v1 脚手架，未来承载文献检索、期刊浏览、个人订阅等场景。

绿地约束：无历史包袱、单人开发、质量优先。设计中已剔除"向后兼容 / 数据迁移 / 团队协作妥协 / 分阶段实施"等维度——它们对此项目均为 YAGNI。

4. 决策交代

v1 涉及 3 个关键决策，每个均评估了至少 2 个备选方案。

4.1 Meta-framework 选 Next.js 15

采纳

医学文献 C 端 SEO 关键（文献详情页需被搜索引擎索引），RSC 让首屏数据预取干净；生态最厚，碰到问题可搜性最强。

备选方案及未采纳原因
TanStack Start · 2025 GA，类型安全最强，Vite 基底

未选：生态较年轻，SSR 不如 Next 成熟；患文献站点的 SSR/SEO 是刚需，押注还不够稳。

Vite + React Router 7（纯 SPA） · 最轻量、构建最快

未选：无 SSR，文献详情页对搜索引擎不友好；需要额外做 prerender，反而复杂。

4.2 代码风格工具选 Biome

采纳

单二进制覆盖 lint + format，速度比 ESLint + Prettier 组合快一个数量级；2025 末已具备 ESLint 主要规则集等价覆盖。配置文件单一（biome.json），心智负担低。

备选方案及未采纳原因
ESLint v9 + Prettier · 生态最厚，插件最全

未选：双工具配置膨胀、跑得慢；本项目无需稀有插件，Biome 的覆盖足够。

oxlint + Prettier · oxlint 是当下最快的 lint

未选：oxlint 仍以 lint 为主，format 还需配 Prettier，回到双工具。一站式不如 Biome。

4.3 仓库形态选「单 app 独立 git 仓」

采纳

v1 只有一个 app；过早引入 monorepo（pnpm workspace / Turborepo）会带来不必要的 workspace 配置成本。未来若加 admin，按需开 patra-admin 独立仓即可——Patra 顶层非 monorepo，这一模式与 patra-api 一致（也是独立仓）。

备选方案及未采纳原因
pnpm workspace monorepo（apps/ + packages/） · 未来易扩 admin 等多 app

未选：YAGNI。当下没有 admin 需求，预留 packages/ui 等仅增加心智成本。

沿用旧名 patra-web · 语义最朴素

未选：与已归档的 demo 同名会造成历史 commit / 配置 / 引用混淆，强制重名得不偿失。

7. 配置要点

TypeScript：strict: true、noUncheckedIndexedAccess: true、paths 配 @/* → src/*。
Biome：启用 recommended 规则集 + import 排序；缩进 2 空格；行宽 100；尾随逗号 all。
Tailwind v4：通过 @import "tailwindcss"; 在 globals.css 引入；不再需要 tailwind.config.js，按 v4 CSS-first 配置（@theme 语法）。
TanStack Query Provider：lazy 初始化（useState(() => new QueryClient())），避免 SSR/Client 双实例。Provider 文件标 'use client'，在 app/layout.tsx 中包裹 {children}。
Husky pre-commit：依次执行
  1. pnpm exec lint-staged：对 *.{ts,tsx,js,jsx,json,md,css} 跑 biome check --write
  2. pnpm typecheck：仓库级 tsc --noEmit（不按 staged 分片——TS 必须整体类型检查）
任一失败则阻止 commit。
Node 22 LTS：通过 .nvmrc + mise .tool-versions 双重锁定，与 Patra 工作区其他项目一致。

8. 风险与缓解

风险说明缓解
低 Tailwind v4 CSS-first 配置与 shadcn/ui 的初始化流程仍在演进，初始化可能踩 CLI 兼容坑 使用 shadcn CLI 最新 init；问题出现时按官方升级 issue 修复，不阻塞 v1
低 React 19 + Next 15 RSC 心智模型对开发者有一定迁移成本 v1 不写业务，问题留到 v1.1 出现真实组件时再处理；Provider 边界已在脚手架阶段画清
低 Biome 与 ESLint 生态覆盖差异：某些社区规则在 Biome 缺失 recommended 覆盖足够；若未来出现缺口可在 plan 阶段并入 ESLint 局部

无中/高风险——脚手架阶段范围窄、可逆性强。

9. 错误处理与可观测性

v1 仅保证：

  • /api/health 返回 200 + { status: "ok", version, timestamp }
  • 未捕获错误走 Next.js 默认 error.tsx 与 not-found.tsx（提供最小占位实现）

未来按需引入：Sentry / OpenTelemetry / 结构化日志。

10. 测试策略

v1 范围：仅一个 smoke 测试，确保渲染管道 + 测试基础设施可运行——用 Testing Library 渲染落地页，断言出现 "patra-portal"。

未来：组件单测、Page 集成测试（Vitest + Testing Library）、e2e（Playwright，待页面出现后引入）。

11. Git 与仓库管理

  • Patra/patra-portal/ 为独立 git 仓库（已 git init）

  • 首个 commit 仅含本 spec（已 commit）；脚手架代码作为后续 commit

12. 验收标准

  • pnpm install && pnpm dev 后访问 http://localhost:3000 显示 "patra-portal" 落地页
  • curl http://localhost:3000/api/health 返回 {"status":"ok",...}
  • pnpm build && pnpm start 生产模式可启动
  • pnpm lint / pnpm typecheck / pnpm test 全部通过
  • git commit 触发 pre-commit 钩子，校验通过后才允许提交
  • README.md 列出全部命令与目录说明
